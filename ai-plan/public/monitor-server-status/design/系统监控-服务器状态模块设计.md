# 系统监控-服务器状态模块设计

## 1. 设计目标

- 定义 `server` 插件 `monitor` 与 `web` 模块 `monitor` 的统一边界。
- 首个能力固定为 `server-status`，只做一个最小可用的状态查询闭环。
- 保持 `title_key + title fallback`、`index.ts + bootstrap-routes.ts`、`group path + route fragment` 的既有仓库约束。
- 让实现自然落在 `server/plugins/monitor/**` 与 `web/src/modules/monitor/**`。

## 2. 非目标

- 不引入 Prometheus、Grafana、WebSocket 或多节点聚合。
- 不把未来信息架构占位直接接入真实运行菜单、路由或权限。
- 不做告警策略、历史持久化或跨节点拓扑。
- 不把共享热点当成长期 ownership。

## 3. 菜单与信息架构

- 真实运行导航采用一个显式分组段：`服务器管理 / 服务器状态`。
- 后端在当前 slice 中显式注册两个菜单项：
  - `/monitor`
    - 标题：`服务器管理`
    - 只作为导航父级，不绑定独立运行页面
  - `/monitor/server-status`
    - 标题：`服务器状态`
    - 指向当前唯一真实页面
- 当前唯一真实页面仍然只有 `server-status`。
- 菜单标题继续使用 `title_key` 作为唯一长期真值，`title` 只作为 fallback。
- 前端按现有 bootstrap 菜单快照装配树形导航，不再依赖硬编码父级标题，也不再让页面或面包屑暴露英文 `index`。
- 未来 IA 占位项 `概览`、`性能趋势`、`健康检查日志`、`配置诊断` 只记录在设计文档中，不进入当前运行态。

## 4. 前端模块设计

- 模块目录固定为 `web/src/modules/monitor/**`。
- 模块入口固定为 `index.ts`，动态路由声明固定为 `bootstrap-routes.ts`。
- 模块内部继续按 `pages`、`api`、`contract`、`locales`、`types` 组织。
- 页面只负责展示聚合状态和手动刷新，不持有壳层逻辑。
- 路由注册只暴露一个 bootstrap 入口，对应单个状态页。

## 5. 后端插件设计

- 插件目录固定为 `server/plugins/monitor/**`。
- 插件名固定为 `monitor`，首个能力名固定为 `server-status`。
- 注册阶段负责声明菜单、权限、路由和消息 key。
- 启动阶段只处理已声明的运行时行为，不新增隐式能力。
- 该能力优先聚合现有核心运行时信息，不新增独立持久化层。

## 6. 权限与 RBAC 设计

- 只保留一个读取权限，建议命名为 `monitor.server-status.read`。
- 菜单和 API 都应受同一权限约束。
- MVP 不包含写权限、编辑权限或分级授权。
- 前端只在具备权限时暴露菜单入口。

## 7. API 设计

- 采用单接口聚合。
- 路由契约继续使用 `group path + route fragment`。
- 建议约定：
  - group path: `/monitor`
  - route fragment: `/server-status`
  - endpoint: `GET /api/monitor/server-status`
- 该接口返回页面所需的最小状态汇总。

## 8. 数据结构与响应示例

- 当前返回值固定包含：
  - `status`
  - `observed_at`
  - `server`
  - `runtime`
  - `dependencies`
  - `summary`
  - `trend`
  - `plugins`
- `server` 承载版本、启动时间、运行时长和应用级摘要。
- `runtime` 承载 Go 运行时、宿主机与内存统计快照。
- `dependencies` 承载数据库、Redis 等核心依赖的状态、说明与延迟样本。
- `summary` 承载依赖和插件的计数型总览，供前端仪表卡片直接消费。
- `trend` 承载单进程内存态短窗口趋势，不做历史持久化。
- `plugins` 承载运行时依赖顺序下的插件版本、依赖关系与状态摘要。

## 9. i18n 设计

- `title_key` 继续作为菜单和动态路由标题的唯一长期真值。
- `title` 只在装配边界用于外部输入回退。
- 前端模块消息建议使用 `monitor.*` 前缀。
- 当前运行态使用两个稳定 key：
  - `monitor.sectionTitle`
  - `menu.monitor.server_status.title`
- 后端错误和状态消息保持插件内收口，不外泄为裸文案。

## 10. MVP 范围

- 一个插件：`monitor`。
- 一个前端模块：`monitor`。
- 一个能力：`server-status`。
- 一个真实运行页：`服务器状态`。
- 一个真实菜单父级：`服务器管理`，仅用于导航层级和面包屑语义。
- 一个 API：聚合状态查询。
- 一个页面：带总览卡片、依赖面板、运行时信息、插件表格和短窗口趋势图的状态仪表页。

## 11. 后续扩展方向

- 信息架构预留：
  - `概览`
  - `性能趋势`
  - `健康检查日志`
  - `配置诊断`
- 增加更细粒度的依赖状态。
- 增加趋势视图或历史快照。
- 增加告警能力。
- 在真正需要时再评估多节点与实时推送。

## 12. 风险与取舍

- 单接口聚合容易变胖，必须控制返回字段数量。
- 只做读模型意味着它不是完整可观测平台。
- 趋势窗口只保留进程内短期数据，页面刷新或进程重启后不保证连续历史。
- 当前状态页继续采用轮询而不是 WebSocket：
  - 页面只读且默认 5 秒轮询已足够覆盖当前单页总览需求
  - 页面不可见暂停和失败退避可以避免无意义高频请求
  - 真正需要准实时多流推送时再重新评估 SSE 或 WebSocket
- 不引入外部监控栈能保持边界清晰，但也意味着指标深度有限。
- 若后续扩展跨到共享热点，必须重新切片，而不是顺手扩写。
- 前端图表与仪表卡片必须依赖主题 token 和颜色模式响应；不允许硬编码只在单一明暗模式下可用的颜色。

## 13. 验收标准

- 插件名为 `monitor`，模块名为 `monitor`。
- 首个能力名为 `server-status`。
- 采用 `server/plugins/monitor/**` 与 `web/src/modules/monitor/**`。
- 真实运行导航表现为 `服务器管理 / 服务器状态`，且面包屑不得出现英文 `index`。
- 当前运行态只存在 `server-status` 一个真实页面；未来 IA 占位不进入运行菜单、路由或权限。
- 继续使用 `title_key + title fallback`。
- 前端继续使用 `index.ts + bootstrap-routes.ts`。
- 后端路由继续使用 `group path + route fragment`。
- API 返回 `server + runtime + dependencies + summary + trend + plugins` 的 richer payload。
- 仪表页在主题切换和 brand theme 切换下仍保持图表、卡片、边框和文案可读性。
- MVP 不包含 Prometheus、Grafana、WebSocket 或多节点。
