# 代码注释与模块文档规范

## 1. 目标

这份规范用于约束 `Graft` 仓库中的手写代码注释与模块导航文档，尤其是 AI 参与生成、修改、重构代码时的默认行为。

目标：

* 让注释成为稳定的架构与维护知识，而不是临时噪音
* 保持注释与实现长期同步，避免多轮 AI 修改后文档失真
* 让新贡献者先通过注释和模块导航理解边界，再进入实现细节
* 把详细设计真值保留在 `ai-plan/design/`，避免模块目录文档膨胀为第二套设计文档

## 2. 适用范围

本规范默认适用于仓库中的手写代码与模块级目录文档，包括：

* `server` 核心与插件中的手写 Go 代码
* `web` 中手写的 TypeScript、Vue 与模块目录文档
* 测试代码中的意图说明与边界说明

以下内容默认豁免：

* 生成代码
* 第三方代码
* 迁移产物
* 构建产物

豁免只作用于产物本身，不向上传递。围绕这些产物的手写封装、适配层、服务边界与调用约束仍必须遵循本规范。

## 3. 核心规则

### 3.1 注释语言

* 注释统一使用中文。
* Go 导出符号仍必须符合标准 Go 文档注释格式，即以符号名开头。
* 如需保留稳定技术名词，可在中文句子中直接使用英文术语。

### 3.2 注释与代码同步

* 注释属于代码的一部分，不得因重构、提取方法、命名调整或格式化被随意删除。
* 当代码逻辑、参数语义、返回值约束、生命周期、失败行为发生变化时，相关注释必须同步更新。
* 当代码与旧注释冲突时，不能直接信任旧注释，必须核对实现、调用方与上下文后再修正。

### 3.3 禁止机械注释

禁止生成低价值或重复字段名的机械注释，例如：

* `Name 是名称`
* `ID 是 ID`
* `设置值`
* `调用方法`
* `返回结果`

注释应优先解释：

* 为什么这样设计
* 为什么这里需要显式边界
* 为什么不能进一步简化
* 为什么存在特殊兼容或生命周期处理

### 3.4 字段注释策略

字段注释只写关键字段，不做全字段机械补齐。重点覆盖：

* 生命周期敏感字段
* 并发共享字段
* 可空或必须非空的约束字段
* 有特殊注册顺序、缓存语义或资源归属的字段

## 4. 注释优先级

时间或改动范围受限时，按以下优先级补齐和审查注释：

1. `public API` 注释
2. 架构边界注释
3. 并发 / 生命周期注释
4. 业务规则注释
5. 普通实现注释

低优先级注释不得挤占高优先级注释的准确性与完整性。

## 5. Go 注释规则

### 5.1 Package 注释

每个手写 Go package 都应通过 `doc.go` 或等价位置提供包注释，说明：

* 包职责
* 与核心或插件边界的关系
* 何时应使用该包，而不是其它更简单位置

### 5.2 导出符号注释

所有导出的 package、type、interface、func、method、const、var 必须有 Go 标准文档注释，并满足：

* 以符号名开头
* 使用中文
* 解释职责、契约、约束、失败语义，而不是复述字面行为

对函数、方法和命令装配入口，优先使用以下两层写法：

* 默认写法：先说明职责，再补充边界、调用顺序、失败行为或使用约束
* 复杂写法：当参数语义、返回值约束、生命周期顺序或副作用不直观时，再增加 `参数：`、`返回值：` 分段

不要求把每个函数都机械写成参数清单。只有像 CLI 编排、生命周期调度、资源装配这类调用约束明显的函数，才使用
类似 `server/internal/cli/dev.go` 的分段说明。

### 5.3 结构体与接口

对结构体、接口等长期稳定边界，注释应优先解释：

* 职责
* 生命周期
* 线程安全或共享约束
* 调用方可依赖的稳定语义

### 5.4 复杂逻辑行注释

以下场景必须优先使用独立行注释，而不是行尾注释：

* 锁与并发控制
* 生命周期切换
* defer / recover 的执行时机
* 缓存与单例构建
* 事务、重试、状态机
* 注册顺序、关闭顺序与兼容约束

对于普通直线流程，不要为了“有注释”而补无信息量行注释；只有当读者无法仅从代码形状看出设计原因时才添加。

## 6. Web 注释规则

`web` 不要求对所有普通导出都机械补注释，但以下内容必须有中文说明：

* 非平凡路由装配
* 权限判定与菜单装配语义
* 依赖后端菜单、权限、插件元数据的前端契约
* 共享状态为何必须进入 store，而不是留在页面局部

## 7. README.md 规则

### 7.1 定位

模块目录下的 `README.md` 是模块导航文档，不是详细设计文档。

它用于帮助贡献者快速回答：

* 这个模块做什么
* 它的边界在哪里
* 主要入口在哪
* 要从哪些文件开始阅读
* 它依赖谁、向谁暴露能力

详细设计、跨模块决策和长期演进约束仍放在 `ai-plan/design/`。

### 7.2 适用粒度

`README.md` 主要要求落在具有独立职责的模块级目录，例如：

* `server/internal/<module>`
* `server/plugins/<name>`
* `web/src/modules/<name>`
* 必要的壳层模块目录

不要求每个薄包、生成目录或产物目录都提供 `README.md`。

### 7.3 最低内容

模块级 `README.md` 至少包含：

* 模块用途
* 职责边界
* 主要入口
* 关键依赖或上下游关系
* 扩展或维护提示

## 8. 测试与协作要求

* 测试函数应说明测试目标、关键场景与边界。
* `TODO` 与 `FIXME` 使用统一格式：`TODO(作者/模块)`、`FIXME(作者/模块)`。
* Code review 必须检查注释是否与实现同步，而不是只检查“是否有注释”。
* 规范新增后，不要求一次性全仓清理历史注释；优先在当前改动范围和高价值模块中增量治理。
