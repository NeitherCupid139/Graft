# 系统配置模型与渲染设计

本文档定义 System Config 的长期信息架构、注册模型、展示模型、编辑模型和阶段性优化路线。

System Config 的目标不是做 Apollo / Nacos 式外部配置中心，也不是用户个性化配置系统。它是 Graft 模块在
`Register` 阶段声明系统级配置定义、由管理员保存覆盖值、再由运行时消费有效值的模块化平台能力。

## 1. 当前状态

当前 System Config 已经形成基础闭环：

- `server/internal/configregistry` 拥有模块注册的配置定义。
- `server/modules/system-config` 拥有管理员覆盖值、API、权限、菜单和服务行为。
- OpenAPI source 拥有 `/api/system-configs` 的 wire contract。
- `web/src/modules/system-config` 消费 OpenAPI generated type，并提供系统配置页面。
- `web/src/shared/schema-form` 已经提供 schema 解析、值预览和 JSON Schema 字段编辑能力。

现有后端定义已经不是纯 flat item。`configregistry.Definition` 已包含：

- `Domain` / `DomainKey` / `DomainLabel`
- `Group` / `GroupKey` / `GroupLabel` / `GroupDescriptionKey`
- `Title` / `TitleKey`
- `Description` / `DescriptionKey`
- `Type`
- `Schema`
- `DefaultValue`
- `Sensitive`
- `RestartRequired`
- `Permission`
- `Order`

现有 OpenAPI `system-config-item` 也已经暴露 `domain`、`group`、`type`、`config_schema`、
`default_value`、`effective_value`、`override_value`、`has_override`、`status`、`updated_at`、
`updated_by_*`、`sensitive`、`masked` 等字段。

因此，当前能力已支持“对象配置值”：

- 访问日志保留清理：`httpx.access-log-retention-cleanup`
- 应用日志保留清理：`logger.app-log-retention-cleanup`
- 审计日志保留清理：`audit.audit-log-retention-cleanup`

这些配置通过 `type=object` 和 `config_schema.properties` 表达 `retentionDays + batchSize`。

当前不足：

- “配置对象”还不是显式合同层，只是一个 `ConfigDefinition` 的 object value。
- field 只存在于 `config_schema.properties`，OpenAPI 没有派生的 `fields` 视图。
- scalar item 与 object item 在页面上已经接近统一，但展示、编辑、JSON 预览和技术标识仍混在页面实现中。
- 后端保存校验只对 object config 使用 scheduler JSON Schema 子集；scalar config 只校验 JSON shape。
- 通知模块已有多个 `notification.*` 配置，但多数字段仍是独立 scalar，且当前缺少 rich `config_schema`。

## 2. 推荐信息架构

长期信息架构采用轻量层级：

```text
Domain -> Group -> Config Object -> Field
```

### 2.1 Domain

`Domain` 表示最高层配置域，用于左侧导航和跨模块配置心智分区。

示例：

- `dashboard`
- `logs`
- `notification`
- `audit`

`Domain` 不等同于 backend module。多个 module 可以属于同一个 domain，例如 access log、app log、audit log 都可以参与
日志保留配置，但它们的 runtime authority 仍分别归属原 owner。

### 2.2 Group

`Group` 表示 domain 内的用户任务分组，用于把相关配置聚合成一个可浏览区域。

示例：

- `quick_actions`
- `log.retention`
- `notification.general`
- `notification.sources`
- `notification.delivery`
- `notification.display`

`Group` 应表达用户要完成的配置任务，而不是简单复制包名、模块名或技术路径。

### 2.3 Config Object

`Config Object` 是页面上可阅读、可编辑、可保存的配置单元。

当前实现中，一个 `configregistry.Definition` 就是一个 Config Object：

- scalar definition 视为只有一个 `value` field 的 Config Object。
- object definition 通过 `config_schema.properties` 派生多个 Field。

Phase 1 和 Phase 2 不需要修改后端合同；前端可先在 view model 中统一 `Config Object`。

Phase 3 如需扩展 OpenAPI，可新增 derived `fields` 视图，但 `config_schema` 仍是 field 渲染 authority，不允许维护第二份字段真相。

### 2.4 Field

`Field` 是 object config 的 schema property，或 scalar config 的虚拟 `value` 字段。

字段元数据来源顺序固定为：

1. `config_schema`
2. `config_schema["x-i18n"]`
3. `item.type`
4. `item.title_key` / `item.description_key`
5. 后端 fallback copy

前端不能为特定 config key 长期硬编码字段标题、枚举文案或控件类型。缺少 rich schema 时，应优先修复模块拥有的
`ConfigDefinition.Schema`，而不是在页面内添加兼容映射。

## 3. 配置项、配置对象与配置组边界

### 3.1 继续作为独立配置项

满足任一条件时，继续使用独立 scalar Config Object：

- 语义独立，管理员通常单独判断、单独修改。
- 权限、敏感性、重启要求或生效时机与同组其它字段不同。
- 运行时消费方只需要一个独立值。
- 该值本身就是一个明确开关，例如 `notification.enabled`。

### 3.2 合并为配置对象

满足以下条件时，优先合并为 object Config Object：

- 1 到 3 个字段强相关，必须一起理解。
- 字段共享同一个 owner、group、permission、sensitive、restartRequired 和生效语义。
- 字段通常一起展示、一起保存、一起回滚默认值。
- 拆成多个 scalar 会让页面堆出“散项”，降低管理员理解成本。

示例：

- `retentionDays + batchSize`
- `showReadDays + popupLimit`
- `enabled + maxItems + strategy`

### 3.3 对象内部字段分区

当一个配置对象有 4 到 8 个字段，并且仍属于同一用户任务时，可以保留一个 object，但 UI 需要字段分区。

当前 backend schema 子集尚未定义字段分区 metadata。实现前应先明确 schema 扩展，如 `x-ui.sections` 或等价 derived
view，避免页面按 key 猜测分区。

### 3.4 拆更多配置对象或配置组

满足任一条件时，应拆为多个 Config Object 或 Group：

- 字段超过 8 到 10 个。
- 字段分属不同用户任务。
- 字段风险等级不同，例如基础开关、投递限制、来源开关混在一起。
- 字段 owner 或运行时消费方不同。
- 保存或回滚应该分开完成。

通知 `sources` 的多个事件开关不应合并成一个巨大 object。它们属于同一 group，但每个来源开关仍可作为独立
Config Object，或后续按来源类别拆成小 object。

## 4. 注册与合同模型

### 4.1 后端注册 authority

模块拥有的 `ConfigDefinition` 是配置定义 authority。

规则：

- `ConfigDefinition` 在模块 `Register` 阶段声明。
- `ConfigDefinition.DefaultValue` 是默认值 authority。
- `ConfigDefinition.Schema` 是字段渲染、字段校验和 i18n metadata authority。
- `system_config_values` 只存管理员覆盖值，不复制默认值。
- `server/internal/config` 只保留启动和环境配置，不接管 System Config 语义。

### 4.2 OpenAPI authority

OpenAPI source 是 wire contract authority。

规则：

- generated Go / TypeScript 类型是 derived artifact，不是 authority。
- 前端 API boundary 必须继续消费 `@/contracts/openapi/generated/schema`。
- 不新增绕过 `request.ts` 的 generated runtime client。
- 如新增 `fields`，它必须是从 `config_schema` 派生的只读视图。

### 4.3 Schema 子集

当前后端 object 校验复用 scheduler JSON Schema 子集，可靠覆盖：

- root object
- `properties`
- `required`
- `additionalProperties`
- property type: `string` / `integer` / `number` / `boolean`
- `enum`
- `minimum` / `maximum`
- `minLength` / `maxLength`

Phase 3 前不要承诺复杂嵌套 object、array item schema、条件校验或动态字段。

### 4.4 迁移策略

本次从旧 flat key 收敛到 object key 不保留兼容旧逻辑。

迁移原则：

- 新 object key 作为唯一 canonical key。
- 旧 flat keys 不再参与运行时读取。
- 不做双读、双写、fallback、桥接读取或灰度兼容。
- 所有消费者必须在同一轮 Loop 中迁移到新 object key。
- 前端、后端、OpenAPI、测试、locale key 与配置注册定义必须一次性收敛。

删除旧 key 后，验证范围必须覆盖：

- 配置注册扫描
- System Config 列表展示
- 编辑保存
- 默认值读取
- override 读取
- 运行时消费方读取
- i18n key-first 校验
- 后端单测 / 前端类型检查

如果发现仍有消费者依赖旧 flat key，应直接修复消费者，而不是新增兼容层。
不允许实现旧 flat key 兼容逻辑；不要新增 migration compatibility map、legacy key fallback、dual read/write 或临时桥接。发现旧 key 消费点时，直接迁移到新 canonical object key，并补齐验证。

## 5. 前端展示与编辑模型

### 5.1 页面结构

System Config 页面属于 settings 页面，不是 CRUD 列表页。

推荐结构：

- 左侧：Domain / Group 树，支持搜索、选中态和空态。
- 右侧：Group header + Config Object cards。
- Card：统一展示标题、说明、状态、当前值、默认值、最后修改、操作。
- Advanced：折叠展示 raw JSON、技术标识、schema 摘要等低频信息。

### 5.2 展示模型

展示应统一为 Config Object Card：

- scalar item 使用同一种 card，只是字段数为 1。
- object item 默认展示核心字段。
- object 字段较多时，只展示关键字段，其余字段折叠。
- 只有存在 override 时才并列展示“当前值 vs 默认值”；没有 override 时可弱化默认值重复展示。
- `技术标识` 不应常驻占据主视觉区域，应进入高级信息或次级操作。
- `最后修改` 在 `has_override=false` 时显示“无覆盖值”或等价文案，比“暂无修改记录”更直接。

状态建议：

- `使用默认值`
- `已修改`
- `无覆盖值`
- `敏感`
- `需重启`

### 5.3 编辑模型

编辑入口根据字段数量选择：

- 单字段 scalar：保留简单 Dialog。
- 1 到 3 字段 object：Dialog 或小 Drawer 均可，优先保持轻量。
- 4 个以上字段：优先 Drawer，提供更好的纵向表单空间。
- raw JSON：作为高级编辑视图，不默认展示。

保存模型：

- 当前阶段仍以整个 `value` 保存。
- 字段级 patch 不进入 Phase 1 到 Phase 3。
- 保存前应执行前端字段校验；后端仍作为最终 authority。
- 对 duration 类字段，schema 必须明确存储单位和展示单位，避免前端猜测。

### 5.4 Field Renderer

字段 renderer 以 `$graft-system-config-field-renderer` 为 baseline。

控件选择顺序：

| Schema signal | Editor |
| --- | --- |
| `enum` present | `Select` |
| `type: "boolean"` | `Switch` |
| `type: "integer"` or `type: "number"` | `InputNumber` |
| `type: "object"` or `type: "array"` | JSON textarea |
| `type: "string"` | `Input` |

如果 schema 缺失、格式错误或无法提供可用 type，才 fallback 到 `item.type`。

TDesign Vue Next 组件覆盖结论：

- `Switch` 覆盖 boolean。
- `InputNumber` 覆盖 integer、number、数量、天数、秒数和带 suffix 的 duration。
- `Select` 覆盖 enum，并支持 string / number / boolean value。
- `Input` 覆盖 string。
- `Textarea` 覆盖 raw JSON object / array。
- `Form` 承载字段级校验和表单提交。
- `Collapse` 承载高级 JSON 与技术标识。
- `Dialog` 适合 scalar 和小 object。
- `Drawer` 适合多字段 object。
- `RadioGroup` 可作为小枚举增强，但不是 baseline；默认继续用 `Select`。

暂不纳入 baseline：

- `Slider`
- `DatePicker`
- `TimePicker`
- nested object visual builder
- array item visual builder

这些能力只有在 schema 和后端校验先定义后才能进入 renderer。

### 5.5 I18n

展示文案解析规则：

- schema title 优先使用 `config_schema["x-i18n"].titleKey`。
- schema description 优先使用 `config_schema["x-i18n"].descriptionKey`。
- placeholder 优先使用 `config_schema["x-i18n"].placeholderKey`。
- enum option 优先使用 `config_schema["x-i18n"].enumLabels.<value>.labelKey` 和 `.descriptionKey`。
- fallback copy 只作为外部输入兜底，不应成为前端长期真相。

新增 schema copy 或 locale key 时，必须落在 owning module 的 locale boundary。

新增系统配置定义时，必须同时补齐三层本地化真相：

- 后端 `ConfigDefinition` 必须声明 `DomainKey`、`GroupKey`、`GroupDescriptionKey`、`TitleKey`、`DescriptionKey`，并在
  owning module 的后端 i18n 注册中提供 `zh-CN` / `en-US` 文案。
- `config_schema` 中可见的 `title`、`description`、`placeholder`、`enumLabels` 必须带对应 `x-i18n` key；这些
  key 与 item 级 `TitleKey` / `DescriptionKey` 不得只停留在 backend fallback copy。
- 浏览器运行时只聚合 `web/src/locales/**` 与 `web/src/modules/*/locales/**`，不会自动消费后端 i18n 注册表。因此
  任何会显示在 System Config 页面上的新增 key，都必须在 owning frontend locale boundary 补齐 `zh-CN` / `en-US`
  catalog；没有前端模块边界时，先落在 `web/src/modules/system-config/locales/**`，并补测试断言页面不显示英文 fallback。

验收要求：

- 页面测试必须覆盖至少一个新增配置项，断言标题、描述、group/domain 文案来自 locale key，而不是后端英文 fallback。
- `bun run lint:i18n` 或 `bun run check` 必须通过。
- 后端模块测试必须覆盖配置定义 metadata key 已注册到后端 i18n。

## 6. 典型配置迁移建议

### 6.1 日志保留清理

当前形态合理，应作为 object config baseline：

- `retentionDays`
- `batchSize`

保持 access log、app log、audit log 各自 owner 不变。

### 6.2 工作台首页快捷入口

工作台首页快捷入口的 authority 已调整为壳层派生视图：

- 默认由当前可见 sidebar / bootstrap 菜单派生。
- 不再把模块声明式 quick link / quick action 注册当作长期 contract。
- 新增菜单时，不再额外评估首页快捷入口注册。

如果未来仍需要“显示数量、排序策略、隐藏规则”这类首页快捷入口展示配置，它只能是 shell-owned UI policy，
不得重新引入模块级 quick link registration 作为第二真相。

### 6.3 通知展示

Phase 4 迁移前是两个 scalar：

- `notification.display.show_read_days`
- `notification.display.popup_limit`

建议迁移为一个 object：

- key: `notification.display`
- fields:
  - `showReadDays`
  - `popupLimit`

这是最适合作为 notification object 迁移试点的配置。

### 6.4 通知通用与来源

`notification.general` 中的 `enabled` 可继续独立，因为它是总开关。

`retention_days` 和 `default_page_size` 可在后续评估是否合并为 `notification.general.defaults`。

`notification.sources` 不建议一次性合并为一个大 object。多个来源开关数量较多，且未来可能绑定来源 owner 或事件类型
metadata，应保持 group 内多个小 Config Object 或按来源类别拆分。

## 7. 分阶段路线

### Phase 1 - UI 一致性，不改后端模型

- 统一 scalar 和 object card 结构。
- JSON 预览改为折叠高级信息。
- 技术标识弱化为高级信息或复制入口。
- 状态文案改为更直接的“使用默认值 / 已修改 / 无覆盖值”。
- 保持现有 `/api/system-configs` 合同不变。

### Phase 2 - 前端 renderer 抽象

- 抽出 `ConfigValueRenderer`。
- 抽出 `ConfigEditorRenderer`。
- 保持 `$graft-system-config-field-renderer` 决策表为 baseline。
- 增加可选扩展位：小枚举 `RadioGroup`、多字段 `Drawer`、高级信息 `Collapse`。
- 补前端字段级校验，覆盖 scalar schema 的 enum/range。

### Phase 3 - 后端 registry 和 OpenAPI 增强

- scalar config 也按 schema 校验 enum/range/length。
- object config 继续校验当前平面 schema 子集。
- 可新增 derived `fields` 响应视图，但 `config_schema` 保持 authority。
- 不引入字段级 patch。

### Phase 4 - 典型配置迁移

- 保持日志保留清理 object baseline。
- 迁移 `notification.display`。
- 迁移 `dashboard.quick_actions`。
- 评估 `notification.general.defaults`。
- 不合并 `notification.sources` 为巨大 object。

### Phase 5 - 验证与验收

- 后端 focused tests:
  - `server/internal/configregistry`
  - `server/modules/system-config`
  - 迁移涉及的 owner module
- 前端 focused tests:
  - shared renderer
  - System Config page
  - i18n governance
- 合同验证:
  - OpenAPI source 和 generated artifact 同步
  - web API boundary 继续消费 generated type
- UI 验收:
  - desktop / mobile 截图
  - object config 编辑
  - scalar config 编辑
  - sensitive config
  - default / modified / no override 状态
