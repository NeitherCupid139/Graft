# 后端查询与数据库访问治理规范

## 1. 目标

这份文档定义 `Graft` 在 `server` 侧实现查询、分页、计数、批量读写与数据库访问时的 AI Guardrail。

本规范是治理规则，不是 ORM 教程，也不是数据库性能调优百科。

它解决的问题：

- 防止 AI 或人工实现把“先跑通”变成 `N+1`、全表扫描、深分页、宽字段误取和不可解释的慢查询
- 为 code review、性能验收和后续 CI 结构化检查提供统一判定标准
- 明确哪些规则必须提供证据，哪些规则当前只能文档审查

本规范覆盖：

- Ent 查询
- 手写 SQL
- repository / store / query service 层数据库访问
- 列表查询、详情查询、统计查询、批量写入、批量更新、批量删除

本规范不覆盖：

- Atlas migration 设计细则
- 索引 DDL 设计全量规则
- 缓存策略本身
- API DTO 语义边界

## 2. Authority 与适用边界

查询语义 authority：

- `server` 中实际发起数据库访问的 query/repository/store 实现

数据库结构 authority：

- Ent schema 与 live migration

API 过滤、排序、分页参数 authority：

- 上游 request/contract 定义

下游 consumer 不得通过本地兼容逻辑掩盖上游查询 authority 的问题。

例如：

- API 默认无限分页，不允许在 handler 临时截断后声称“已治理”
- 索引缺失导致查询退化，不允许只靠减少返回字段来伪装性能问题已解决

## 3. 总体 Guardrail

### 3.1 默认要求

所有新查询或修改后的热点查询默认满足：

- 明确查询用途：列表 / 详情 / 统计 / 存在性判断 / 批量处理
- 明确过滤条件、排序键、分页策略与字段选择
- 不引入 `N+1`
- 不依赖无界全表扫描作为常态路径
- 不使用 `SELECT *`
- 对大字段、长文本、JSON、blob 类字段采用显式取数策略
- 对批量操作给出批次边界
- 当查询被认定为热点、列表主路径、审计路径、后台任务主循环或已知慢查询修复时，必须附带 `EXPLAIN` 或等价证据

### 3.2 禁止默认接受的实现

- 先查主表再逐条补查关联数据
- 无过滤或弱过滤下直接扫描大表并在内存中过滤
- 列表接口默认不分页
- 用 `SELECT *` 读取列表数据
- 用 `COUNT(*)` 作为每个列表请求的固定前置动作，且没有说明成本和必要性
- 用深 `OFFSET` 解决所有翻页场景
- 在单事务中无上限处理大量行且无分批策略
- 用“数据库会优化”代替索引与执行计划说明

## 4. 查询形态规则

### 4.1 N+1 Guardrail

禁止：

- 查询列表后，对每一行再发一次关联查询
- 在循环里按主键、外键、状态等重复取数

允许：

- 先取主记录集合，再用一次 `IN` 查询批量取关联数据
- 使用明确受控的 eager loading，但必须确认不会把无关大字段一并带出

必须提供的评审点：

- 主查询发几次 SQL
- 关联数据补齐发几次 SQL
- 关联结果如何映射回主结果

CI 适合做：

- 静态规则扫描明显的循环内数据库调用模式

当前仍以文档审查为主：

- eager loading 是否导致隐性宽查询
- 某些 repository 包装是否在内部偷偷二次查询

### 4.2 全表扫描 Guardrail

禁止把全表扫描当成默认业务路径。

允许存在全表扫描的前提：

- 表规模明确很小且受平台约束
- 这是一次性维护脚本或后台修复路径
- 已记录为什么不能在当前切片补足索引或过滤条件

评审必须检查：

- 查询是否有有效过滤条件
- 排序是否能利用已有索引
- 是否依赖函数包裹、前缀模糊、隐式类型转换等导致索引失效

热点查询必须提供：

- `EXPLAIN` 或 `EXPLAIN ANALYZE` 证据
- 说明预期使用的索引或明确说明当前就是小表扫描

### 4.3 分页默认值 Guardrail

所有列表查询默认必须分页，除非该接口明确是：

- 下拉枚举且结果集天然很小
- 内部批处理迭代器且不直接暴露给用户请求
- 明确受上限保护的管理动作

默认规则：

- request contract 必须定义分页参数
- 未传分页参数时必须有默认页大小
- 必须有最大页大小上限
- 默认排序必须稳定，优先采用可索引且可复现的排序键

推荐方向：

- 普通后台列表默认 `page_size` 应保守
- 深翻页明显的列表应评估 keyset/cursor，而不是长期依赖大 `OFFSET`

评审必须给出：

- 默认页大小
- 最大页大小
- 排序键
- 深分页策略是接受、限制还是未来升级

CI 适合做：

- request/handler/repository 是否显式接收分页参数
- 是否存在未设置 limit 的列表查询

### 4.4 `SELECT *` Guardrail

列表查询禁止 `SELECT *`。

详情查询也不应默认 `SELECT *`，除非：

- 实体字段确实就是最小返回集
- 表很窄且没有大字段
- 已确认没有把未来敏感字段一并暴露到下游组装链路

必须显式列出字段的场景：

- 列表
- 导出预览
- 任意包含大字段/敏感字段的表
- 跨层 DTO 装配前的中间查询

评审必须检查：

- 返回字段是否和当前 use case 对齐
- 是否顺手把不必要字段带到内存、日志或 JSON 序列化链路

CI 适合做：

- 手写 SQL 中 `SELECT *` 的阻断

文档审查为主：

- Ent 默认全字段查询是否应改成显式字段选择

### 4.5 Count 策略 Guardrail

`COUNT` 不是默认免费操作。

列表查询必须先回答：

- 是否真的需要总数
- 总数是否必须强一致
- 是否可以延迟、近似、条件性返回，或只返回 `has_more`

默认要求：

- 不要在每个列表请求中机械地先查一遍 `COUNT(*)`
- 当过滤条件复杂、表大、连接多时，`COUNT` 需要单独评估成本
- `COUNT` 查询要和列表查询一样审查索引与过滤条件

允许策略：

- 首屏或显式请求时才计算总数
- 用额外取 `limit + 1` 判断 `has_more`
- 将复杂统计与列表主体解耦

评审必须记录：

- 是否返回总数
- 总数的用途
- 若不返回总数，前端或调用方如何判定还有下一页

### 4.6 EXPLAIN 证据 Guardrail

以下场景必须提供执行计划证据：

- 新增热点列表查询
- 慢查询修复
- 新增复杂 join / 聚合 / 子查询
- 已知大表上的筛选、排序、统计查询
- 新增索引并声称其支撑了查询路径

可接受证据：

- `EXPLAIN`
- `EXPLAIN ANALYZE`
- 数据库层截图或文本输出
- 在 PR/任务说明中摘录关键计划结论

证据至少回答：

- 预期索引是否被使用
- 是否出现全表扫描、回表过重、排序代价过高
- 计划是否支持当前分页/统计路径

CI 目前不适合直接阻断：

- 执行计划依赖真实数据分布和数据库环境

因此这部分当前属于：

- 强制评审证据
- 非自动化阻断规则

### 4.7 查询超时 Guardrail

所有可能变慢的查询路径都必须有超时意识。

要求：

- 数据库访问必须继承可取消的 `context`
- 后台任务、导出、统计、批处理查询需要有明确 timeout 或 deadline 策略
- 不允许默认无限等待数据库返回

评审必须确认：

- timeout 由哪一层注入
- 超时后返回什么错误语义
- 是否会产生重试风暴或半完成副作用

CI 适合做：

- 检查关键 repository/query API 是否接收 `context.Context`

文档审查为主：

- timeout 数值是否合理
- 是否按查询类型区分超时预算

### 4.8 大字段获取 Guardrail

大字段包括但不限于：

- 长文本
- JSON 大对象
- blob / 二进制内容
- 审计详情原文
- 配置快照、请求体、响应体等可膨胀字段

规则：

- 列表禁止默认取大字段
- 详情仅在真实需要时取大字段
- 大字段优先拆成按需读取，而不是搭车跟随常规查询

评审必须回答：

- 哪些字段属于大字段
- 为什么当前查询需要它
- 是否存在列表误取、联表误取或日志误打出风险

CI 目前多为文档审查：

- 是否为大字段需要结合 schema 语义判断

### 4.9 批量操作 Guardrail

批量插入、更新、删除、回填、迁移修复必须有批次边界。

默认要求：

- 明确单批大小
- 明确事务边界
- 明确失败重试或幂等策略
- 不在单次请求或单事务中无上限处理整表

当批量操作依赖先查后写时，还必须防止：

- 查询结果全集加载到内存
- 循环内逐行更新形成写侧 `N+1`

评审必须给出：

- 批大小
- 总量预估
- 是否需要游标/主键窗口推进
- 是否会与在线查询竞争锁或 I/O

CI 适合做：

- 发现明显无界循环数据库写入模式

### 4.10 索引与查询语义 Guardrail

索引不是“有就行”，必须服务真实查询语义。

每个热点查询都应明确：

- 过滤列
- 排序列
- 连接列
- 覆盖字段或回表接受度

禁止：

- 先写查询，再假设 DBA 或未来某个切片会补索引
- 为通过一次慢查询而堆叠缺乏语义解释的索引

评审必须检查：

- 查询语义与索引顺序是否匹配
- 排序是否与过滤一起受同一索引支持
- 复合索引是否真服务主路径，而不是只看单列命中

CI 目前只能覆盖弱规则：

- 文档中声明索引 owner
- 变更热点查询时要求存在 evidence 段落或引用

## 5. 证据要求

### 5.1 必须提供证据的场景

- 新增或修改列表主查询
- 新增复杂统计查询
- 慢查询修复
- 新增批量任务主循环
- 声称“已避免 N+1 / 已命中索引 / 已控制大字段”的变更

### 5.2 最低证据包

最低证据包至少包含：

- 查询用途
- 主要 SQL/Ent 查询形态
- 分页与排序策略
- 字段选择策略
- `COUNT` 策略
- 若属热点路径，附 `EXPLAIN` 结论

### 5.3 允许仅文档审查的场景

- 小表后台枚举
- 只读详情且字段窄、调用频率低
- 非热点内部管理脚本

但即使如此，也不能违反：

- 无界分页
- `SELECT *`
- 查询无超时意识
- 明显 `N+1`

## 6. Review 清单

评审后端查询变更时至少检查：

- 是否引入 `N+1`
- 是否默认分页且带上限
- 是否存在 `SELECT *`
- `COUNT` 是否有必要
- 是否错误读取大字段
- 是否需要 `EXPLAIN` 证据而未提供
- 查询超时和取消语义是否存在
- 批量操作是否有批次边界
- 索引是否匹配过滤与排序语义

## 7. CI 适合规则与文档规则

### 7.1 适合进入 CI 的规则

- 禁止手写 SQL `SELECT *`
- 列表查询路径必须显式 `limit`
- repository/query 层必须接收 `context`
- 明显循环内数据库调用模式告警
- 任务说明或 PR 模板中要求热点查询 evidence 字段

### 7.2 当前仅适合文档与评审的规则

- 是否构成真实热点查询
- `COUNT` 是否值得保留
- `EXPLAIN` 结论是否足够好
- 大字段是否应拆分按需读取
- 索引设计是否最优

## 8. 违规处理

若当前切片无法直接满足本规范，必须记录：

- 违反了哪条 guardrail
- canonical authority 在哪里
- 为什么这次不能直接修
- 临时风险是什么
- 后续清理触发条件是什么

不得用“后续再优化”作为无证据豁免。

## 9. 落地要求

后续若仓库引入 AI Guardrail 自动检查，本规范应优先落地为：

- SQL 结构规则
- 分页/limit 结构规则
- context/timeout 结构规则
- 热点查询 evidence 模板规则

在没有自动化前，本规范仍作为 code review 阻断依据。
