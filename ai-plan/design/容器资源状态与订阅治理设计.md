# 容器资源状态与订阅治理设计

本文档定义 `Graft` 容器资源数据流的 canonical authority、前后端职责边界，以及面向 `List / Detail / Dashboard`
共享消费的 `ContainerStatsManager` 目标架构。

本设计不直接搬运外部项目实现；它只基于当前 `Graft` 容器模块与对照审计结论，收敛适合本仓库边界的资源状态层。

## 1. 当前问题与目标

当前 `Graft` 容器管理已经具备：

- 后端容器 metadata 列表与详情接口
- 后端 `resourceStatsCache`
- 后端 `statsCollector -> realtime topic`
- 前端详情页 `container.stats:{id}` 订阅
- 前端列表页 `container.stats.list` 聚合订阅

但当前仍缺少统一资源状态层，导致资源数据 authority 分散在多个消费点：

- 列表页通过 HTTP `resource` 读取容器资源摘要
- 详情页通过 HTTP `resource` 做初始值，再通过 realtime patch 局部覆盖
- Dashboard 当前尚未消费统一容器资源状态，但后续若接入，会面临第三套来源

本设计目标：

- 明确 metadata 与 stats 的 authority 分离
- 为前端提供单一可信的容器资源状态层
- 让 `List / Detail / Dashboard` 共享同一份 stats 状态
- 保留当前后端 `collector + cache + canonical topic` 方向，不回退到页面各自直连 runtime 的模式

## 2. Authority 定义

### 2.1 Metadata Authority

容器结构性信息的 canonical authority 为：

- `server/modules/container/**`
- `GET /api/ops/containers`
- `GET /api/ops/containers/{id}`

metadata 包括但不限于：

- container id / short id / name / names
- image / image id
- state / status / started_at / created_at
- labels
- ports
- networks / primary_ip / network_summary
- orchestrator metadata
- action availability

metadata 的最终解释权不属于 realtime stats topic。

### 2.2 Stats Authority

实时资源状态的 canonical authority 为：

- backend collector 产生的容器 stats snapshot
- canonical realtime topics
  - `container.stats.list`
  - `container.stats:{id}`

stats 包括但不限于：

- CPU
- Memory
- Network IO
- PIDs
- Resource Summary

HTTP `resource` 字段保留，但在治理语义上降级为：

- 初次渲染种子值
- collector last-known snapshot 的一次性投影
- 包含 `collected_at` 的 latest-known projection，供 HTTP seed 与 realtime snapshot 比较 freshness

前端最终显示态必须以共享 stats store 为准，而不是以任一页面局部 state 为准。

### 2.3 History Authority

history 不属于当前 MVP 容器资源 authority 主链。

如果未来引入趋势图或短时回放：

- history 应作为独立的 `History Store`
- 不得与 latest stats state 混在一个匿名页面局部数组里

## 3. 当前后端资源链路

当前 `Graft` 后端已具备如下链路：

```text
Docker ContainerList / ContainerInspect
  -> DockerRuntime.List / Detail
  -> Summary / Detail metadata
  -> resourceStatsCache.current(id)
  -> List API / Detail API

Docker ContainerStatsOneShot
  -> statsCollector
  -> DockerRuntime.CollectStatsSnapshots
  -> resourceStatsCache.set(id)
  -> realtime publish
  -> topic: container.stats.list
  -> topic: container.stats:{id}
```

当前实现语义：

- `DockerRuntime.List()` 返回 metadata row，并用 `resourceStatsCache.current(id)` 回填 `resource`
- `DockerRuntime.Detail()` 返回 metadata detail，并用 `currentResourceSummary(id)` 回填 `resource`
- `statsCollector` 周期采样，更新 cache，并向 realtime hub 发布 snapshot
- `resourceStatsCache` 已承担 backend latest-known stats store 角色

当前缺口分成两类：

- backend 采样正确性
  - 当前 `ContainerStatsOneShot` 在部分 Docker / runtime 组合下无法稳定提供可用 `precpu_stats`
  - 若 CPU 计算继续直接依赖 Docker 返回的 `precpu_stats`，则 `cpu_percent` 可能长期为 `null`
- frontend 统一消费层
  - 即使 backend 已能稳定产出 latest snapshot，若前端仍由多个页面各自 patch / seed / 覆盖资源态，也会继续出现 authority 分散和显示不一致

因此本设计负责解决第二类问题；第一类问题需要在 backend 采样链上单独修复，例如：

- 保留 one-shot，但由服务端维护上一帧样本自行计算 CPU delta
- 或改为 runtime 级持续 stats stream

## 4. 当前前端问题

当前前端已有两个容器资源写入面：

- 列表页：
  - HTTP `payload.items` 先进入 module-owned `Metadata Store`
  - `resource` 仅作为 `Stats Store.seed`
- 详情页：
  - 详情结构与挂载增强仍由页面局部持有
  - `resource` 已改为通过 `ContainerStatsManager` selector 组合读取

这意味着：

- phase-1 之前列表页和详情页没有共享同一份 stats 状态
- phase-2 已将 `resource` authority 从页面局部 patch 上提到模块级 store
- 后续 Phase 3/4 仍需补齐订阅统一与 Dashboard 共享消费

治理目标是把这些局部 patch 逻辑上提为模块级资源状态基础设施。

## 5. 目标架构

### 5.1 总体图

```text
List API / Detail API
  -> Metadata Store
  -> Stats Store seed

container.stats.list
  -> Module-owned Subscription Manager
  -> Stats Store applyRealtime
  -> List / Dashboard selectors

container.stats:{id}
  -> Detail Subscription Manager
  -> Stats Store applyRealtime

UI selectors
  -> compose(metadata + stats)
  -> List / Detail / Dashboard
```

### 5.2 核心对象

#### Metadata Store

职责：

- 持有容器结构性信息
- 接收 List API 和 Detail API 的 metadata upsert
- 不解释 realtime stats

负责字段：

- container name
- image
- status
- labels
- ports
- network summary
- orchestrator metadata
- action availability

#### Stats Store

职责：

- 持有 latest realtime resource state
- 接收 HTTP seed snapshot
- 接收 realtime snapshot
- 维护 freshness / staleness / availability
- 阻止旧 HTTP seed 覆盖更新的 realtime snapshot

负责字段：

- CPU
- Memory
- Network
- PIDs
- Resource Summary
- collected_at
- source
- freshness state

#### Subscription Manager

职责：

- 管理 canonical topic 订阅
- 统一开关 socket
- 引用计数
- 生命周期
- idle grace cleanup

它不直接持有 UI 文案或页面行为。

当前治理补充：

- `container.stats.list` 是面向概览消费的 canonical list topic
- 它进入 container module 内部的共享 `Stats Store`，不能降级成页面局部 `row patch` 通道
- 容器管理列表页与 Dashboard 容器概览都应通过模块级 manager / facade 复用同一个 list-level subscription
- `container.stats:{id}` 继续保留给详情级消费，不作为概览页逐条订阅的默认路径

## 6. `ContainerStatsManager` 设计

### 6.1 位置

`ContainerStatsManager` 先放在：

- `web/src/modules/container/**`

原因：

- 当前它是容器模块专属基础设施
- topic、DTO、selector 和状态语义都由容器模块 authority 持有
- 现在升到 `web/src/shared/**` 会过早把 container-specific contract 伪装成平台层

### 6.2 推荐组成

```text
ContainerStatsManager
  |- metadataStore
  |- statsStore
  |- subscriptionManager
  |- selectors
```

推荐职责：

- `upsertList(items)`
- `upsertDetail(detail)`
- `seedStats(id, resource, collectedAt)`
- `acquire(id)`
- `release(id)`
- `getSummaryView(id)`
- `getDetailView(id)`
- `getResourceState(id)`

Phase 2 当前最小落地：

- 已实现 `Metadata Store`
- 已实现 `Stats Store.seed` 与 `Stats Store.applyRealtime`
- 已实现 list/detail selector 组合读取
- 尚未实现完整 `Subscription Manager` 的 acquire/release/ref-count 生命周期

### 6.3 数据写入规则

#### HTTP 写入

- List API / Detail API 的 metadata 总是写入 `Metadata Store`
- 若响应带 `resource`，则只写入 `Stats Store.seed`
- `seed` 不得被页面直接视为最终显示 authority

#### Realtime 写入

- 只有 `container.stats:{id}` 可以写入 realtime stats
- realtime snapshot 进入 `Stats Store.applyRealtime`
- 页面不得各自维护“收到消息就直接 patch row/detail”的长期逻辑

### 6.4 冲突规则

若 HTTP 与 realtime 都提供 resource：

- metadata 仍以最新 HTTP 结果为准
- stats 以 `collected_at` 更新顺序决定
- 没有 `collected_at` 的旧 HTTP seed 不得覆盖更新的 realtime snapshot

默认策略：

- `realtime newer than http seed` 时保留 realtime
- `http seed newer than current stale stats` 时允许覆盖
- `realtime unavailable snapshot` 不得抹掉 last-known usable snapshot，除非明确进入 expired / removed 状态

## 7. 生命周期设计

### 7.1 创建

- 页面声明需要的 `containerIds`
- `SubscriptionManager.acquire(id)`
- 首个订阅者触发 topic socket 建立
- HTTP 返回的 `resource` 先作为 seed
- 收到 realtime 后升级为 live stats

### 7.2 更新

- List refresh / Detail refresh 更新 metadata
- `Stats Store` 根据 `collected_at` 与 freshness 规则处理 seed / realtime 冲突
- UI 全部通过 selectors 读组合态，不直接改 `row.resource`

### 7.3 销毁

- 页面销毁或容器离开当前可见范围后 `release(id)`
- ref count 为 `0` 后进入短暂 idle grace
- grace 超时后关闭 socket
- 若 metadata 也不再存在，可清理 stats state

### 7.4 删除 / 不可达

- 若容器被 remove 或 detail 返回 not found：
  - `Metadata Store.remove(id)`
  - `Stats Store.remove(id)`
  - `SubscriptionManager.release(id)` 并关闭连接

## 8. 页面共享策略

### 8.1 List

- 列表页只负责声明当前可见容器集合
- 列表行资源展示从 `ContainerStatsManager` selectors 读取
- `rows.value` 不再是资源状态最终 authority，只是 metadata carrier

### 8.2 Detail

- 详情页通过同一个 manager 读取容器 metadata + stats
- 详情页不再直接维护独立 stats socket state
- realtime 状态文案可以保留在详情页 UI，但底层订阅和状态写入由 manager 负责
- 详情页的 pause / resume 只改变模块级 subscription acquire / release，不再直接拥有 socket controller

### 8.3 Dashboard

- 若 Dashboard 展示容器资源卡片或容器资源摘要，应直接消费同一 `Stats Store`
- 不新增新的容器 stats API
- 不新增 Dashboard 自有资源状态层
- 若 Dashboard 需要读取容器模块共享资源态，必须通过 `web/src/modules/container/contract/**` 暴露的最小 facade
  消费，不得直接导入容器页面实现，也不得把容器专属 stats manager 伪装成平台级 `shared/**` authority
- Dashboard 可拥有自身的 metadata projection collection，但 latest stats authority、realtime acquire/release 与
  freshness 规则仍归 `ContainerStatsManager`

## 9. OpenAPI 与后端建议

### 9.1 当前保持兼容

当前接口保持：

- `ContainerSummary.resource`
- `ContainerDetail.resource`

它们继续存在，避免立即 breaking change。

### 9.2 推荐增量字段

建议在 `ContainerResourceSummary` 增加：

- `collected_at`

用途：

- 明确 HTTP seed 与 realtime snapshot 的可比较时间
- 让 backend cache / HTTP projection / canonical realtime topic 共享同一 freshness 字段
- 减少页面局部“保留旧值”猜测逻辑

### 9.3 后端不做的事

本设计不要求后端：

- 改为按页面连接数实时直连 Docker stats stream
- 删除 `resourceStatsCache`
- 删除当前 List / Detail `resource` 字段

后端继续以当前 `collector + cache + canonical topic` 为 authority 主链。

## 10. 实施阶段

### Phase 1：资源所有权分离

目标：

- 明确 metadata 与 stats 的 authority
- 给 `ContainerResourceSummary` 增加 `collected_at`
- 把 HTTP `resource` 重新定义为 seed snapshot

### Phase 2：引入 `ContainerStatsManager`

目标：

- 建立 `Metadata Store + Stats Store`
- 页面开始通过共享 selectors 读资源状态

### Phase 3：统一订阅管理

目标：

- `SubscriptionManager` 统一 acquire / release / ref count / idle close
- 列表和详情共享同一份 stats topic 连接管理
- list 刷新失败不得通过全局 reset 抹掉其他页面仍在持有的容器 stats authority；只允许清理 list metadata 并释放当前可见订阅

### Phase 4：Dashboard 共享资源层

目标：

- Dashboard 复用同一 stats 状态
- 不增加第三套资源 authority
- 如需单独的 Dashboard metadata projection，必须保持为 container module-owned collection key，而不是新的 dashboard-owned
  stats store

### Phase 5：History Store（可选）

目标：

- 引入短时 ring buffer
- 支撑趋势图与资源回放
- latest state 与 history state 显式分离
- history 继续保持 module-owned frontend store，不升级为新的 shared/platform authority

## 10.1 当前节奏补充

- backend canonical 主链继续保持 `statsCollector -> resourceStatsCache -> container.stats:{id}`
- 默认采样周期调整为 `1s`，但仍由 container module system config 管理，可按部署覆写
- HTTP `resource` 仍仅是 seed snapshot / latest-known projection，不因更高采样率升级为最终 authority
- 列表、详情、Dashboard 的 CPU / 内存变化强调只属于 frontend derived visual state，不新增第二套 stats authority
- 当 backend 保留 `ContainerStatsOneShot` 时，CPU 百分比不再依赖 Docker 返回的 `precpu_stats` 作为唯一前提；服务端应维护上一帧可用样本来自行计算 delta CPU

## 11. 验证要求

后端：

- `resourceStatsCache` 继续保证 last-known usable snapshot 语义
- `ContainerResourceSummary.collected_at` 映射正确
- `container.stats:{id}` 仍保持统一 topic authority

前端：

- List 和 Detail 同时展示同一容器时只建立一条订阅
- 手动刷新不得回退已更新的 realtime stats
- 释放最后一个订阅者后按 idle grace 关闭连接
- Dashboard 接入后不得新增直接 patch `resource` 的写入点

## 12. 结论

`ContainerStatsManager` 应成为 `Graft` 容器管理模块的基础设施层。

它的边界应当是：

- module-owned frontend infrastructure
- backend collector/cache/topic authority 的直接消费层
- `List / Detail / Dashboard` 的统一资源状态解释层

它不应直接变成平台级 shared runtime，也不应把容器资源 authority 下放回每个页面的局部状态。
