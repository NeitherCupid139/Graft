# 契约治理与魔法值治理规范

## 1. 目标

这份文档定义 `Graft` 的 `Contract Governance Specification`，用于统一治理 `server`、`web` 与跨端共享的稳定契约。

本规范解决的问题不只是“抽常量”，而是：

* 哪些值属于稳定契约，哪些只是局部字面量
* 如何让契约具备明确类型，而不是把字符串简单搬家
* 契约如何拥有生命周期、所有权、兼容策略与漂移检测
* agent、本地开发与后续自动化如何围绕同一套契约真值工作

本规范覆盖两类对象：

* typed contract
  * 例如 permission code、event name、route name/path、header、storage key、error code、message key、feature flag、配置 key、状态枚举
* magic value governance
  * 重点治理高风险、跨边界、易漂移的字面量，而不是追求零字面量

---

## 2. 适用范围与术语

### 2.1 Contract

本仓库中的 contract 指“调用方可依赖其稳定语义”的值、类型或结构，典型包括：

* `server`
  * permission code
  * event name
  * HTTP header / cookie name
  * config key / env key
  * error code
  * message key
  * route path
  * 跨插件状态枚举
* `web`
  * route name / route path
  * storage key
  * request header / auth scheme
  * API path
  * API error code
  * feature flag key
  * query key
  * 前端按值分支的状态枚举
* cross-boundary
  * permission code
  * error code
  * message key
  * event name
  * menu / route / permission 相关稳定语义

### 2.2 Canonical Contract

同一语义只能存在一个 canonical definition。兼容期 alias 可以存在，但不能演变为第二个长期真值。

### 2.3 Magic Value

magic value 指未经明确归属、会影响契约、安全、导航、存储、国际化、配置或运行语义的字面量。

本规范不要求消灭所有字面量，只治理高风险和跨边界值。

---

## 3. 分层治理原则

### 3.1 必须集中治理的值

以下值默认视为 contract，不得散落：

* permission code
* route name / 特殊 route path
* storage key
* header 名称
* auth scheme
* error code
* message key
* event name
* config key / env key
* feature flag key
* 需要前后端按值分支的状态枚举

### 3.2 可保留的局部字面量

以下值允许保留在局部实现中：

* 单文件单次使用、且不构成契约的值
* 样式、布局、动画相关字面量
* 测试样本输入
* 标准库或框架已经提供良好表达的协议常量
* 抽取后会明显降低可读性的实现细节值

### 3.3 禁止事项

* 为了“零字面量”而创建新的 `constants.go`、`constants.ts`、`enums.ts` 垃圾桶
* 同一语义在多个目录重复定义
* 用文案本身替代稳定 key
* 用弱语义别名绕过 canonical contract
* 为了通过检查而牺牲局部可读性

---

## 4. Typed Contract Strategy

### 4.1 Go

`server` 中高风险字符串契约优先采用 typed string，而不是裸 `string` 常量。

路由 path 是例外场景：

* 当路由值只服务于模块内 Gin 注册、菜单路径复用、测试拼接或局部 allowlist 判断时，优先使用 plain `string`
  的 `group path + route fragment`
* 不要为这类路由常量再包一层无语义 typed alias、`String()` 方法或 `full path` 镜像常量
* 如确实需要完整路径，优先在局部通过轻量 helper 动态组合，而不是长期维护第二套 full-path 真值

推荐类型：

```go
type PermissionCode string
type ErrorCode string
type MessageKey string
type EventName string
type HeaderName string
type RoutePath string
type ConfigKey string
```

推荐写法：

```go
type PermissionCode string

const (
    UserReadPermission PermissionCode = "user.read"
    UserSessionRevokePermission PermissionCode = "user.session.revoke"
)
```

治理要求：

* 注册器、跨插件接口、统一错误响应、事件总线、配置读取等边界优先接 typed contract
* 只在适配层或最终序列化边界转换为裸 `string`
* 不允许“定义了 typed string，但核心 API 继续普遍收裸 `string`”

### 4.2 TypeScript

`web` 中高风险契约优先采用 `literal union`，在基础设施边界再按需引入 branded type。

推荐写法：

```ts
export const ROUTE_NAMES = {
  LOGIN: 'Login',
  RESTRICTED_SESSION: 'RestrictedSession',
} as const;

export type RouteName = (typeof ROUTE_NAMES)[keyof typeof ROUTE_NAMES];
```

基础设施边界可进一步使用 branded type：

```ts
type Brand<T, B extends string> = T & { readonly __brand: B };

export type StorageKey = Brand<string, 'StorageKey'>;
export type RoutePath = Brand<string, 'RoutePath'>;
```

治理要求：

* 业务模块优先用字面量对象 + union，避免早期泛滥复杂类型工具
* storage、router、request、contracts 等基础设施层可逐步引入 branded type
* 不允许只把字符串搬进对象，却继续在关键边界接受任意 `string`

### 4.3 Typed Contract 的验收标准

以下至少满足两项，才视为真正进入 typed governance：

* contract 有明确类型名
* contract 有专属归属目录
* 调用边界接 typed contract，而不是裸 `string`
* contract 拥有 lifecycle / owner / replacement 等元数据
* CI 可以做 drift / deprecated / orphan 检测

## 5. Contract 归属与发现规则

### 5.1 查找已有归属的顺序

新增高风险 contract 前，按以下顺序查找：

1. 同文件或同模块现有局部 contract
2. 同模块或同插件 `contract` 目录
3. 平台级 `contract` / `contracts` 目录
4. `server/internal/pluginapi` 等跨插件稳定边界
5. registry metadata / 索引文档

命中后应复用，不再平行新增。

### 5.2 归属规则

推荐目录：

* `server/internal/contract/<domain>`
* `server/plugins/<plugin>/contract`
* `server/internal/pluginapi`
* `web/src/contracts/<domain>`
* `web/src/modules/<module>/contract`

归属原则：

* 平台公共值放平台 contract
* 插件或模块私有稳定值放各自 contract
* 跨插件 contract 放 `pluginapi` 或等价稳定边界
* 不允许把所有值继续堆到一个通用 constants 目录

### 5.3 Discoverability

仓库需要 discoverability，但不需要“单一大字典”。

推荐方式：

* 真正定义仍分散在各自边界内
* 用轻量 registry metadata 记录：
  * canonical id
  * value
  * type
  * owner
  * lifecycle
  * source file
  * replacement
  * alias target

registry 是索引与治理视图，不是第二份正文实现。

## 6. Contract 拆分与命名空间规则

### 6.1 单文件职责限制

单个 contract 文件只承载一种语义，例如：

* `headers`
* `error codes`
* `message keys`
* `permissions`
* `routes`
* `storage keys`
* `feature flags`
* `status enums`

禁止把多种语义混入一个文件。

### 6.2 拆分阈值

达到任一条件就应拆分：

* 文件超过约 `150` 行
* 导出符号超过约 `20` 个
* 已同时承载两类以上语义
* review 时已经无法快速判断新值应该放哪里

### 6.3 命名空间规则

Go 示例：

* `server/internal/contract/http/headers.go`
* `server/internal/contract/i18n/message_keys.go`
* `server/plugins/user/contract/permissions.go`

TypeScript 示例：

* `web/src/contracts/http/headers.ts`
* `web/src/contracts/auth/error-codes.ts`
* `web/src/contracts/router/route-paths.ts`
* `web/src/modules/user/contract/status.ts`

## 7. 命名规范

### 7.1 Go

good：

```go
const RequestLocaleHeader HeaderName = "X-Graft-Locale"
const UserSessionRevokePermission PermissionCode = "user.session.revoke"
const AuthTokenExpiredCode ErrorCode = "AUTH_TOKEN_EXPIRED"
```

bad：

```go
const Header = "X-Graft-Locale"
const Revoke = "user.session.revoke"
const Err2 = "AUTH_TOKEN_EXPIRED"
```

要求：

* 名称必须体现领域、类别、用途
* 禁止 `Perm`、`Hdr`、`Cfg`、`Evt`、`Msg` 这类弱语义缩写

### 7.2 TypeScript

good：

```ts
export const STORAGE_KEYS = {
  USER_SESSION: 'graft_user_session',
  LOCALE: 'graft_locale',
} as const;
```

```ts
export const ROUTE_PATHS = {
  LOGIN: '/login',
  RESTRICTED_SESSION: '/auth/restricted-session',
} as const;
```

bad：

```ts
export const KEYS = {
  USER: 'user',
  LANG: 'locale',
};

export const PATH = '/login';
```

要求：

* 名称必须带语义类别
* 禁止 `KEY`、`VALUE`、`ITEM` 之类无上下文命名泛滥

## 8. 风险等级模型

### 8.1 分级

* `P0`
  * 安全、鉴权、跨端核心契约，错误后可能导致静默失效或权限问题
* `P1`
  * 重要平台契约，错误后会造成导航、存储或协作异常
* `P2`
  * 模块级稳定语义，错误后一般是模块故障而非平台级事故
* `P3`
  * 重复值与风格债，主要用于报告和逐步治理

### 8.2 典型映射

`P0`

* permission code
* auth error code
* 关键 header
* event name
* 认证特殊 route
* token / cookie key

`P1`

* storage key
* API path
* menu code / path
* message key
* config key / env key

`P2`

* 模块状态枚举
* query key
* tab key
* 模块内 feature flag

`P3`

* demo 状态值
* 局部重复字符串

### 8.3 阶段性执行要求

在自动化尚未完整落地前：

* `P0/P1` 必须先建立 canonical 归属与文档规则
* 新增高风险 contract 不允许继续散落为裸字面量
* `P2/P3` 先以报告与增量治理为主

## 9. Contract Lifecycle

每个高风险 contract 都应具备以下状态之一：

* `experimental`
* `stable`
* `deprecated`
* `removed`

兼容期可额外使用：

* `alias`

### 9.1 状态语义

`experimental`

* 允许小范围试用
* 不保证长期稳定

`stable`

* 可被跨模块、跨端、跨插件依赖
* 修改需要走 review 与兼容性治理

`deprecated`

* 已有 replacement
* 不允许新代码继续依赖
* 保留有限兼容窗口

`removed`

* 不再出现在代码、runtime 输出和索引中

`alias`

* 指向 canonical contract
* 必须记录 target、reason、created_at、expire_at

### 9.2 Compatibility Window

以下变更默认需要 compatibility window：

* route rename
* storage key rename
* error code rename
* permission rename
* event rename
* API path evolution

兼容窗口结束后必须清理旧引用与过期 alias。

## 10. Contract Ownership 与 Review

### 10.1 必要元数据

高风险 contract group 至少应具备：

* `owner`
* `reviewer`
* `approval_boundary`
* `stability`
* `source_of_truth`

推荐补充：

* `replacement`
* `schema_version`
* `consumers`
* `migration_note`

### 10.2 Approval Boundary

* `local`
  * 单模块或单插件 contract
* `platform`
  * 平台公共 contract
* `cross-boundary`
  * 前后端共享或跨插件共享 contract

`cross-boundary` contract 的 rename / remove / deprecate 不允许单边决定。

### 10.3 必须重点 review 的变更

* stable error code 新增或修改
* permission rename / remove
* event rename / remove
* route path rename
* storage key rename
* API path evolution
* message key rename
* alias compatibility window 新增或延期

以下变更必须附 migration note：

* route path rename
* storage key migration
* error code rename
* permission rename
* event rename
* API path 兼容演进
* `deprecated -> removed`

---

## 11. Compatibility Governance

### 11.1 Route Path Rename

* 保留旧 path redirect 或兼容 route
* 新代码优先切新 contract
* 兼容窗口后删除旧 path 引用

### 11.2 Storage Key Migration

* 读时兼容旧 key
* 写时优先写新 key
* 读取旧 key 后可迁移到新 key
* 兼容窗口结束后移除旧读取逻辑

### 11.3 Error Code Rename

* 默认优先新增更准确 code，再 deprecate 旧 code
* 尽量避免直接 rename 且立即替换

### 11.4 Permission Rename

* 默认视为高风险
* 应优先采用“双注册 + 数据迁移 + consumer 清理”

### 11.5 Event Rename

* 优先双 publish 或兼容订阅
* 不允许直接替换导致旧 consumer 静默失效

### 11.6 API Path Evolution

* 尽量并行 endpoint 或显式兼容路径
* 避免直接替换导致旧前端或旧调用方失效

## 12. Drift Detection 与自动化目标

这部分定义治理目标；Phase 1 先以文档与手动 review 为准，后续再落脚本、hook 与 CI。

### 12.1 Cross-End Drift

需检测：

* `server` 有、`web` 没有
* `web` 有、`server` 没有
* 同一语义的值不一致
* lifecycle 或 replacement 不一致

重点对象：

* error code
* message key
* permission code
* event name
* 特殊 route contract

### 12.2 Deprecated Drift

需检测：

* deprecated contract 是否仍被新代码引用
* deprecated contract 是否缺 replacement
* compatibility window 是否已过期

### 12.3 Orphan Contract

需检测：

* 已定义但生产代码已无引用
* registry 中仍存在，但 runtime 已不再注册或消费

### 12.4 Duplicate Semantic Contract

需检测：

* 同一语义是否被多处平行定义
* 同值不同名、同义不同值的候选冲突

### 12.5 Typed Enforcement 目标

长期目标：

* Go analyzer 检查关键边界是否仍接裸 `string`
* ESLint / AST rule 检查前端关键入口是否直接写高风险字面量
* CI 输出 drift report、deprecated report、orphan report、duplicate semantic report

---

## 13. Baseline / Allowlist 规则

历史问题允许阶段性保留，但必须受控。

每条 baseline / allowlist 至少应记录：

* `id`
* `path`
* `rule`
* `severity`
* `owner`
* `reason`
* `created_at`
* `cleanup_phase`
* `expire_at`

治理要求：

* `owner` 不能为空
* `reason` 必须具体，禁止只写 `legacy`、`todo`、`temporary`
* `expire_at` 不能为空
* 过期例外后续应升级为 warning 或 failure

---

## 14. Generated / Demo / Starter 豁免策略

### 14.1 完全跳过

* generated code
* third-party code
* build artifacts
* migration 产物正文

### 14.2 warning-only

* demo 页面
* starter 迁移中的样板模块
* mock 数据

### 14.3 正常阻断

* `server/internal/**`
* `server/plugins/**`
* `web/src/app/**`
* `web/src/router/**`
* `web/src/stores/**`
* `web/src/api/**`
* `web/src/modules/**`

warning-only 不应成为永久状态；一旦模块进入真实 runtime，应回到正常治理边界。

## 15. AI Agent Governance

agent 在新增或修改高风险 contract 时，必须按以下顺序行动：

1. 搜索同模块 contract
2. 搜索平台 contract
3. 搜索跨边界 contract 与 registry metadata
4. 复用 canonical definition
5. 只有不存在合理归属时才新增

agent 不得：

* 为局部方便复制已有 contract
* 在新代码里继续依赖 deprecated contract
* 把 alias 当作长期真值
* 为了通过扫描制造新的全局常量垃圾桶

agent 处理 alias / deprecated contract 时：

* 新代码优先用 canonical target
* 兼容桥接层才允许继续读写 alias
* 若 compatibility window 已结束，应优先清理旧引用

## 16. 成熟度模型与当前阶段

### 16.1 Maturity Model

`Level 0`

* 字面量散落
* 无 owner、无 lifecycle、无 drift 检测

`Level 1`

* 高风险值开始集中定义
* 有分类规则，但多为裸字符串常量

`Level 2`

* 关键 contract 已类型化
* 有 owner、lifecycle、phase-1 文档约束
* 可做基本 drift review

`Level 3`

* 有 analyzer / ESLint / AST rule
* 有 deprecated / orphan / duplicate 报告
* 关键兼容窗口进入常规 review

`Level 4`

* registry metadata、typed enforcement、CI drift report 联动
* baseline 与 alias 生命周期可持续治理

### 16.2 当前仓库推荐阶段

`Graft` 当前推荐处于 `Level 1 -> Level 2` 过渡：

* 先建立规范、归属、typed contract 与 lifecycle
* 再逐步补自动化扫描、hook 与 CI

当前阶段不推荐：

* 为所有 i18n key 生成双端强类型常量
* 引入全仓单源 codegen 平台
* 以“零字面量”为目标做大重构
* 在 owner / lifecycle 未稳定前先做重量级 schema version 平台

---

## 17. Phase 1 文档与落地要求

Phase 1 只建立治理基线，不要求立即完成全仓重构。

当前阶段的最低要求：

* `AGENTS.md` 明确 contract governance 强制规则
* 本文档作为契约治理与魔法值治理的设计真值
* 新增高风险 contract 时先找 canonical 归属，再决定是否新增
* 新增 shared contract、lifecycle 变更或 compatibility 变更时同步更新相应设计文档
* 在脚本、hook、CI 尚未落地前，不得声称 contract drift automation 已经通过；只能报告预期检查与当前缺口

这意味着 Phase 1 的完成目标是“文档真值先稳定、边界先稳定、命名先稳定”，而不是“把所有旧字面量一次性清空”。
