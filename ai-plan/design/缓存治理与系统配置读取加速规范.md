# 缓存治理与系统配置加速探索报告

## 1. 结论摘要

### 1.1 当前最大问题

当前 `Graft` 已具备系统配置页面与运行时配置注册能力，但系统配置读取链路仍未形成统一缓存层：

- `server/modules/system-config/service.go` 的 `Get` / `ResolveDefaultConfig` / `IsBooleanConfigEnabled` 仍按调用逐项读取 `configregistry` 与 `store.GetOverride(...)`。
- `server/internal/moduleapi/notification.go` 暴露的 `SystemConfigResolver` 目前只有布尔配置读取能力，且没有本地 snapshot、singleflight、版本号或显式失效机制。
- 容器、通知、调度、bootstrap 菜单过滤等路径已经开始依赖运行时系统配置，但每次读取都可能访问数据库。
- 部分容器配置仍由启动期 `server/internal/config/config.go` 的 env/viper 默认值驱动，尚未完全切换到统一的 System Config runtime authority。

这意味着系统配置已经从“管理后台配置项”演变成“运行时热点读路径”，但当前实现仍按低频管理写模型设计，存在明显放大数据库访问与运行时漂移的风险。

### 1.2 推荐方案

基于当前仓库形态，推荐：

- Phase 1 采用 `方案 C + singleflight`：
  - 单进程本地 typed snapshot 缓存
  - 以全量系统配置 snapshot 为主缓存对象
  - 读取时合并 `configregistry default + DB override`
  - 进程内 TTL + 显式失效
  - singleflight 防止并发击穿数据库
- Phase 2 预留 `方案 D`：
  - 本地缓存继续保留
  - 多节点时通过 Redis pub/sub 或轻量版本号轮询广播失效
- 不推荐把 Redis 作为系统配置 authority，也不推荐一开始把所有缓存都做成 Redis 缓存。

### 1.3 是否需要 Redis pub/sub

当前阶段不需要立刻引入 Redis pub/sub 作为 Phase 1 前提条件，原因是：

- 仓库当前更偏单体/模块化后端，单进程治理收益最高、改动最小。
- 现有真实热点首先是“重复 DB 读”与“模块绕过统一 resolver 的风险”，而不是多节点一致性。
- 项目已经具备 Redis 能力，Phase 2 再接入多节点失效即可，不必为了未来多节点提前把所有实现复杂化。

### 1.4 是否需要立刻改造

需要立刻完成治理与设计收口，并尽快推进 Phase 1 实现。原因：

- `server/modules/container/service.go`
- `server/modules/notification/publisher.go`
- `server/internal/scheduler/runtime.go`
- `server/modules/user/bootstrap.go`

这些运行时路径已经在频繁读取配置，但当前没有统一缓存层。继续扩展功能而不先补齐缓存治理，会把数据库读取与运行时语义漂移问题扩大到更多模块。

## 3. 系统配置读取链路

### 3.1 文字链路图

当前系统配置链路可归纳为：

`configregistry.Definition 注册默认值`
-> `system-config store 持久化 override`
-> `system-config Service.Get/List/ResolveDefaultConfig/IsBooleanConfigEnabled`
-> `SystemConfigResolver 或 scheduler.DefaultConfigResolver`
-> `container / notification / scheduler / bootstrap menu 等运行时消费方`

更细的读取路径如下：

1. 各模块在注册阶段通过 `server/internal/configregistry` 注册系统配置 definition、默认值、schema、`restart_required` 等元数据。
2. `server/modules/system-config/service.go` 在读取时：
   - 先从 `registry.Get(key)` 取 definition
   - 再从 `store.GetOverride(ctx, key)` 取 override
   - 有 override 则覆盖 default
   - 敏感配置则屏蔽返回值
3. `ResolveDefaultConfig(ctx, key)` 用于调度器类对象配置，底层仍调用 `Get(ctx, key)`。
4. `IsBooleanConfigEnabled(ctx, key, fallback)` 用于通知、容器、菜单 feature gate 等布尔开关，底层仍调用 `Get(ctx, key)`。

### 3.2 是否每次访问数据库

当前结论：是，命中这些读取路径时通常会访问数据库。

依据：

- `server/modules/system-config/service.go` 的 `snapshot(...)` 直接调用 `s.store.GetOverride(ctx, definition.Key)`。
- 当前未发现系统配置 effective value 的本地缓存、Redis 缓存、请求级缓存或 singleflight。

### 3.3 读取粒度

当前实现同时存在三种粒度，但 runtime 主路径本质上偏“按 key 单条读取”：

- `List(ctx)`：全量遍历 definition，并逐条查 override
- `Get(ctx, key)`：按 key 单条读取
- `ResolveDefaultConfig(ctx, key)` / `IsBooleanConfigEnabled(ctx, key, fallback)`：按 key 单条读取

这意味着管理页列表是 `N` 次 override 查询，而运行时热点是高频单 key 查询。

### 3.4 默认值与 typed config

当前已有：

- default config authority：`configregistry.Definition.DefaultValue`
- fallback default：由 definition 默认值天然承担
- typed boolean 解析：`IsBooleanConfigEnabled(...)`
- scheduler object config 解析：`ResolveDefaultConfig(...)` 返回 JSON 字符串，后续由 scheduler schema 校验/消费

当前缺失：

- 通用 typed config resolver
- domain/group 粒度 typed snapshot
- 配置版本号
- 统一的 runtime-hot / restart-required 生效策略

### 3.5 并发重复读取问题

当前存在明显并发重复读取风险：

- 并发请求同时读取相同配置 key 时，没有 singleflight。
- `List()` 会逐条访问 override，后台管理页刷新时易放大查询。
- 通知发布、bootstrap、scheduler job default 读取若成为热点，会放大数据库压力。

### 3.6 修改配置后如何通知运行时生效

当前写路径在 `server/modules/system-config/service.go` 中为：

- `Update(...)` -> `store.SetOverride(...)` -> `snapshot(...)`
- `Reset(...)` -> `store.DeleteOverride(...)` -> `snapshot(...)`

当前未发现：

- 本地缓存失效
- Redis pub/sub 失效广播
- 版本号 bump
- 显式 runtime refresh 挂钩

因此当前运行时“生效”依赖于消费方每次重新查库，而不是显式缓存刷新。

### 3.7 `restart_required` 与热读取语义漂移

当前系统配置模型已经支持配置项元数据，例如需要重启与 schema 展示，但运行时策略尚未闭环：

- 部分配置是 runtime-hot 读取，例如通知来源开关、容器布尔开关、scheduler default config。
- 部分配置仍主要依赖启动期 env/default 注入，例如 `ops.container.runtime`、`ops.container.docker.endpoint`、`logs.default_tail`、`logs.max_tail` 等。

这意味着“页面上可编辑”与“运行时立即生效”并不完全等价，存在 UX 与 authority 漂移风险。

### 4.1 当前重点风险

1. `ops.container.enabled` 并不是现状代码中的唯一真实 key，当前容器能力更多由：
   - `ops.container.runtime.enabled`
   - `ops.container.actions.dangerous_enabled`
   - `ops.container.shell.enabled`
   驱动，设计与排查时应优先以真实注册 key 为准。
2. 容器模块存在“部分 runtime-hot、部分启动配置”的双轨语义。
3. 通知与 scheduler 已经接入统一 resolver，但 resolver 本身无缓存，因此仍会查库。
4. bootstrap 菜单已按权限过滤，同时存在按系统配置 feature gate 过滤，意味着开关变更会直接影响前端菜单可见性。

## 5. 推荐缓存规范

## 5.1 缓存分层

### Request scoped cache

适用场景：

- 一次请求内重复解析同一用户、同一配置、同一权限集合
- 一次请求内的重复 RBAC / bootstrap / locale / typed config 派生对象

规则：

- 生命周期严格限定在单次请求
- 不做跨请求共享
- 不承担一致性职责

### Process local cache

适用场景：

- 读频繁、写较少
- authority 仍在本进程可回源的数据
- 单节点先优化
- 容忍极短 stale read

优先对象：

- system config effective snapshot
- typed config snapshot
- RBAC 权限聚合结果
- bootstrap 菜单 gating 派生结果
- dashboard 聚合读模型
- 短时容器挂载 usage

### Redis distributed cache

适用场景：

- 多节点共享读压力明显
- 失效广播比数据本身更重要
- 数据可序列化、跨节点复用收益明确

优先用途：

- 配置失效广播
- 短期共享聚合统计
- 已存在的监控趋势缓存

不作为首选用途：

- system config authority source
- 所有热点一股脑上 Redis

### Startup immutable cache

适用场景：

- 注册表、definition、menu、cron、dashboard descriptors

规则：

- 启动后只读
- 不提供运行时修改入口
- 不需要 Redis

### No cache

适用场景：

- 实时状态强依赖最新值
- 高错误代价的短生命周期结果
- 查询结果高度个性化且命中率低

例子：

- 日志分页查询结果
- 容器实时 CPU/内存状态
- 单次 shell ticket 状态
- 当前请求的审计明细查询

## 5.2 配置类缓存默认策略

系统配置读取建议默认采用：

- 进程内 typed snapshot 缓存
- 以全量 snapshot 为基础对象
- 支持从全量 snapshot 派生 domain/group/key 视图
- 支持 TTL
- 支持显式失效
- 支持默认值 fallback
- 支持 `runtime-hot` / `restart-required` 分类
- 支持配置版本号与审计信息

### 方案判断

#### 方案 A：每个 key 独立缓存

优点：

- 直观
- key 级失效精细

缺点：

- 当前 `Get` / `ResolveDefaultConfig` / `IsBooleanConfigEnabled` 调用点多，容易演化成大量零散缓存逻辑
- typed object 配置读取会出现重复聚合与重复 JSON 解析
- 多 key 场景容易放大 miss 风险

#### 方案 B：按 domain/group 缓存

优点：

- 比 per-key 更符合 system config 管理模型

缺点：

- 现有 store 层按 key 读写，没有天然 group 批量读取面
- 需要额外 group 索引或 group 扫描逻辑

#### 方案 C：全量系统配置 snapshot 缓存

优点：

- 最符合当前仓库阶段
- 管理页本身需要全量 definition + override 视图
- runtime bool/object/string/int 读取都可从同一个 snapshot 派生
- 便于统一版本号、审计信息与 typed resolver

缺点：

- 单次 miss 需要全量装载
- 配置量继续扩大后需要评估 snapshot 大小

#### 方案 D：本地缓存 + Redis pub/sub

优点：

- 多节点一致性最好

缺点：

- 当前阶段复杂度偏高
- 不应作为 Phase 1 的前置条件

#### 方案 E：本地缓存 + 短 TTL

优点：

- 最简单

缺点：

- 多节点一致性弱
- 仅靠 TTL 时“配置刚改完为何不生效”体验较差

### 推荐结论

当前推荐：

- Phase 1：`方案 C + singleflight + 显式本地失效`
- Phase 2：升级为 `方案 C + D`
- 临时 fallback 可接受 `E`，但不应长期只靠 TTL

理由：

- `Graft` 当前更偏单体/模块化后端，先把单进程 authority 与热点读问题解决最有效。
- 当前 system config 消费方既有按 key bool，也有 object default config，全量 snapshot 更便于统一 typed resolver。
- 将来多节点扩展时，可以在不改读 API 的前提下增加 Redis pub/sub。

## 5.3 缓存一致性规范

### 单节点

- DB commit 成功后，立即失效本地 config snapshot
- 下一次读通过 singleflight 回源重建 snapshot
- 修改失败不得改写缓存版本

### 多节点

- 首选：DB commit 成功后发布 Redis invalidation event
- 备选：短周期 version polling
- Redis 不可用时允许退化到短 TTL + 本地失效

### 生效语义

- `runtime-hot`：
  - 默认要求修改后尽快生效
  - 单节点目标：本地立即失效，下一次读取即新值
  - 多节点目标：数秒内完成失效传播
- `restart-required`：
  - 可更新 DB
  - 页面必须明确提示“已保存，重启后生效”
  - 后端不应在运行时假装热切换成功

### 顺序规则

- 顺序必须是：
  - 校验
  - 写 DB
  - 写审计
  - commit 成功
  - 失效本地缓存
  - 发布跨节点失效事件
- 不得在 DB commit 前先广播缓存失效

### 默认值变更

- definition default 变更属于 authority 变更
- 若影响 runtime-hot 读取，发布版本后必须清空或重建 snapshot

### 预热与降级

- 启动时可以不强制预热，优先 lazy load
- 若 system config 管理页或启动阶段已知很快会使用，可选择后台预热
- Redis 不可用时：
  - system config 本地缓存仍可继续工作
  - 不得因为 Redis 不可用而让系统配置读取整体不可用

### stale read

- 允许 stale read，但必须有上限
- 推荐上限：
  - 单节点 Phase 1：TTL 30s 内，但配置写后应显式失效，因此常态远低于 30s
  - 多节点 Phase 2：传播期目标 1s 到 5s，最坏不超过 30s

## 5.4 缓存 key 规范

Redis key 与逻辑命名统一约定：

- `graft:system-config:snapshot:v1`
- `graft:system-config:version:v1`
- `graft:system-config:key:{domain}:{group}:{item}`
- `graft:rbac:permission:user:{user_id}`
- `graft:menu:bootstrap:user:{user_id}`
- `graft:dashboard:summary:{scope}`
- `graft:monitor:server-status:trend:{window}`

说明：

- `system-config:snapshot:v1` 更适合作为 Redis 共享 snapshot 或调试定位名，不意味着 Redis 是 authority。
- 进程内缓存可沿用同一逻辑命名，但不要求与 Redis 物理一一对应。

### 不建议进入 Redis 的 key

- 单次请求级对象
- 日志分页结果
- 容器实时详情快照
- shell ticket / 临时 websocket ticket
- 仅在本节点短时复用、跨节点收益极低的本地 IO cache，例如 `mountUsageCache`

## 5.5 TTL 规范

| 类型 | 建议 TTL | 说明 |
|---|---|---|
| 系统配置 snapshot | 30s，本地显式失效优先 | authority 在 DB，TTL 只是兜底 |
| 权限 / 菜单聚合 | 30s 到 2m，配合显式失效 | 用户授权变更后应可主动失效 |
| 用户会话 | 按认证策略，不视作普通缓存 | session/token 与业务缓存分开治理 |
| 仪表盘统计 | 5s 到 30s | 允许轻微延迟，但需区分实时状态与聚合指标 |
| 日志列表查询 | 不建议缓存列表结果 | 可以缓存 filter metadata，不缓存分页数据 |
| 容器状态 | 1s 到 5s 以内，仅限聚合摘要 | 实时详情不应长缓存 |
| 模块运行时状态 | 5s 到 30s | health/status 可短缓存，异常应快速刷新 |
| 健康检查结果 | 5s 到 15s | 仅缓存探针结果，不缓存错误结论过久 |

原则：

- 缓存的是“可复用数据”，不是“错误对象”
- 实时状态与配置快照必须分开看待
- 错误响应默认不进入长 TTL 缓存

## 6. 系统配置加速设计

## 6.1 服务层设计

优先复用现有 authority，不另起炉灶。建议在 `server/modules/system-config` 内部收敛以下能力：

- `ConfigSnapshotProvider`
  - 负责装载全量 effective config snapshot
  - 合并 `configregistry` default 与 DB override
- `TypedConfigResolver`
  - 在 snapshot 基础上提供 bool / int / string / object 读取
  - 对外逐步替代过薄的 bool-only 读取
- `ConfigCache`
  - 进程内缓存 snapshot、版本号、过期时间
- `ConfigInvalidator`
  - 写入成功后本地失效
  - 预留跨节点失效入口
- `ConfigChangePublisher` / `Subscriber`
  - Phase 2 需要时接入 Redis pub/sub
- `ConfigVersion`
  - snapshot 版本号，可由时间戳、单调递增版本或 override hash 表示

如果实现时发现名称与现有仓库习惯不匹配，可以不强制新建同名类型，但职责必须完整存在。

## 6.2 读路径设计

目标读路径：

`模块运行时读取配置`
-> `TypedConfigResolver`
-> `进程内 snapshot`
-> `未命中/过期 -> singleflight 装载`
-> `registry default + DB override 合并`
-> `typed value 返回`
-> `记录 metrics / debug log`

关键要求：

- 所有运行时系统配置读取统一经 resolver，不允许模块直接查 `system_config` override 表。
- singleflight 必须在 snapshot 装载层，避免并发热点下重复查库。
- bool、string、int、object 的 typed parse 结果可附着在 snapshot 内，避免每次 JSON 反序列化。

## 6.3 写路径设计

目标写路径：

`校验配置项`
-> `写数据库`
-> `写审计日志`
-> `commit 成功`
-> `失效本地 snapshot`
-> `可选发布 Redis 失效事件`
-> `返回新版本号与 effective snapshot`

页面与 API 语义建议：

- 返回值中可附带：
  - `version`
  - `effective_source`
  - `runtime_apply_mode`
- 当前已有审计字段必须保留，不允许为了缓存改造绕过审计。

## 6.4 失效路径设计

### Phase 1 单节点

- `Update/Reset` 成功后直接调用本地 invalidator
- 下次读走 singleflight 重建

### Phase 2 多节点

- 在本地 invalidation 后发布：
  - `graft.system-config.invalidated`
  - payload 至少包含 `version`、`keys`、`updated_at`
- 订阅方收到后：
  - 若 payload 版本高于本地版本，则清理本地 snapshot

## 6.5 单节点方案

推荐作为第一落地点：

- system-config service 内新增本地 snapshot cache
- resolver 全部复用这个 cache
- scheduler / notification / bootstrap / container 统一改走缓存化 resolver
- 不要求先改造 Redis

## 6.6 多节点预留方案

为未来多节点保留但不提前复杂化：

- Redis pub/sub 仅负责 invalidation signal
- snapshot 主体仍以本地内存为第一读取来源
- Redis 故障不影响本地配置读取

## 6.7 前端展示建议

系统配置页面建议补充但保持简洁：

- 配置项是否 `运行时生效`
- 是否 `需重启`
- 当前 effective source：
  - default
  - override
- 最近加载版本
- 是否已覆盖默认值
- 高级信息中展示：
  - cache scope
  - runtime apply mode
  - version
- 可选“刷新运行时缓存”按钮，仅管理员可见

注意：

- 不要把缓存内部细节塞到主表单主视觉
- 用户关心的是“改完何时生效”，不是缓存术语本身

## 7. 分阶段实施计划

## Phase 1：单进程本地 snapshot + singleflight + 显式失效

- 改动文件范围
  - `server/modules/system-config/**`
  - `server/internal/moduleapi/**` 如需扩展 resolver 只读接口
  - `server/modules/container/**`
  - `server/modules/notification/**`
  - `server/internal/scheduler/**`
  - `server/modules/user/bootstrap.go`
- 后端改动
  - 收敛统一 config snapshot cache
  - 增加 singleflight
  - 写入后显式失效本地缓存
  - 容器模块收敛到统一 resolver，减少启动期默认值双轨
- 前端改动
  - 系统配置页增加“运行时生效 / 需重启 / 已覆盖默认值”提示
- OpenAPI 是否需要变化
  - 可选；若增加版本号或 runtime apply mode 展示字段则需要
  - 若仅后端内部优化，可先不变
- 测试方式
  - system-config service 单测
  - 并发 singleflight 单测
  - container / notification / scheduler 行为回归测试
  - DB 查询次数基准或 mock 断言
- 回滚方式
  - 保留旧 resolver 路径 behind internal switch 或最小回退 patch
  - 出现异常时禁用缓存层，回退为直接读取

## Phase 2：Redis pub/sub 或版本号轮询，多节点一致性

- 改动文件范围
  - `server/modules/system-config/**`
  - `server/internal/eventbus` 或 Redis 集成边界
- 后端改动
  - 增加 config invalidation event
  - 订阅端收到高版本后清空本地 snapshot
- 前端改动
  - 可选显示“最近同步时间 / 当前版本”
- OpenAPI 是否需要变化
  - 通常不需要
- 测试方式
  - 多节点模拟
  - Redis 不可用降级测试
- 回滚方式
  - 关闭 pub/sub，仅保留本地 TTL + 显式失效

## Phase 3：扩展到 RBAC/menu/dashboard/container runtime 等热点缓存

- 改动文件范围
  - `server/modules/rbac/**`
  - `server/internal/menu/**`
  - `server/internal/dashboard/**`
  - `server/modules/container/**`
- 后端改动
  - 把高频聚合读模型接入统一缓存规范
  - 明确哪些保留 request scoped，哪些进入 process local
- 前端改动
  - 菜单 / bootstrap 变更后的 UX 提示优化
- OpenAPI 是否需要变化
  - 视 debug / admin 能力而定
- 测试方式
  - 权限、菜单、dashboard 热点回归
  - 缓存失效后行为一致性测试
- 回滚方式
  - 按热点模块逐项关闭缓存

## Phase 4：指标、调试面板、缓存治理文档和测试门禁

- 改动文件范围
  - `server/**` 指标与日志
  - `ai-plan/design/**`
  - `.agents/skills/**`
  - `scripts/**` 如需治理检查
- 后端改动
  - 增加 cache hit/miss/load/invalidate metrics
  - 增加 admin debug endpoint 或只读诊断面
- 前端改动
  - 高级信息或调试面板入口
- OpenAPI 是否需要变化
  - 如新增 debug endpoint，则需要
- 测试方式
  - 指标暴露检查
  - 治理文档与 skill 校验
- 回滚方式
  - 调试能力与指标可以独立关闭

## 8. 不建议做的事情

- 不要缓存日志分页查询结果。
- 不要把容器实时 CPU、内存、即时状态缓存太久。
- 不要把系统配置 authority 完全依赖 Redis。
- 不要让模块绕过统一 `ConfigResolver` 直接查 system config override 表。
- 不要为了“立即生效”在每个模块里各写一套本地缓存。
- 不要把前端长期缓存为权限、菜单或系统配置 effective value 的 authority。
- 不要把 `restart-required` 配置伪装成 runtime-hot。
- 不要为了缓存命中率缓存错误对象或失败响应太久。

## 9. 验证清单

- 修改 `ops.container.runtime.enabled` 后，无需重启时容器访问能力是否按预期生效。
- 修改 `ops.container.actions.dangerous_enabled` 后，危险按钮与后端动作校验是否同步生效。
- 修改 `notification.source.scheduled_task_success.enabled` 后，手动任务成功通知是否按最新值生效。
- 高频读取系统配置时，数据库查询次数是否明显下降。
- 并发读取相同配置时，是否由 singleflight 合并加载。
- `ResolveDefaultConfig` 路径是否不再每次直接查库。
- Redis 不可用时，本地缓存方案是否仍可降级工作。
- 多节点场景下，配置修改后的失效机制是否可扩展。
- 标记 `restart-required` 的配置是否在 UI 与后端语义上保持一致，不再被误导为“立即生效”。
