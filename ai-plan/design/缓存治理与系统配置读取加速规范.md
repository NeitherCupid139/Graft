# 缓存治理与系统配置读取加速规范

## 1. 目标

这份文档定义 `Graft` 的仓库级缓存治理规则，用于约束：

- 何时应该缓存，何时不应该缓存。
- 缓存 authority 归谁，缓存机械层归谁。
- 何时必须复用现有 shared facility。
- 何时允许 module-local cache。
- 何时允许 Redis 仅作为 transport 或 authority-owned data store。

目标：

- 让热点读路径先做 authority-first 判断，再决定是否缓存。
- 收紧模块把 Redis 当作任意缓存后门的行为。
- 保持 `server/internal/**` shared helper 只承载缓存机械能力，不抢业务 authority。
- 让 system-config、scheduler、notification、container、bootstrap gating 等热点读路径遵循统一判断口径。

非目标：

- 不要求所有热点默认都缓存。
- 不把 Redis 设为默认缓存答案。
- 不为了“统一”发明无人复用的空壳 abstraction。
- 不把 `web`、Redis、临时 DTO 或 compatibility layer 提升为 authority。

## 2. 适用范围

本规范适用于：

- `server/internal/**` 下新增或修改的 shared cache mechanical layer。
- `server/modules/**` 下新增或修改的热点读路径。
- system-config、RBAC/menu/bootstrap、dashboard summary、notification gating、scheduler default config、container runtime gating 等读多写少路径。
- 任意打算引入本地缓存、singleflight、snapshot、TTL、Redis invalidation、Redis read-through 或 distributed cache 的后端改动。

本规范不替代：

- 根 `AGENTS.md` 的 startup、authority-first、validation、commit 与 closeout 规则。
- `server/AGENTS.md` 的后端边界、生命周期和验证真相。
- topic recovery 文档中的阶段历史、trace 和记忆材料。

## 3. Authority-First 决策模型

任何缓存任务在实现前都必须先回答四个问题：

1. 真正的 authority 是什么。
2. 这个读路径是否真的热。
3. 允许的 stale window 是多少。
4. 写入后如何失效或重建。

### 3.1 Authority 规则

- 缓存永远是 authority 的加速层，不是 authority 本身。
- authority 必须先定位到 canonical owner，再决定是否缓存。
- 若上游 authority 错了，先修 authority，不要先补缓存兼容层。
- `web` 不得成为 effective permission、menu、runtime config 或 system-config effective value 的 authority。
- Redis 不得成为 system-config、route/menu/permission、shared contract 或 bootstrap semantics 的 authority。

常见 authority 链示例：

- system config
  - `server/internal/configregistry/**` definition
  - `server/modules/system-config/**` override persistence + effective resolver
  - downstream runtime consumers
- scheduler default config
  - `server/internal/cronx/**` job definition
  - `server/modules/system-config/**` effective config resolver
  - `server/internal/scheduler/**` runtime consumer
- bootstrap/menu gating
  - menu/permission registry + system-config authority
  - `server/modules/user/bootstrap.go` consumer
- monitor trend
  - `server/modules/monitor/**` authority-owned runtime sample pipeline
  - Redis only as trend storage medium

### 3.2 Hotspot 证明要求

只有满足至少一项时，才把路径视为缓存候选热点：

- 单次请求或短时间内重复读取同一 authority。
- 同一 authority 被多个 consumer 高频读取。
- miss 成本明显高于允许的本地内存或短期 stale 成本。
- 读路径处在登录 bootstrap、通知发布、调度运行、容器动作 gating、概览聚合等高频执行面。

以下场景默认不应优先缓存：

- 实时状态、瞬时健康检查、日志分页结果、容器即时详情。
- 读频率低且回源成本可接受。
- authority 仍不稳定、字段语义尚未收口的路径。

## 4. 当前仓库共享机制现状

当前仓库已经存在少量稳定共享机制，但没有“全仓通用缓存框架”。

### 4.1 已存在且应优先复用的共享机制

- `server/internal/configregistry/**`
  - 启动期只读 definition registry。
  - 负责 system-config 默认值、schema、元数据和 runtime apply semantics declaration。
- `server/internal/moduleapi/SystemConfigResolver`
  - system-config effective read 的跨模块稳定能力边界。
  - 当前稳定读取面包括 `IsBooleanConfigEnabled(...)` 与 `ResolveDefaultConfig(...)`。
- `server/modules/system-config/service.go`
  - system-config authority-owned process-local snapshot cache。
  - 已包含 `singleflight`、显式本地失效、Redis best-effort invalidation transport 和调试态。
- `server/internal/redisx/**`
  - core-owned Redis transport opener。
  - 只负责创建和验证 Redis client，不提供缓存 authority。
- 启动期只读 registry
  - `server/internal/menu/**`
  - `server/internal/cronx/**`
  - `server/internal/dashboard/**`
  - 这些是 registry / definition cache，不是通用 runtime data cache。

### 4.2 已存在但仍属 authority-owned 或 module-local 的机制

- `server/modules/container/mount_usage.go`
  - module-local process-local TTL cache。
  - 适用于本地磁盘扫描结果，不应外推成仓库通用 Redis cache 模板。
- `server/modules/monitor/module.go`
  - authority-owned Redis trend storage。
  - Redis 在这里承载 monitor trend 数据本身，不是“任意模块都可以直接照抄”的缓存后门。

### 4.3 当前仓库结论

- 仓库已经有共享缓存机械能力的局部实现。
- 仓库当前没有需要强推的统一 generic cache package。
- 后续若没有新的重复机械需求，默认不新增 `server/internal` 通用缓存包。

## 5. 缓存策略分类

每个目标读路径必须归入且只归入一个 primary strategy：

- `no cache`
- `request-scoped cache`
- `process-local cache`
- `startup-immutable cache`
- `redis-authority-data`
- `redis-invalidation-transport`

### 5.1 No Cache

适用：

- 实时状态优先于低延迟。
- 数据波动快，stale read 风险高。
- 读频率不足以证明缓存收益。

示例：

- 容器实时状态。
- 日志分页结果。
- 一次性诊断查询。

### 5.2 Request-Scoped Cache

适用：

- 单请求内会重复解析同一份数据。
- 不需要跨请求共享。

限制：

- 生命周期必须严格限定在单请求。
- 不承担跨请求一致性职责。

### 5.3 Process-Local Cache

适用：

- authority 仍在当前服务可回源的 DB / registry / stable contract。
- 读频繁、写少或写可显式失效。
- 允许极短 stale window。

要求：

- 必须定义 stale window 或显式“仅靠失效，不靠 TTL”。
- miss 回源若成本明显，优先配合 `singleflight` 或等价并发合并。
- 写入成功后必须定义明确的 invalidation path。

### 5.4 Startup-Immutable Cache

适用：

- definition、registry、schema、静态 metadata。
- 启动后只读，进程重启或重新注册才变化。

要求：

- 不要把启动期 registry 误描述为 runtime mutable data cache。
- 若后续要承载 runtime mutable semantics，必须升级到正确 authority 链，而不是偷用 registry。

### 5.5 Redis-Authority-Data

适用：

- 模块本身拥有这份 Redis 数据的 authority。
- 数据天然是短期样本、共享聚合、队列、窗口统计或 trend store。
- 多节点共享读取是问题本体，而不是副作用。

限制：

- 必须由 authority owner 明确拥有 key space、retention、serialization 和 query semantics。
- 不得把此模式类比为“任何业务模块都可直接往 Redis 放缓存”。

### 5.6 Redis-Invalidation-Transport

适用：

- authority 仍在 DB / registry / stable resolver。
- 需要跨节点传播本地缓存失效事件。

限制：

- Redis 只承载 invalidation signal，不承载 canonical value authority。
- publish / subscribe 失败时必须有明确降级行为。

## 6. Shared Mechanism Policy

### 6.1 什么时候必须复用现有 shared facility

以下场景必须先复用既有 shared facility，而不是各自重写缓存：

- 读取 system-config effective value
  - 必须通过 `server/modules/system-config/**` + `server/internal/moduleapi.SystemConfigResolver`
  - 不得绕过 resolver 或直查 override 表
- 读取 scheduler default config
  - 必须通过 `server/internal/scheduler/**` 已接入的 default config resolver
- 消费系统配置布尔开关、默认对象配置、bootstrap gating
  - 必须优先复用统一 resolver / snapshot provider
- 使用 Redis client
  - 必须经由 core runtime 注入的 `ctx.Redis` / `server/internal/redisx/**`
  - 不得自行创建第二套临时 Redis authority path

### 6.2 什么时候允许 module-local cache

仅当同时满足以下条件时允许：

- 该缓存只服务本模块局部热点。
- 其 authority 仍明确归当前模块或其已声明依赖。
- 没有足够证据证明应提升为 `server/internal` shared helper。
- 缓存键、payload、生命周期明显依赖本模块业务语义。

示例：

- `server/modules/container/mount_usage.go` 的本地 TTL cache。

### 6.3 什么时候允许新增 `server/internal` shared cache mechanical layer

只有当 authority discovery 证明以下条件同时成立时才允许：

- 至少两个独立 authority owner 遇到相同缓存机械问题。
- 现有 authority-owned helper 无法继续承载该机械层而不泄漏业务语义。
- 新 shared package 可以只抽取机械能力，不引入业务 authority。
- API 可以稳定落在 `server/internal/**`，且不需要 module business key、DTO、override table schema 或 module-specific policy。

允许抽取的内容仅限：

- process-local cache 容器
- `singleflight` / request coalescing helper
- invalidation bus / signal envelope
- 时钟、TTL、测试钩子

不得抽取或塞入 shared package 的内容：

- override table access
- module business key naming
- 业务 DTO / 业务状态枚举
- 某模块专属的 effective value 组装逻辑
- 第二套 config authority

### 6.4 当前仓库决策

当前仓库不新增新的 generic shared cache package。

理由：

- 已存在的热点没有证明需要第二个共享 mechanical layer 才能推进。
- `system-config` 的 snapshot cache 是 authority-owned helper，不适合被误抽成通用“全仓缓存框架”。
- `monitor` 的 Redis trend storage 和 `container` 的 mount TTL cache 都带明显 authority / domain 语义，不适合统一抽象。

## 7. Direct Redis Usage Policy

### 7.1 允许的 direct Redis usage

仅允许以下三类：

- core transport
  - `server/internal/redisx/**` 负责创建与探活 Redis client
- authority-owned Redis data
  - 如 `server/modules/monitor/**` 的趋势样本存储
- best-effort invalidation transport
  - 如 `server/modules/system-config/**` 的 snapshot invalidation signal

### 7.2 默认禁止的 direct Redis usage

以下场景默认禁止：

- 模块把 Redis 当作任意读路径的临时缓存后门。
- 模块绕过统一 resolver / snapshot provider，直接把 DB authority 镜像进 Redis。
- 用 Redis 保存 effective permission、menu、bootstrap、system-config 值并让其成为 consumer 读取 authority。
- 在没有 shared helper 审查或 skill 判定的情况下，新模块自行创建 ad-hoc cache key、TTL 和 invalidation 方案。
- 把 `web` 推导结果或 display-only state 写回 Redis 再反向影响 authority。

### 7.3 例外处理

若任务必须新增 direct Redis usage，closeout 必须记录：

- authority owner
- Redis 用途是 `authority-data` 还是 `invalidation-transport`
- 为什么不能复用现有 shared facility
- key space / retention / invalidation / degrade path
- 何时需要收敛成 shared helper 或上游 authority repair

## 8. System-Config 专项规则

### 8.1 Authority

system-config authority 固定为：

- `server/internal/configregistry/**`
- `server/modules/system-config/**`

所有运行时 effective read 都必须经过：

- `server/modules/system-config/service.go`
- `server/internal/moduleapi.SystemConfigResolver`
- 或其明确注册的 resolver / snapshot provider

### 8.2 允许的缓存形态

system-config 当前允许且推荐：

- process-local full snapshot cache
- `singleflight` 合并并发 miss
- `Update/Reset` 成功后的显式本地失效
- Redis best-effort invalidation transport 用于多节点传播

### 8.3 禁止事项

- 禁止模块直查 system-config override 表。
- 禁止为单个 consumer 再造第二套 per-key Redis cache。
- 禁止把 Redis 写成 system-config authority。
- 禁止让 `web` 根据页面逻辑推断 runtime apply semantics。

### 8.4 Runtime Apply Semantics

- `runtime_apply_mode` 是 runtime apply semantics 的 canonical authority。
- `effective-source` 继续只表达 `default` / `override`。
- 若某配置仍需重启，不得因存在缓存而伪装成 runtime-hot。

## 9. 新热点接入要求

当发现新的热点读路径时，必须先完成以下判断：

1. authority owner 是谁。
2. 当前是否已有 shared facility。
3. primary strategy 是什么。
4. stale window 是否允许。
5. invalidation / reload 谁负责。
6. 是否需要多节点一致性。
7. 是否需要登记为 shared asset 或显式 deferred。

若路径是热点且当前未治理，只允许三种结果：

- `reused-existing`
  - 复用已有 shared facility
- `added-cache`
  - 在正确 authority owner 下补缓存
- `deferred-with-reason`
  - 明确写出暂不缓存的原因、保护措施和触发条件

不允许静默留下“疑似热点但无人负责”的状态。
