# 本地化与 i18n 治理规范

## 1. 目标

这份文档定义 `Graft` 前后端本地化的长期治理边界。

目标：

- 保持 `server/internal/i18n.Service` 作为后端唯一 i18n facade。
- 保持 embedded locale YAML 作为 backend 面向用户可见本地化文案的 canonical truth。
- 保持 locale 资源的 embed、load、validate、freeze 与 registry construction 只由 `server/internal/i18n` 集中拥有。
- 保持 `web/src/locales/**` 作为前端 locale 状态、消息聚合和查找入口。
- 将 backend 可维护展示文案迁移到 locale 资源文件，并禁止在生产 Go 中新增用户可见硬编码本地化文案。
- 让 message key、namespace、fallback、JSON Schema `x-i18n` 与菜单/权限/配置元数据拥有清晰 owner。
- 支持分阶段迁移，不做一次性全量重写。

非目标：

- 不把 `go-i18n` 暴露给业务模块、`configregistry`、`httpx`、`moduleapi` 或 web。
- 不把前端长期 UI 文案所有权放到 server。
- 不改变现有 HTTP 错误响应格式、`LocaleHeader` / `Accept-Language` 解析逻辑或 OpenAPI wire contract。
- 不把 fallback 文案提升为第二套长期本地化真相。
- 不允许在生产 Go 中保留新的用户可见本地化 fallback，除非按例外治理显式登记。
- 不允许业务模块自行加载 locale 文件或在 `server/internal/i18n` 外构造平行 message registry。

## 2. 分层边界

### 2.1 Server

`server/internal/i18n.Service` 是 server 唯一项目级 facade。

外部调用方只允许消费项目概念：

- `Namespace`
- `LocaleTag`
- `MessageKey`
- `MessageResource`
- `Registration`
- `LookupRequest`
- `RegisterMessages`
- `Lookup`
- `Message`
- `ResolveLocale`
- registry diagnostics such as `RegisteredMessageKeyIDs` / `RegisteredMessageResources`

业务模块、`configregistry`、`httpx`、`moduleapi` 不得直接依赖底层 catalog provider、YAML loader 或
`github.com/nicksnyder/go-i18n`。

`Service` 内部可长期支持两类 provider：

- 当前 map catalog provider。
- 未来可选 `go-i18n` provider。

provider 切换不得改变 facade API、namespace/key 归属、fallback 顺序、duplicate-key 规则、freeze 语义或诊断接口。

### 2.2 Web

`web/src/locales/**` 拥有前端 locale 状态、回退语言、消息聚合入口和请求 locale 透传。

模块消息源由 `web/src/modules/<name>/locales/**` 拥有。模块 key 不得复制到 root catalog 形成第二套真相。

页面、壳层和模块不得绕过中心聚合入口直接拼装第二套 messages。

locale 切换默认只允许触发响应式文案与组件 locale 更新，不允许把整棵页面树 remount 当作 i18n 刷新手段。

禁止示例：

- 给 `router-view`、壳层 layout、header 操作区、整页容器绑定 `:key="locale"` 或等价 locale-key，借此强制卸载重建当前页面树。
- 依赖 locale 切换时的 `onMounted` / `setup` 重跑来刷新公告、通知、tab、breadcrumb、表格或其它可见 UI 状态。

若 locale 变化后需要刷新局部派生数据，应在该局部使用响应式 `t(...)`、computed、`watch(locale)` 或明确的数据刷新逻辑，而不是扩大为全局 remount。

### 2.3 Cross-Boundary

跨边界可见文案优先传递稳定 key：

- 菜单标题：`title_key` 为 authority；生产 Go 不默认保留 `title` fallback。
- 错误显示：`messageKey` 为主，`message` 仅在现有 HTTP wire shape 必须保留时作为结果字段存在。
- 权限展示：`display_key` / `description_key` 为 authority；生产 Go 不默认保留 `display` / `description` fallback。
- 系统配置：`domain_key`、`group_key`、`group_description_key`、`title_key`、`description_key` 和 JSON Schema
  `x-i18n` 为 authority；生产 Go 不默认保留 label/title/description fallback。

server 只负责后端返回描述元数据、必要错误消息和 key/fallback；web 负责前端页面长期 UI 文案。

前端不得把 locale catalog 旁路成第二套双语对象真相，包括但不限于：

- `'zh-CN': '...'` / `'en-US': '...'` 直接内嵌在生产 TS/Vue UI prop 对象中。
- `[LOCALE.ZH_CN]: '...'` / `[LOCALE.EN_US]: '...'` 这类 computed property 双语对象。

上述形态都属于用户可见硬编码，必须进入 `web/src/locales/**` 或 `web/src/modules/<name>/locales/**` 再通过 key 或 lookup 使用。

前端导航与页面标题的英文本地化补充规则：

- 中文可继续保留 `xx管理`、`模块运行时` 等后台语义化命名。
- 英文导航、breadcrumb、tab 与页头可使用更短的产品名词，不要求逐字保留 `管理/Management`。
- 当页面已通过 eyebrow、source 或壳层分组表达所属域时，英文 page/tab title 默认避免再拼接父级域前缀，例如优先使用 `Containers`、`Modules`、`Roles`，而不是 `Operations - Container Management`。
- `title_key` 仍是标题语义的唯一稳定真相；若英文标题需要收短，应优先改 locale value，不新增并行 key。

## 3. Server Catalog 规范

### 3.1 资源路径

当前 backend locale 资源 ownership 策略：

```text
server/internal/i18n/locales/
  core.zh-CN.yaml
  core.en-US.yaml
  display.zh-CN.yaml
  display.en-US.yaml

server/internal/moduleruntime/locales/
  zh-CN.yaml
  en-US.yaml

server/modules/<module>/locales/
  zh-CN.yaml
  en-US.yaml
```

其中：

- `server/internal/i18n/locales/*.yaml` 只用于 `core`、`display` 等 i18n infrastructure 自有 namespace。
- `server/modules/<name>/locales/*.yaml` 用于 module-owned namespace；locale resource 的物理归属随模块目录维护。
- `server/internal/moduleruntime/locales/*.yaml` 用于 internal runtime/platform component 自有 namespace。
- owner package 可以自己 `go:embed locales/*.yaml`，但只允许暴露只读 embedded resource descriptor。
- module 与 internal owner 不得自行解析 YAML、校验 locale/key、执行 duplicate 检查、维护 registry、调用 `Freeze`，这些职责仍由 `server/internal/i18n` 独占。
- 不允许 runtime filesystem scan 任意目录；允许的 embedded 资源集合必须由 owner package 编译期显式声明，并由 runtime 在 `Freeze` 前统一交给 `server/internal/i18n` 注册。

### 3.2 文件命名

`server/internal/i18n/locales/*.yaml` 文件名格式固定为：

```text
{namespace}.{locale}.yaml
```

- `namespace` 必须等于 `i18n.Registration.Namespace`。
- `locale` 必须是当前支持的 `LocaleTag`，现阶段只接受 `zh-CN` 与 `en-US`。
- 文件名解析出的 namespace 和 locale 是注册真相，文件内容不得重复声明第二套 namespace/locale。

module-owned 与 internal runtime-owned locale 文件默认使用：

```text
{locale}.yaml
```

- 这些 owner-local 资源的 `namespace` 由 owner package 暴露的只读 resource descriptor 提供。
- `locale` 仍必须是当前支持的 `LocaleTag`。
- owner-local 文件路径是物理归属信息，不是第二套 runtime loader 入口。

### 3.3 内容格式

第一阶段采用 flat key-value YAML：

```yaml
systemConfig.domains.dashboard: 工作台配置
```

选择 flat YAML 的原因：

- 现有 key 已经是点分层稳定 contract。
- 可直接转换为 `[]i18n.MessageResource`。
- 避免 nested YAML 与点号 key 的双重结构语义。
- YAML 支持注释，更适合人工维护和翻译审校。

JSON 不适合作为主格式，因为它不支持注释。TOML 可作为 go-i18n 未来互操作格式评估项，但不是当前默认。

### 3.4 Key 命名

继续沿用现有 key 风格：

- `systemConfig.domains.*`
- `systemConfig.groups.*`
- `systemConfig.groupDescriptions.*`
- `systemConfig.items.*.title`
- `systemConfig.items.*.description`
- `systemConfig.fields.*.title`
- `systemConfig.fields.*.description`
- `systemConfig.options.*`
- `menu.*.title`
- `rbac.permissionCatalog.*.display`
- `rbac.permissionCatalog.*.description`

新增 key 必须能从前缀看出 owner，不得用展示文案本身替代稳定 key。

## 4. Server Loader 设计

loader 与 resource registrar 都是 `server/internal/i18n` 的内部基础设施，不是业务 API，也是 backend locale 资源唯一注册入口。

建议结构：

```text
server/internal/i18n/loader/
  embedded.go
  yaml.go
  validate.go
```

职责：

- 支持 `go:embed`、测试用 `fs.FS`，以及 runtime 提供的 raw embedded resources。
- 从 `server/internal/i18n/locales/*.yaml` 收集 i18n infrastructure 自有资源。
- 从 owner package 暴露的 resource descriptor 收集 module-owned 与 internal runtime-owned 资源。
- 对 `server/internal/i18n/locales/*.yaml` 从文件名解析 namespace 和 locale。
- 对 owner-local resources 信任 descriptor 提供的 namespace、locale 与 logical source，再对其执行统一校验。
- 解析 flat YAML 为 `i18n.Registration`。
- 保留文件内重复 key、空 key、空文案、unsupported locale、非法文件名的错误。
- 调用现有 `Service.RegisterMessages`，复用 duplicate-key 与 freeze 前注册校验。
- 支持 runtime 在模块 `Register` 前完成 locale resource 预注册，再在启动期统一 `Freeze`。
- 支持 `testing/fstest.MapFS` 做 loader 单测。

loader 不负责：

- 运行时扫描任意目录。
- 改写 `LookupRequest`。
- 绕过 `RegisterMessages`。
- 为业务模块返回 loader 专属 DTO。
- 允许 module 自己决定另一套 locale 资源目录、注册时机或 freeze 时机。
- 让 `server/internal/i18n` 反向 import `server/modules/*`。

## 5. Template 与 go-i18n

`LookupRequest.TemplateData` 是长期预留字段。

阶段策略：

- Phase 1 可保持无复杂模板渲染，或在 map provider 内部用最小 `text/template` 支持命名变量。
- 需要 plural rules、复杂模板、翻译平台导入导出或更多 locale 后，再评估 `go-i18n` provider。
- 未来接入 `go-i18n` 时，`LookupRequest.TemplateData` 映射到 `LocalizeConfig.TemplateData`。
- go-i18n message ID 使用 server canonical ID，即 `composeMessageID(namespace, key)` 的结果。
- `MessageKey`、`Namespace`、`LocaleTag` 仍保留项目自有类型。

go-i18n 不得作为模块直接依赖，不得进入 `configregistry`、`httpx`、`moduleapi` 或 web。

## 6. 迁移阶段

### Phase 0: 盘点和分类

- 扫描 `RegisterXXXMessages`、`[]i18n.Registration`、`[]i18n.MessageResource`、`LocaleZHCN` / `LocaleENUS`、
  `RegisterMessages`、`MessageKey`。
- 分类为平台错误、菜单/权限/模块标题、系统配置、通知/公告/任务展示、JSON Schema `x-i18n`、fallback 文案。
- 输出迁移清单和优先级。

### Slice 1: raw embedded resource registration

- 新增 `server/internal/i18n` 从 raw embedded resources 注册的入口。
- 保持 `i18n.Service` facade、当前 map catalog 与 `server/internal/i18n/locales/*.yaml` 现状可继续工作。
- runtime 在模块 `Register` 前统一完成 locale resource 预注册。
- 补 i18n loader / registrar 单测。

### Slice 2: 低风险模块试点

- 选择 `announcement` 或 `container` 作为首个 module-owned locale 迁移样例。
- owner package 自己 `go:embed locales/*.yaml`，只暴露只读 resource descriptor。
- runtime 统一注册这些资源；module `registerMessages()` 继续只校验 key 已存在。
- 删除对应集中目录资源，不保留长期双来源。

### Slice 3: 剩余业务模块迁移

- 迁移 `audit`、`container`、`monitor`、`rbac`、`scheduler`、`system-config`、`user`。
- `system-config` 按 module-owned locale 处理，不再继续留在 `server/internal/i18n/locales/`。
- 默认对 compile-time compiled modules 全量注册 locale resources，不以 enable allowlist 过滤注册集。

### Slice 4: internal runtime/platform 迁移

- 将 `module-runtime` locale 从集中目录迁到 `server/internal/moduleruntime/locales/*.yaml`。
- 保持 `core` / `display` 继续由 `server/internal/i18n/locales/*.yaml` 拥有。
- 如后续 `httpx`、`logger` 等 internal platform component 需要 owner-local locale，沿相同模式处理。

### Slice 5: 治理与阻断规则收尾

- 更新 design、skill、public recovery 与相关 scanner / CI 脚本。
- 阻止新增 `server/internal/i18n/locales/modules/*.yaml`。
- 迁移完成后不再接受集中 module locale 目录作为长期真相。

### Phase 5: go-i18n provider 评估

- 仅在 plural/template/翻译平台工作流成为真实需求时接入。
- 接入必须隐藏在 `i18n.Service` 或 internal provider 内部。
- 2026-06-18 当前结论：`go-i18n` 暂不引入。
  - 现状 `server/internal/i18n.Service` 仍以 facade + map catalog + embedded flat YAML loader 满足当前 server 需求。
  - `LookupRequest.TemplateData` 仍是预留桥接字段，当前 server 侧没有已落地的 plural rules、复杂模板渲染或翻译平台导入导出流程。
  - Phases 1-4 已将当前优先级内的 server-side locale 资源迁到 embedded YAML，并保留 `RegisterMessages`、`Freeze`、
    `RegisteredMessageResources`、fallback 与 diagnostics 语义不变。
  - 因此现在引入第二个 provider 只会增加实现分叉、测试面和 provider 泄漏风险，不能提供已被证明需要的收益。
- 未来重新开启 provider 评估的触发条件必须至少满足一项，并以新的 bounded batch 处理：
  - 需要 locale-aware plural rules，且不能被当前稳定 key + 文案资源模型安全表达。
  - 需要面向调用方稳定暴露命名模板插值，且最小内部模板桥接不足以满足需求。
  - 需要与翻译平台或批量翻译工作流做受治理的导入导出，而 flat YAML 维护成本已经成为真实阻塞。
  - 需要新增超出 `zh-CN` / `en-US` 的 locale，并证明现有 provider 的维护或校验成本不可接受。
- 若未来重启评估，必须同时满足这些治理约束：
  - 业务模块、`configregistry`、`httpx`、`moduleapi` 与 web 仍只依赖 `server/internal/i18n.Service` facade。
  - `Namespace`、`LocaleTag`、`MessageKey`、`Registration`、`LookupRequest`、`Freeze`、
    `RegisteredMessageResources` 与当前 map-catalog compatibility 不得被破坏。
  - provider 切换前必须补足针对 plural/template/workflow 真实场景的直接验证，而不是仅以库能力假设驱动接入。

## 7. 治理校验

server 侧必须逐步建立：

- `zh-CN` 和 `en-US` key 集合一致。
- locale 文件 key 不为空。
- value 不为空或全空白。

## 8. 当前例外台账

2026-06-18 当前 archive-readiness 审计结论必须以代码与验证事实为准，而不是沿用历史 closeout：

- 当前 backend 无登记中的生产 Go 用户可见本地化硬编码例外。
- 当前 frontend 无登记中的生产 TS/Vue 双语 UI 硬编码例外。
- `audit` built-in target label 已收口为 stable locale key + embedded locale YAML：
  - `server/modules/audit/storeent/repository.go` 仅保留 `TargetType -> locale key` 技术映射。
  - `audit.target.user`
  - `audit.target.role`
- `audit.target.permission`
- `audit.target.audit`
- `audit.target.serverStatus`
- `audit.target.auth`
  - canonical owner 是 `server/modules/audit`，对应 locale resource 已落在 `server/modules/audit/locales/*.yaml`。
  - repository/store 层不得重新引入用户可见中文/英文 fallback；缺 key 时应暴露稳定 key 缺失或空值，而不是回退到新的硬编码文案。
- `permission.Item{Name, Description}` 不得再在生产注册源中保留用户可见英文 fallback；如果当前 API 仍需要 display text，必须通过 `DisplayKey` / `DescriptionKey` 和 `server/internal/i18n.Service` 解析得到。
- core dashboard/runtime 不得再以 Go 英文 copy 充当 Widget / Action 或 shell-derived homepage shortcut 的长期真相；已有 title/description/label key 时，值必须来自 embedded locale YAML。
- `web/src/store/modules/tabs-router.ts` 这类壳层状态不得内嵌 `工作台 / Workspace` 双语对象；home/workbench 可见标题必须复用 locale key 或 catalog lookup。
- `bun run lint:i18n` 必须拦截 plain locale object 与 computed-property locale object 两种双语硬编码形态。

除显式重新登记的例外外，backend 不应继续保留新的生产 Go 用户可见本地化硬编码。
- 同 namespace + locale 下不允许重复 key。
- 生产 Go 不得新增用户可见硬编码本地化 copy；仅允许技术标识、稳定 key、route/path、resource 名与日志/审计内部标识保留为 Go 字符串。
- 菜单、Widget、QuickLink、Retention Job、Cron Action、Explorer/Permission/ConfigDefinition 的可见字段默认必须来自 locale resource。
- 临时例外必须记录文件、字段、原因、移除条件与验证范围。
- 新模块必须同时提交 locale YAML、稳定 key、`zh-CN` / `en-US` 覆盖与直接测试。
- Go 中引用的 `TitleKey`、`DescriptionKey`、`GroupKey`、`DomainKey`、`LabelKey` 能在 catalog 中找到。
- JSON Schema `x-i18n.titleKey`、`descriptionKey`、`enumLabels.*.labelKey`、`descriptionKey` 能找到。
- `RegisterMessages` 重复注册失败。
- `Freeze` 后注册失败。
- `Lookup` fallback 行为保持兼容。
- unused key 初期只 warning，不直接 fail。

web 侧继续通过 `bun run lint:i18n` 与 `bun run check` 执行 strict key-first governance。
