# 通知中心设计

本文档冻结 Notification Center Phase0 架构与实施边界。后续进入 Phase1 时，`server/modules/notification`、
`server/internal/moduleapi.NotificationPublisher`、OpenAPI、`web/src/modules/notification` 与 shell
`Notice.vue` 的实现必须以本文为 canonical authority。

当前阶段以 `server/modules/notification`、`server/internal/moduleapi.NotificationPublisher`、
`web/src/modules/notification` 和 System Config 中的 `notification.*` 配置域作为实现真相，不把通知中心扩展成资源监控告警平台。

## 1. Notification 与 Alert 的边界

Notification Center 只处理用户可处理事件，必须同时满足：

- 需要用户查看。
- 需要用户处理。
- 能跳转到业务上下文。

典型 Notification：

- Audit 登录失败。
- Audit 权限拒绝。
- Audit 高危审计事件。
- Scheduler 任务失败。
- 后续 Phase2 的高风险配置变更、模块 degraded/disabled 等可处理运行态事件。

Alert 面向运维告警，不属于 Notification Center MVP。典型 Alert：

- CPU 使用率过高。
- 内存使用率过高。
- 磁盘使用率过高。
- PostgreSQL Down。
- Redis Down。
- 资源类监控阈值告警。

Alert 需要冷却策略、恢复通知、升级机制和值班通知，后续必须独立设计 Alert / Monitoring authority。Notification Center 可以在未来接收 Alert 系统发布的“需要用户处理的站内通知”，但不能拥有资源阈值、依赖健康、告警升级或恢复通知的判定规则。

MVP 明确排除：

- CPU 使用率过高。
- 内存使用率过高。
- 磁盘使用率过高。
- PostgreSQL Down。
- Redis Down。
- 资源类监控阈值告警。
- 外部通知渠道。
- SSE / WebSocket。
- 用户订阅规则。
- 快捷动作执行。

## 2. Notification Center 的能力定位

Notification Center 是平台业务模块，不属于 `server/internal` core runtime。推荐归属：

```text
server/modules/notification
```

定位原因：

- 它有独立业务数据、权限、OpenAPI、页面和保留期治理，符合 Graft 业务模块边界。
- 它不是数据库、日志、HTTP、事件总线、权限注册器、服务容器等 core runtime primitive。
- 它需要给 Audit、Scheduler 等模块提供稳定发布接口，因此跨模块能力应通过 `server/internal/moduleapi` 暴露，而不是让来源模块 import notification 内部实现。
- 它应支持模块注册事件来源，但事件判定 authority 留在来源模块。

Notification Center 负责：

- 持久化通知事实与用户投递状态。
- 提供通知列表、未读统计、标记已读、全部已读、删除当前用户投递记录。
- 实现 `moduleapi.NotificationPublisher` 并注册到服务容器。
- 在 System Config 注册 `notification.*` 配置域，控制通知总开关、来源开关、站内投递开关与展示默认值。
- 治理 category、severity、target_type、navigation_kind typed contract。
- 为前端 shell 顶部铃铛和 `/notifications` 页面提供统一 API。
- 执行保留期清理 Job。
- 作为全局收件箱通过顶部通知入口进入，不注册左侧业务菜单。

Notification Center 不负责：

- 判断 Audit、Scheduler、Monitor、SystemConfig、ModuleRuntime 事件是否成立。
- 采集资源指标或依赖服务健康状态。
- 外部渠道发送。
- 告警升级、冷却、恢复通知和值班通知。
- 用户订阅规则。
- 快捷动作执行。

## 3. server/modules/notification 模块职责

推荐后端目录结构：

```text
server/modules/notification/
├─ contract/
│  ├─ permission.go
│  ├─ category.go
│  ├─ severity.go
│  ├─ target.go
│  └─ navigation.go
├─ migrations/
├─ store/
├─ module.go
├─ service.go
├─ publisher.go
├─ route_registration.go
├─ mapper_http.go
├─ module_registration.go
└─ retention.go
```

职责划分：

- `contract/permission.go`：定义 `notification.view`、`notification.read`、`notification.manage`。
- `contract/category.go`：定义 `SECURITY`、`TASK`、`CONFIG`、`OPERATIONS`、`SYSTEM`。
- `contract/severity.go`：定义 `info`、`warning`、`error`、`critical`。
- `contract/target.go`：定义 `USER`、`ROLE`、`PERMISSION`、`SYSTEM`。
- `contract/navigation.go`：定义 `AUDIT_INCIDENT`、`AUDIT_LOG`、`SCHEDULER_RUN`、`SYSTEM_CONFIG_ITEM`、`MODULE_RUNTIME_ITEM`。
- `migrations/`：承载 notification 双表迁移。
- `store/`：承载 notification 模块内部 repository interface 与 Ent/PostgreSQL 实现。
- `module.go`：模块生命周期、依赖声明、权限和路由接入入口。
- `service.go`：列表、未读计数、已读、全部已读、删除、权限过滤与事务边界。
- `publisher.go`：实现 `moduleapi.NotificationPublisher`，负责校验、幂等、落库和投递 fan-out。
- `route_registration.go`：注册 MVP OpenAPI 对应路由。
- `mapper_http.go`：处理 HTTP DTO 与 service model 的映射。
- `module_registration.go`：compile-time module registration。
- `config.go`：注册 System Config `notification.*` 配置域，并为 publisher 提供来源开关解析。
- `retention.go`：保留期清理 Job。

边界规则：

- 来源模块不得 import `server/modules/notification` 内部实现。
- Audit / Scheduler 只能依赖 `server/internal/moduleapi.NotificationPublisher`。
- notification 模块负责实现 publisher 并注册到服务容器。
- 发布通知失败不得阻断原业务流程，只记录 app log。
- notification 模块不得复制 Audit、Scheduler、Monitor 的事件判定规则。

## 4. server/internal/moduleapi.NotificationPublisher

跨模块接口路径固定为：

```text
server/internal/moduleapi/notification.go
```

`NotificationPublisher` 是跨模块 stable capability，只表达“发布一条站内通知”的意图。它不暴露 notification 模块内部 service、store、Ent entity、migration 或 HTTP handler。

推荐职责：

- 接收来源模块传入的标题、正文、级别、分类、来源、目标、导航、幂等键和事件时间。
- 校验 typed contract。
- 将 `target_type` 解析为最终用户投递。
- 持久化 `notification_events` 与 `notification_deliveries`。
- 返回投递结果，便于来源模块记录 bounded log。

发布方职责：

- 判断事件是否应该发布通知。
- 提供稳定的 `source_module`、`event_type`、`resource_type`、`resource_id`、`occurred_at`。
- 提供 `navigation_kind` 与 `navigation_payload`，但不自行拼接前端 URL。
- 对发布失败保持业务流程继续，只记录 app log。

## 5. 数据库双表模型

MVP 固定采用双表：

```text
notification_events
  id
  title_key
  title
  message_key
  message
  severity
  category
  source_module
  event_type
  resource_type
  resource_id
  resource_name
  navigation_kind
  navigation_payload jsonb
  metadata jsonb
  dedupe_key
  occurred_at
  expires_at
  created_at

notification_deliveries
  id
  event_id
  recipient_user_id
  target_type
  target_ref
  read_at
  deleted_at
  created_at
```

`notification_events` 保存通知事实，表达“一件事是什么”。`notification_deliveries` 保存用户投递状态，表达“这件事投递给谁、是否已读、是否删除”。

不用单表的原因：

- 同一事件可能投递给多个用户，单表会重复标题、正文、metadata 和导航信息。
- 已读、删除是用户维度状态，不应污染事件事实。
- `USER`、`PERMISSION`、后续 `ROLE`、`SYSTEM` 的 fan-out 需要保留目标解析来源。
- 保留期清理可以分别治理事件事实和投递状态。
- 双表为后续投递诊断、分类统计、去重冷却留出扩展点，同时不引入渠道、订阅、动作等过早表。

MVP 不新增订阅表、渠道表、动作表、推送连接表、重试表或 dead-letter 表。

## 6. target_type 投递模型

`target_type` 固定为 uppercase typed contract：

| target_type | target_ref | 语义 | MVP 状态 |
| --- | --- | --- | --- |
| `USER` | user id | 发给单个用户 | 实现 |
| `ROLE` | role code / role id | 发给拥有角色的用户 | 只保留 contract |
| `PERMISSION` | permission code | 发给拥有权限的用户 | 实现 |
| `SYSTEM` | empty 或系统目标码 | 系统广播 / 全局目标 | 只保留 contract |

投递规则：

- `recipient_user_id` 是普通用户查询、已读、删除的授权边界。
- `target_type` 与 `target_ref` 必须保存在 `notification_deliveries`，用于追溯 fan-out 来源。
- MVP 同步 fan-out 到最终用户，不引入异步队列、重试或 dead-letter。
- `PERMISSION` 可用于投递给具备某权限的用户，例如具备审计查看权限或任务管理权限的用户。
- `ROLE`、`SYSTEM` 在 MVP 只进入 contract，不实现 fan-out 逻辑。

## 7. Typed contract

Notification Center 的跨层魔法值必须纳入 typed contract，并在 server、OpenAPI、web 中保持一致。

### severity

| 值 | 语义 |
| --- | --- |
| `info` | 普通信息 |
| `warning` | 需要关注 |
| `error` | 明确失败 |
| `critical` | 高危或高影响事件 |

### category

| 值 | 语义 |
| --- | --- |
| `SECURITY` | 安全与审计 |
| `TASK` | 定时任务与执行历史 |
| `CONFIG` | 系统配置 |
| `OPERATIONS` | 运行态与模块运维 |
| `SYSTEM` | 平台系统通知 |

MVP 实际使用：

- Audit 使用 `SECURITY`。
- Scheduler 使用 `TASK`。
- `CONFIG`、`OPERATIONS`、`SYSTEM` 只作为 contract 保留或 notification 模块内部极少量系统提示使用。

### target_type

固定值：

- `USER`
- `ROLE`
- `PERMISSION`
- `SYSTEM`

MVP 实现：

- `USER`
- `PERMISSION`

MVP 只保留 contract：

- `ROLE`
- `SYSTEM`

### navigation_kind

固定值：

- `AUDIT_INCIDENT`
- `AUDIT_LOG`
- `SCHEDULER_RUN`
- `SYSTEM_CONFIG_ITEM`
- `MODULE_RUNTIME_ITEM`

MVP 实现：

- `AUDIT_INCIDENT`
- `AUDIT_LOG`
- `SCHEDULER_RUN`

Phase2 再实现：

- `SYSTEM_CONFIG_ITEM`
- `MODULE_RUNTIME_ITEM`

禁止新增 `external_url`、`quick_action`、`sse`、`websocket` 等 navigation kind 规避 MVP 边界。

## 8. MVP 接入来源与配置开关

MVP 当前已接入：

| 来源模块 | 事件类型 | severity 建议 | category | navigation_kind |
| --- | --- | --- | --- | --- |
| Audit | 登录失败 | `warning` / `error` | `SECURITY` | `AUDIT_LOG` |
| Audit | 权限拒绝 | `warning` / `error` | `SECURITY` | `AUDIT_LOG` |
| Audit | 高危审计事件 | `critical` | `SECURITY` | `AUDIT_INCIDENT` |
| Scheduler | 任务失败 | `error` | `TASK` | `SCHEDULER_RUN` |

MVP 配置域先暴露但事件生产者仍需后续补齐的来源：

| 配置项 | 目标事件 | 当前状态 |
| --- | --- | --- |
| `notification.source.system_announcement.enabled` | 系统公告发布、管理员创建的站内通知 | 配置开关已注册；发布/管理入口后续实现 |
| `notification.source.scheduled_task_success.enabled` | 定时任务手动执行完成 | 配置开关已注册；成功事件默认关闭，生产者后续实现 |
| `notification.source.system_config_change.enabled` | 系统配置修改、重置或配置校验失败 | 配置开关已注册；配置变更事件生产者后续实现 |
| `notification.source.access_log_retention_failure.enabled` | 访问日志清理任务失败 | 配置开关已注册；access-log retention 失败通知生产者后续实现 |

System Config 中固定注册以下 Notification 配置项：

- `notification.enabled`：是否启用站内通知功能，默认 `true`。
- `notification.retention_days`：通知保留天数，默认 `30`。
- `notification.default_page_size`：通知中心默认分页大小，默认 `20`。
- `notification.delivery.in_app.enabled`：是否启用站内投递，默认 `true`。
- `notification.delivery.dedupe_window_seconds`：相同来源、目标和业务对象通知去重窗口，默认 `300`。
- `notification.delivery.max_batch_recipients`：批量投递单批最大接收人数，默认 `1000`。
- `notification.display`：通知展示 object 配置，默认 `{"showReadDays":7,"popupLimit":5}`。
  - `showReadDays`：已读通知默认展示时间范围。
  - `popupLimit`：通知铃铛弹层最多展示条数。

关闭 `notification.enabled`、`notification.delivery.in_app.enabled` 或某个来源开关只阻止后续创建新的通知事实与投递记录；已有 `notification_events` 与 `notification_deliveries` 不会被删除。

来源模块必须提供：

- `source_module`
- `event_type`
- `resource_type`
- `resource_id`
- `resource_name`
- `occurred_at`
- `dedupe_key`
- `navigation_kind`
- `navigation_payload`
- `target_type`
- `target_ref`

MVP 不接入纯资源监控告警。Monitor 只有在 Phase2 中提供“非资源类运行态事件”时，才可评估进入 Notification Center。

通知中心明确不接入：

- CPU、内存、磁盘、Redis / PostgreSQL Down 等资源与依赖运维告警。
- 高频 access log / audit log 普通事件。
- 需要冷却、恢复、升级和值班渠道的告警。

## 9. 统一跳转规范

通知表不保存前端 URL 字符串，不允许每个来源模块自行拼接路由。来源模块只提交 `navigation_kind` 与结构化 `navigation_payload`，前端 notification 模块负责统一解析为路由。

推荐 payload 语义：

| navigation_kind | navigation_payload | 目标 |
| --- | --- | --- |
| `AUDIT_INCIDENT` | `{ "incident_id": "...", "audit_log_id": "..." }` | 审计事件详情并打开详情抽屉 |
| `AUDIT_LOG` | `{ "audit_log_id": "..." }` | 审计日志详情 |
| `SCHEDULER_RUN` | `{ "task_id": "...", "run_id": "..." }` | 任务执行历史并定位运行记录 |
| `SYSTEM_CONFIG_ITEM` | `{ "config_key": "...", "config_group": "..." }` | 系统配置项 |
| `MODULE_RUNTIME_ITEM` | `{ "module_id": "..." }` | 模块运行态详情 |

Phase1 只实现 Audit 与 Scheduler 的解析。SystemConfig 与 ModuleRuntime 的跳转解析延后到 Phase2。

## 10. OpenAPI MVP

MVP 只暴露当前用户通知的查询、未读统计、已读和删除投递接口。

固定接口：

| Method | Path | Permission | 说明 |
| --- | --- | --- | --- |
| `GET` | `/api/notifications` | `notification.view` | 查询当前用户通知 |
| `GET` | `/api/notifications/unread-count` | `notification.view` | 查询当前用户未读数 |
| `POST` | `/api/notifications/{delivery_id}/read` | `notification.read` | 标记当前用户单条投递已读 |
| `POST` | `/api/notifications/read-all` | `notification.read` | 标记当前用户全部或筛选范围已读 |
| `DELETE` | `/api/notifications/{delivery_id}` | `notification.read` | 删除当前用户单条投递 |

`GET /api/notifications` 查询参数：

- `status=all|unread|read`
- `severity`
- `category`
- `source_module`
- `occurred_from`
- `occurred_to`
- `page`
- `page_size`

MVP 不暴露：

- 创建通知 API。
- 订阅规则 API。
- 外部渠道 API。
- SSE / WebSocket API。
- 快捷动作执行 API。
- 管理员全局通知视图 API。

创建通知只能通过 server 内部 `moduleapi.NotificationPublisher` 调用。

## 11. 前端架构

推荐路径：

```text
web/src/modules/notification/
├─ api/
├─ contract/
│  ├─ paths.ts
│  ├─ permissions.ts
│  ├─ category.ts
│  ├─ severity.ts
│  └─ navigation.ts
├─ components/
│  ├─ NotificationBellPanel.vue
│  ├─ NotificationFilters.vue
│  ├─ NotificationTable.vue
│  └─ NotificationDetailDrawer.vue
├─ pages/
│  └─ list/
│     └─ index.vue
├─ locales/
├─ types/
├─ bootstrap-routes.ts
└─ index.ts
```

页面结构：

- 顶部通知铃铛：未读数量、下拉预览、快速已读、打开通知中心。
- `/notifications` 列表页：全部、未读、已读。
- 筛选：严重级别、来源模块、分类、时间范围。
- 空态必须区分“当前没有任何通知”和“当前筛选条件下没有通知”，并说明通知由系统公告、任务失败、安全事件、配置变更等事件产生。
- 接口 401/403/404/500 等错误不得展示为普通空态；顶部铃铛和列表页都必须显示错误态或错误提示。
- 操作：标记已读、全部已读、删除。
- 详情：通过 `NotificationDetailDrawer.vue` 展示通知内容、来源、事件时间和跳转入口。

壳层规则：

- `web/src/layouts/components/Notice.vue` 只作为 shell 接入点。
- `Notice.vue` 不直接保存业务假数据。
- 通知 API、类型、导航解析、i18n 全部放在 `web/src/modules/notification`。
- `/notifications` 是 notification 模块声明的菜单外全局 shell 路由，不依赖后端 bootstrap 菜单生成。
- `/notifications` 不生成 `/notifications/index` 作为最终 URL。
- Notification Center 不显示在左侧业务菜单；顶部通知弹层是 MVP 唯一主入口。
- 顶部铃铛 MVP 使用打开刷新。
- 可选：未读数 60 秒轻量轮询。
- 不做 SSE / WebSocket。
- `Notice.vue` 不直接 import notification 页面组件；它只消费 notification 模块公开 facade 或 contract。

## 12. 权限模型

权限：

| Permission | MVP 规则 |
| --- | --- |
| `notification.view` | 查看自己的通知和未读数 |
| `notification.read` | 标记自己的通知已读、全部已读、删除自己的投递 |
| `notification.manage` | 未来查看全局投递与管理清理配置，MVP 可先注册但不开放复杂管理页面 |

授权规则：

- 普通用户只能访问自己的 `notification_deliveries`。
- 标记已读、全部已读、删除都必须校验 `recipient_user_id` 为当前用户。
- 管理员全局视图不进入 MVP。
- `notification.manage` 不得在 MVP 中被用于绕过当前用户投递边界。
