# 共享资产复用治理规范

## 1. 目标

这份文档定义 `Graft` 的 Shared Asset Registry 治理规则，用于帮助未来 AI agent 在新增 helper、组件、
composable、mapper、service、repository helper、UI pattern、OpenAPI mapper 或治理脚本前，先发现并复用已有
共享资产。

Shared Asset Registry 是人工策展的治理索引，不是源码树清单，也不是自动生成的 inventory。

目标：

- 让可复用资产有稳定发现入口。
- 降低重复实现、平行 helper、重复 UI pattern 和重复 mapper 的概率。
- 保持 `web`、`server`、OpenAPI 与 AI governance 的 authority 边界清晰。
- 只登记高价值共享资产，不把 registry 变成全量文件列表。

非目标：

- 不自动登记每个 helper、组件、模块私有文件或生成文件。
- 不把模块私有业务逻辑提升为跨模块共享资产。
- 不让 registry 替代真实源码、`AGENTS.md`、OpenAPI source 或模块 contract。

## 2. Registry 位置

当前 registry 分为三类：

- `.ai/registries/web-shared-assets.yaml`
  - 前端共享组件、composable、utility、页面模式和前端治理辅助资产。
- `.ai/registries/server-shared-assets.yaml`
  - 后端稳定 runtime helper、moduleapi、registry、验证 helper 和测试 helper。
- `.ai/registries/cross-boundary-assets.yaml`
  - OpenAPI、i18n key-first、route / menu / permission、request correlation 等跨边界资产。

`.ai/registries/**` 是可提交治理资料；`.ai/environment/**` 仍只保存生成的本机工具事实，不能放设计治理。

仓库 skill 仍统一放在 `.agents/skills/**`。不得新增 `.ai/skills/**` 作为第二套 skill 路径。

## 3. 维护触发

当当前任务或未来任务新增、移动、重命名、删除这些区域下的可复用资产时，agent 必须评估 registry 是否需要更新：

- `web/src/shared/**`
- `web/src/components/**`
- `web/src/composables/**`
- `web/src/utils/**`
- `web/src/modules/*/shared/**`
- `web/src/modules/*/presenter/**`
- `server/internal/**`
- `server/internal/moduleapi/**`
- `server/modules/*/moduleapi/**`
- `openapi/**`
- `scripts/validate_*.py`
- `.agents/skills/**`

触发只要求评估，不代表必须新增条目。

## 4. 登记标准

只有满足至少一项时才新增 registry 条目：

- 已被至少两个模块复用。
- 由稳定共享边界拥有，例如 `web/src/shared/**`、`server/internal/httpx/**`、`server/internal/moduleapi/**`。
- 属于未来 agent 高重复风险的通用能力。
- 属于跨边界资产，例如 OpenAPI contracts、i18n key-first presentation、moduleapi、scheduler registry、
  configregistry、logger、httpx、migration validation、route / menu / permission contract。

不得登记：

- 每个普通 helper、组件或模块私有文件。
- 只服务单个模块且没有稳定对外 contract 的业务逻辑。
- generated-only artifact 作为 authority。
- `.ai/venv/**`、`node_modules/**`、`dist/**`、`tmp/**`、`coverage/**` 或 build output 下的文件。

模块私有资产可以登记为 module-owned duplicate-risk pattern，但必须明确 `do_not_use_when`，避免跨模块误用。

## 5. 移除与重命名

如果已登记资产的 `path` 消失，agent 不得静默删除 registry 条目。必须先判断资产是：

- moved
- renamed
- replaced
- intentionally removed

如果移除或替换 registry 条目，closeout 必须报告：

```text
registry_entries_removed_or_replaced:
- name: <asset-name>
  removed_reason: <why removed>
  replaced_by: <replacement asset name/path, if any>
```

## 6. Registry Schema

每个 registry 文件使用紧凑 YAML：

```yaml
schema_version: 1
updated_at: "YYYY-MM-DD"
assets:
  - name: "management-components"
    type: "frontend-component-suite"
    path: "web/src/shared/components/management"
    owner: "web/shared"
    purpose: "Reusable management table layout, pagination, action column, and compact timestamp helpers."
    use_when:
      - "Building module list or CRUD management tables."
    do_not_use_when:
      - "The UI is module-private and not a management/list surface."
    examples:
      - "web/src/modules/user/pages/index.vue"
    related_assets:
      - "observability-helpers"
    validation_commands:
      - "cd web && bun run check"
```

必填字段：

- `name`
- `type`
- `path`
- `owner`
- `purpose`
- `use_when`
- `do_not_use_when`

可选字段：

- `examples`
- `related_assets`
- `validation_commands`
- `notes`

允许的 `type` 值：

- `frontend-component`
- `frontend-component-suite`
- `frontend-composable`
- `frontend-utility`
- `frontend-page-pattern`
- `frontend-contract`
- `backend-moduleapi`
- `backend-runtime-helper`
- `backend-registry`
- `backend-test-helper`
- `backend-validation-helper`
- `cross-boundary-contract`
- `governance-skill`
- `governance-validator`

## 7. 复用 Preflight

新增可复用代码前，agent 必须执行 Shared Asset Reuse Preflight：

1. 根据任务 class 读取对应 registry。
2. 搜索 registry 中同类 `type`、`purpose`、`use_when`。
3. 使用 `rg` 在真实源码中搜索相似命名和实现。
4. 判断候选资产是 shared-owned、module-owned、contract-owned 还是 generated-derived。
5. 能复用则复用；不能复用则记录拒绝原因。
6. 如果新增、移动、重命名或删除触发区域内资产，评估是否维护 registry。
7. 如果已登记路径消失，按“移除与重命名”规则处理。

closeout 必须报告：

```text
shared_asset_preflight:
- status: used | not_applicable
- registries_checked:
- assets_reused:
- assets_considered_but_rejected:
- new_registry_entries:
- registry_entries_removed_or_replaced:
- validation_commands:
```

## 8. Validator 行为

`scripts/validate_shared_asset_registries.py` 是轻量结构检查，不是 registry 自动生成器。

它应该：

- 解析 YAML。
- 校验必填字段。
- 校验允许的 `type` 值。
- 校验 `asset.path` 存在。
- 校验 `examples` 路径存在。
- 拒绝 generated-only artifact 作为 authority。
- 拒绝 `.ai/venv`、`node_modules`、`dist`、`tmp`、`coverage` 或 build output 路径。

当前策略：

- 已登记路径缺失必须失败。
- 新发现的未登记文件最多产生 warning，不阻断任务。
- 不自动重写 registry。

## 9. AGENTS 与 Skill 关系

根 `AGENTS.md` 只保留短硬规则和 skill 入口；详细规则留在本设计文档。

`.agents/skills/graft-shared-asset-reuse/SKILL.md` 负责可执行流程和 closeout 证据格式。

如果本规范、registry、skill 与 `AGENTS.md` 分叉，先按根 `AGENTS.md` 判断 authority，再在同一治理切片中修正文档或 skill。
