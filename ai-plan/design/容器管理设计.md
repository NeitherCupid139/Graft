# 容器管理设计

本文档定义“运维管理 -> 容器管理 MVP”的产品 IA、契约、模块边界、安全边界、当前完成态和后续延期边界。该能力是
`Graft` 的运维业务模块，不是特定容器运行时专属产品能力。

当前状态：

- MVP 已完成容器列表、详情、日志、start / stop / restart / remove、批量操作、权限、审计、系统配置、前端列表治理和容器 Shell 能力闭环。
- 删除 / remove 容器与批量 start / stop / restart / remove 已纳入 MVP，并已按 authority-first 路径同步设计、
  OpenAPI、后端模块和前端页面消费层。
- `container-management-console-redesign` 主题已完成最终收敛；恢复证据归档到
  `ai-plan/public/archive/container-management-console-redesign/`。
- 容器创建、镜像更新、Kubernetes 和长周期监控仍不属于当前 MVP；交互式容器 Shell 已纳入当前 MVP，但服务器终端和文件编辑仍不在范围内。

## 2. 产品命名与 IA

菜单和产品能力统一命名为“容器管理”，不要命名为特定容器运行时管理。

原因：

- 第一版底层只支持首个本地容器运行时适配器，但产品语义是容器运维能力。
- 权限、菜单、OpenAPI tag、前端模块和后端模块必须预留给后续 runtime 扩展，例如 Podman、containerd、远程 Docker Host
  或 Kubernetes。
- 特定运行时命名只作为第一版 runtime adapter 实现名 `DockerRuntime` 或配置键 `ops.container.docker.endpoint` 出现。

菜单位置：

```text
运维管理
└─ 容器管理
```

路由路径：

- 后端 API：`/api/ops/containers`
- 前端页面：`/ops/containers`

现有“服务器管理”边界：

- 当前仓库已有 monitor 主题落过“服务器管理 / 服务器状态”，它表达服务器状态、运行时、依赖服务等只读状态观察。
- “运维管理”表达可执行运维动作的能力集合，例如容器启停、日志读取和未来其它运维操作。
- 容器 start / stop / restart 是运维动作，不应挂在“服务器管理”下，避免把状态观察和高危操作混在一个菜单概念里。

## 3. MVP 范围

包含：

- 本机容器运行时 socket 管理。
- 容器列表。
- 容器详情。
- 容器来源识别：`standalone`、`compose`、`swarm`、`kubernetes`、`unknown`。
- 容器日志读取，支持 `tail`、`since`、`timestamps`、`stdout`、`stderr`。
- `start` / `stop` / `restart`。
- `remove` / 删除容器，默认不强制删除运行中容器；运行中容器删除必须显式传入 `force=true`。
- 批量 `start` / `stop` / `restart` / `remove`，由后端统一聚合结果、权限、审计和部分失败明细。
- 权限校验。
- 审计记录。
- 交互式容器 Shell：签发单次票据、WebSocket 升级、权限校验、功能开关、Origin allowlist、会话审计和前端详情页 Shell 面板。
- 前端列表、详情 Drawer、日志 Drawer 或全屏 Dialog、高危操作确认。
- 列表行操作可用性由后端返回的 `can_start` / `can_stop` / `can_restart` / `can_remove` 表达，并必须同时受
  `ops.container.actions.dangerous_enabled` 约束。
- action 响应通过 `message_key` / `message` 返回成功反馈；前端优先使用稳定 key 本地化。
- 批量 action 响应返回 `total`、`success_count`、`failed_count`、`items` 和 `request_id`，前端据此展示全部成功、
  部分失败或全部失败反馈。
- 列表支持服务端 `limit` / `offset` 分页，以及 `keyword` / `state` / `health` 筛选。
- 列表支持服务端 `orchestrator` 来源筛选，来源识别 authority 固定在后端，不允许前端自行解析 Docker labels。
- 列表响应返回 `items`、`total`、`limit`、`offset`、`summary` 和 `runtime`。
- 列表行展示低成本运行时摘要字段，包括短 ID、主名称、端口、主 IP / 网络摘要、资源可用性、健康状态、
  Compose metadata、orchestrator metadata 和 action availability。
- 详情 Drawer 只在用户打开详情时读取详情接口；日志 Drawer 只在用户打开日志时读取日志接口。
- 列表页使用统一 TableCard toolbar、批量操作区、列设置、密度切换、`ManagementTablePagination` 和内部横向滚动策略。
- 所有确认类交互必须使用 TDesign Vue Next `DialogPlugin.confirm` 或 `t-dialog`，禁止使用浏览器原生
  `window.confirm` / `alert` / `prompt`。
- 非 `standalone` 容器的高危操作收敛由后端根据来源识别结果和系统配置统一决定；前端只消费结果，不得维护第二套策略真相。

不包含：

- 服务器终端。
- 文件编辑。
- 删除镜像。
- `prune`。
- 镜像构建、拉取、推送。
- 容器创建。
- 多节点或远程 Docker Host。
- Kubernetes。
- 长周期资源监控采集。
- 单独操作日志表。

## 4. OpenAPI 契约草案

OpenAPI tag 使用 `container`，展示说明为“容器管理”。不得使用 `docker` 作为业务 tag 或业务前缀。

路径：

```text
GET  /api/ops/containers
GET  /api/ops/containers/{id}
GET  /api/ops/containers/{id}/mounts/usage
POST /api/ops/containers/{id}/mounts/{mountId}/usage/refresh
GET  /api/ops/containers/{id}/logs
POST /api/ops/containers/{id}/shell/sessions
GET  /api/ops/containers/{id}/shell/ws
POST /api/ops/containers/{id}/start
POST /api/ops/containers/{id}/stop
POST /api/ops/containers/{id}/restart
POST /api/ops/containers/{id}/remove
POST /api/ops/containers/batch-actions
```

`id` 表示 container id 或 name。OpenAPI description 必须明确：

- 客户端调用前必须对 `id` 执行 `encodeURIComponent`。
- 后端读取 path 参数后必须执行 `PathUnescape`。
- unescape 后必须拒绝空值、斜杠和控制字符。
- 校验失败返回 `ops.container.error.invalidContainerRef`。

挂载空间统计：

- `ContainerMount` 必须返回 `mount_id`，该值由后端基于 Docker inspect 得到的 `destination + source + type`
  生成稳定 hash，不得把 `source` 裸路径作为 URL path 参数。
- `ContainerMount.usage` 只允许附带最近一次缓存统计结果；容器详情接口不得同步执行 `du` 或文件系统全量扫描。
- `GET /api/ops/containers/{id}/mounts/usage` 只返回当前容器所有挂载的最近一次统计结果，不触发昂贵扫描。
- `POST /api/ops/containers/{id}/mounts/{mountId}/usage/refresh` 只刷新当前容器 inspect 返回的指定挂载项，后端必须用
  `mount_id` 回查 inspect 挂载，不允许客户端传任意 path。
- `MountUsage.status` 取值为 `not_measured`、`measured`、`pending`、`unsupported`、`permission_denied`、`not_found`、
  `timeout`、`error`。
- bind 挂载统计宿主机 `source` 路径；volume 只有在当前 runtime 能解析 volume mountpoint 时才统计；tmpfs MVP 返回
  `unsupported`，不得假装已统计。
- 统计实现必须有 timeout 和短缓存，失败返回结构化 status，不用 500 表示普通统计失败。

日志查询参数：

- `tail`：整数，默认 `200`，最大 `2000`。
- `since`：时间戳或持续时间语义由 OpenAPI source 明确，后端必须稳定解析并返回本地化错误。
- `timestamps`：是否包含日志时间戳。
- `stdout`：是否读取 stdout。
- `stderr`：是否读取 stderr。

列表查询参数：

- `limit`：整数，默认 `20`，最大 `100`。
- `offset`：从 `0` 开始的偏移量。
- `keyword`：按容器 ID、短 ID、主名称、镜像、状态、运行时、端口、标签、Compose metadata 和低成本网络字段做大小写不敏感匹配。
- `state`：可选标准状态筛选，取值为 `created`、`running`、`paused`、`restarting`、`removing`、`exited`、`dead`、`unknown`。
- `health`：可选健康状态筛选，取值为 `healthy`、`unhealthy`、`starting`、`none`、`unavailable`；当运行时列表路径无法低成本提供健康状态时，列表行使用
  `unavailable` 语义。

响应模型至少包含：

- `ContainerSummary`
- `ContainerDetail`
- `ContainerOrchestratorInfo`
- `ContainerPort`
- `ContainerMount`
- `ContainerMountUsage`
- `ContainerMountUsageListResponse`
- `ContainerNetwork`
- `ContainerLogResponse`
- `ContainerShellSessionResponse`
- `ContainerActionResponse`
- `ContainerRuntimeInfo`
- `ContainerListSummary`
- `ContainerResourceSummary`

契约约束：

- `ContainerDetail` 不暴露敏感环境变量。
- `ContainerSummary` 和 `ContainerDetail` 必须包含 `orchestrator` 字段，表达容器来源、受管信息、摘要字段、
  风险提示 code 和推荐动作 code。
- `ContainerResourceSummary.cpu_percent` 表示相对宿主机总 CPU 容量归一化后的百分比；列表与详情不得再把该字段解释为
  Docker 多核累加百分比。
- 日志响应不返回超出 `max_tail` 的内容。
- action 响应必须返回稳定结果、容器标识、操作名、runtime 和操作后的状态。
- action 响应成功时应返回 `message_key` / `message`，用于前端 key-first 成功反馈。
- `remove` 请求体为 `{ force?: boolean }`，默认 `false`；当目标为运行中容器且 `force=false` 时必须拒绝并返回
  `ops.container.error.invalidState`。
- 批量请求体为 `{ action, ids, force? }`，`action` 取值为 `start`、`stop`、`restart`、`remove`，`ids` 最多 100 个。
- 批量响应 `items[]` 必须逐项返回 `id/name/action/success/error_code/message_key/message`，不得用首个错误掩盖部分成功。
- OpenAPI source 修改后，generated Go / TypeScript 产物只是 derived artifact，必须由后续阶段同步生成。
- 列表响应必须返回 `items`、`total`、`limit`、`offset`、`summary` 和 `runtime`。
- 列表 `ContainerSummary` 可包含 `short_id`、`name`、`health`、`primary_ip`、`networks`、`network_summary`、
  `resource`、`restart_count`、`compose_project`、`compose_service`、`orchestrator`、`can_start`、`can_stop`、
  `can_restart`、`can_remove`。
- 列表不得预加载 raw inspect、日志、环境变量或大体量原始 JSON。
- 第一版 Docker list 路径只使用 `/containers/json` 可直接提供的低成本字段；`resource.available=false` 且
  `unavailable_reason=stats_not_collected` 表示列表未做 Docker stats 轮询；`health=unavailable` 表示未做逐行 inspect。
- `restart_count` 只有在详情 inspect 路径可低成本取得时返回；列表路径保持 nullable。

`ContainerOrchestratorInfo` 语义：

- `type`
  - `standalone | compose | swarm | kubernetes | unknown`
- `managed`
  - 表示当前容器是否属于上层编排或控制面结果，而不是直接把容器本身视为 canonical authority
- `display_name`
  - 仅作 fallback；前端可见文案仍优先按 `type` 做本地化
- `confidence`
  - `high | medium | low`
- `project` / `service` / `stack` / `namespace` / `pod` / `container` / `task`
  - 来源摘要字段，仅在对应编排类型下返回
- `working_dir` / `config_files`
  - 仅用于 Compose 项目来源元数据；不得与容器运行时 `working_dir` 混淆
- `warnings`
  - 稳定 warning code 列表，不直接承载最终可见文案
- `recommended_action`
  - 稳定推荐动作 code，不直接承载最终可见文案
- `action_level`
  - `readonly | warn | allow`
- `batch_action_allowed`
  - 当前来源在系统配置策略下是否允许参与批量高危操作

来源识别优先级：

1. `kubernetes`
2. `swarm`
3. `compose`
4. `standalone`
5. `unknown`

判定原则：

- 无任何受管编排信号时直接视为 `standalone`，不得把普通 Docker 容器误报为 `unknown`
- 同时命中多个编排体系核心信号，或命中受管信号但缺少关键信息无法安全归类时，使用 `unknown`
- 来源识别 authority 固定在后端 inspect / metadata resolver，前端不得自行解析 labels

## 6. 后端模块设计

推荐模块路径：

```text
server/modules/container
```

不要放入 `server/internal/**`，因为容器管理是业务模块，不是 core runtime。

建议结构：

```text
server/modules/container/
├─ descriptor.go
├─ module.go
├─ module_registration.go
├─ route_registration.go
├─ route_errors.go
├─ service.go
├─ runtime.go
├─ runtime_adapter.go
├─ mapper_http.go
└─ contract/
   ├─ permission.go
   ├─ route.go
   └─ message.go
```

模块接入：

- 通过 `NewModuleSpec()` 接入 `server/internal/moduleregistry`。
- 新增模块后运行 `cd server && go generate ./internal/moduleregistry`。
- `Register` 阶段注册路由、菜单、权限、i18n、系统配置定义。
- `Boot` 阶段只启动已声明 runtime 行为；MVP 不需要后台 goroutine。
- `Shutdown` 阶段关闭 Docker client 相关资源。

第一版不建数据库表：

- 容器状态来自本地容器运行时 API 实时读取。
- 操作记录复用 audit log。
- 不新增单独 action log 表。

## 7. Runtime 接口

后端 service 只依赖 `ContainerRuntime`，第一版实现 `DockerRuntime`。

```go
type ContainerRuntime interface {
    Info(ctx context.Context) (ContainerRuntimeInfo, error)
    List(ctx context.Context, query ContainerListQuery) ([]ContainerSummary, error)
    Detail(ctx context.Context, id ContainerRef) (ContainerDetail, error)
    Logs(ctx context.Context, id ContainerRef, query ContainerLogQuery) (ContainerLogs, error)
    Shell(ctx context.Context, id ContainerRef, command string) (terminal.Session, error)
    Start(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
    Stop(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
    Restart(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
    Remove(ctx context.Context, id ContainerRef, force bool) (ContainerActionResult, error)
}
```

第一版 `DockerRuntime`：

- 默认 endpoint 通过 `ops.container.docker.endpoint` 表达。
- 只支持 local Docker。
- 运行时 socket 不存在映射为 `ops.container.error.runtimeSocketMissing`。
- 当前进程无权限访问 socket 映射为 `ops.container.error.runtimePermissionDenied`。
- 运行时 endpoint、socket、权限或 daemon 连接失败映射为连接类错误 key，例如
  `ops.container.error.runtimeUnavailable`。
- 容器不存在映射为 `ops.container.error.containerNotFound`。
- 操作状态不允许映射为 `ops.container.error.invalidState`。
- 操作超时映射为 `ops.container.error.timeout`。

运行时 client 优先使用官方 SDK；引入依赖前必须记录许可证兼容性、socket 权限风险和本地部署前提。

## 8. 审计设计

`start` / `stop` / `restart` 与容器 Shell 会话必须在 service 层发布 audit event，成功和失败都记录。

审计字段：

- `ResourceType`: `container`
- `ResourceID`: container id
- `ResourceName`: container name
- `action`:
  - `ops.container.start`
  - `ops.container.stop`
  - `ops.container.restart`
  - `ops.container.shell.session.requested`
  - `ops.container.shell.ticket.issued`
  - `ops.container.shell.ticket.rejected`
  - `ops.container.shell.session.started`
  - `ops.container.shell.session.closed`
  - `ops.container.shell.session.failed`

metadata：

- `container_id`
- `container_name`
- `image`
- `action`
- `runtime`: 首个本地容器运行时适配器 key
- `result`
- `error`
- `status_before`
- `status_after`

审计记录不得包含敏感环境变量、secret、token、authorization header、Docker inspect 原始敏感字段。

## 9. 系统配置设计

第一版配置 key：

```text
ops.container.runtime.enabled = false
ops.container.runtime = first-adapter
ops.container.docker.endpoint = unix:///var/run/docker.sock
ops.container.logs.default_tail = 200
ops.container.logs.max_tail = 2000
ops.container.actions.dangerous_enabled = false
ops.container.actions.compose_level = warn
ops.container.actions.swarm_level = readonly
ops.container.actions.kubernetes_level = readonly
ops.container.actions.unknown_level = readonly
ops.container.shell.enabled = false
```

实施约束：

- 模块加载开关仍然只由启动期配置控制，例如 `GRAFT_MODULES_ENABLED` / `modules.enabled`。
- 系统配置不能被解释为 backend module enable/disable authority；模块运行时页只读展示当前进程内模块注册、启用、
  依赖、迁移、schema 和配置状态。
- `ops.container.runtime.enabled` 是容器运行时访问开关，不是 container 模块启用开关。
- 当 `ops.container.runtime.enabled=false` 时，container 模块仍可加载，但 API 不得访问 Docker/container runtime，
  并返回 `ops.container.error.runtimeDisabled`；前端文案应表达“容器运行时访问未启用”。
- 当 `ops.container.runtime.enabled=true` 时，仅表示允许访问容器运行时，仍可能因 endpoint、socket 或权限返回
  `ops.container.error.runtimeUnavailable` 等连接类错误。
- `ops.container.actions.dangerous_enabled` 是独立高危操作开关，只控制 `start` / `stop` / `restart` 以及未来
  `delete`、批量删除等高危能力；即使 runtime enabled 为 true，该开关为 false 时仍只允许列表、详情、日志和 Shell 之外的
  已启用只读能力。
- `ops.container.actions.compose_level`、`swarm_level`、`kubernetes_level`、`unknown_level` 是来源级操作策略真相，
  取值固定为 `readonly`、`warn`、`allow`。
- `readonly`
  - 隐藏容器级高危操作，不允许参与批量高危操作。
- `warn`
  - 允许单容器高危操作，但必须展示显著来源风险提示；不允许参与批量高危操作。
- `allow`
  - 允许单容器高危操作，并允许参与批量高危操作；仍保留来源信息展示，但不再因来源策略额外阻断。
- 第一版默认值：
  - `compose = warn`
  - `swarm = readonly`
  - `kubernetes = readonly`
  - `unknown = readonly`
- `dangerous_enabled` 是总闸；当该值为 `false` 时，来源级 `*_level` 不得放开任何高危操作。
- `ops.container.shell.enabled` 是独立容器 Shell 功能开关；即使 runtime enabled 为 true，也必须显式开启后端才允许签发
  Shell 会话票据和升级 WebSocket 连接。
- `runtime.enabled` 和 `actions.dangerous_enabled` 可以走现有 boolean resolver；来源级 `*_level` 应走现有 enum
  schema + system config renderer，不得回退为前端本地 key 映射。
- `shell.enabled` 与 Shell WebSocket allowlist 也应走现有稳定配置入口，不得在前端或下游 consumer 伪造开关状态。
- `runtime`、`endpoint`、`default_tail`、`max_tail` 第一版优先用模块默认值或进程配置，不为 MVP 扩展全局
  `SystemConfigResolver`。
- 如果 `endpoint` 进入系统配置定义，必须标注 `Sensitive` 或 `RestartRequired`。
- 配置定义的 schema 是系统配置 UI 渲染 authority；前端不得按 key 写本地兼容映射。
- bootstrap 菜单由后端统一裁剪，依据至少包含模块是否加载、用户权限和功能开关；内部可在 menu item 上声明 feature
  gate，但不得在 OpenAPI/bootstrap menu wire shape 暴露该内部字段。
- 当 `ops.container.runtime.enabled=false` 时，MVP 先隐藏容器管理菜单；容器模块的系统配置项必须仍可在系统配置页搜索和编辑。

## 10. 前端设计

推荐路径：

```text

```

页面：

- route：`/ops/containers`
- page type：`list-form-detail`
- PageHeader：`容器管理`
- FilterBar：关键字、状态、健康状态、查询、重置。
- FilterBar：关键字、状态、健康状态、来源、查询、重置。
- TableCard toolbar：刷新、列设置、密度切换。
- TableCard footer：`ManagementTablePagination`，默认 pageSize 20，options 为 `10 / 20 / 50 / 100`。

表格列：

- 默认列：状态、容器、镜像、来源、端口映射、IP / 网络、资源、运行状态 / 健康、创建时间、操作。
- 可选列：启动时间、重启策略、镜像 ID、标签等低频字段。
- `started_at` 和 `restart_policy` 不作为默认列。
- 操作列固定右侧，默认提供详情和更多菜单；更多菜单包含日志、start / stop / restart、复制 ID，并受权限与后端
  `can_start` / `can_stop` / `can_restart` / `can_remove` 以及 `orchestrator.action_level` / `batch_action_allowed`
  控制。
- 刷新只出现在 TableCard toolbar，不出现在 PageHeader。
- 表格默认禁止 body/page 级横向滚动；可见列超出容器时只允许表格内部横向滚动。

交互：

- 详情使用 Drawer。
- 日志使用 Drawer，支持刷新、tail、since、timestamps、stdout/stderr、复制和可关闭的安全自动刷新。
- 列表必须显示来源 tag 和来源摘要；详情顶部摘要区必须显示来源信息与风险提示。
- `start` / `stop` / `restart` 使用确认弹窗，确认文案包含容器显示名。
- 当 `orchestrator.action_level=warn` 时，确认弹窗必须追加来源风险说明。
- 当 `orchestrator.action_level=readonly` 时，危险操作入口必须隐藏或禁用到无法误触，不得只靠文案提示。
- 批量高危操作只允许作用于 `batch_action_allowed=true` 的容器；混选时必须显式提示被来源策略跳过的项。
- 操作按钮使用 `v-permission`。
- 不展示后端未提供的 CPU / 内存空列。
- 列表页必须消费服务端 `limit` / `offset` 分页与 `total`，不得长期保留前端本地分页来掩盖 OpenAPI / 后端分页缺口。
- API 类型必须消费 `@/contracts/openapi/generated/schema`，不得手写第二套 DTO。
- 详情 Drawer 宽度为操作台式宽布局，展示 identity、state、runtime、network/ports、mounts、labels/metadata，并将当前详情响应的
  raw JSON 默认折叠。
- 日志 Drawer 默认不预加载日志；打开日志入口后才通过日志接口加载。
- 详情页保留 `shell` tab，只有在该 tab 激活后才向后端请求 Shell 会话票据；未激活时不得预创建 WebSocket 或后台会话。
- Shell 面板必须复用后端返回的 `message_key` / `message` 做 key-first 错误展示，并显式处理 disabled、forbidden、
  container-not-running、ticket-invalid、ticket-expired、ticket-used、origin-denied、command-not-found、

## 11. 安全边界与错误处理

安全边界：

- 默认不做服务器终端。
- 不做文件编辑。
- 不做删除镜像。
- 不做 prune。
- 不做容器创建。
- 所有写操作必须同时通过权限、`ops.container.actions.dangerous_enabled` 和审计。
- 所有非 `standalone` 容器的高危操作都必须同时通过来源识别和来源级系统配置策略，不得让前端自行放宽。
- 容器 Shell 只允许进入当前目标容器，必须先走认证 HTTP 票据签发，再走单次消费的 WebSocket 升级，不得接受匿名、
  可重放或跨容器复用的会话票据。
- 容器 Shell 默认关闭，必须同时通过 `ops.container.shell` 权限、`ops.container.shell.enabled` 开关、运行中容器状态校验
  和 Origin allowlist 校验。
- 容器 Shell 不得扩展为宿主机 Shell、任意文件编辑或跨容器跳板能力。

错误场景：

- 运行时 socket 不存在。
- 当前用户无权限访问运行时 socket。
- 运行时 daemon 未启动。
- 容器不存在。
- 容器状态不允许执行操作。
- 日志过大。
- 操作超时。
- 容器运行时访问未启用。
- 高危操作未启用。

API 必须返回可本地化错误 key 和 fallback message。
