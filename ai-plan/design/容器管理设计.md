# 容器管理设计

本文档定义“运维管理 -> 容器管理 MVP”的产品 IA、契约、模块边界、安全边界和分阶段实施计划。该能力是
`Graft` 的运维业务模块，不是特定容器运行时专属产品能力。

## 2. 产品命名与 IA

菜单和产品能力统一命名为“容器管理”，不要命名为特定容器运行时管理。

原因：

- 第一版底层只支持首个本地容器运行时适配器，但产品语义是容器运维能力。
- 权限、菜单、OpenAPI tag、前端模块和后端模块必须预留给后续 runtime 扩展，例如 Podman、containerd、远程 Docker Host
  或 Kubernetes。
- 特定运行时命名只作为第一版 runtime adapter 实现名 `DockerRuntime` 或配置键 `ops.container.docker.endpoint` 出现。

菜单位置：

```text
运维管理
└─ 容器管理
```

路由路径：

- 后端 API：`/api/ops/containers`
- 前端页面：`/ops/containers`

现有“服务器管理”边界：

- 当前仓库已有 monitor 主题落过“服务器管理 / 服务器状态”，它表达服务器状态、运行时、依赖服务等只读状态观察。
- “运维管理”表达可执行运维动作的能力集合，例如容器启停、日志读取和未来其它运维操作。
- 容器 start / stop / restart 是运维动作，不应挂在“服务器管理”下，避免把状态观察和高危操作混在一个菜单概念里。

## 3. MVP 范围

包含：

- 本机容器运行时 socket 管理。
- 容器列表。
- 容器详情。
- 容器日志读取，支持 `tail`、`since`、`timestamps`、`stdout`、`stderr`。
- `start` / `stop` / `restart`。
- 权限校验。
- 审计记录。
- 前端列表、详情 Drawer、日志 Drawer 或全屏 Dialog、高危操作确认。

不包含：

- 服务器终端。
- `exec` 进入容器。
- 文件编辑。
- 删除容器。
- 删除镜像。
- `prune`。
- 镜像构建、拉取、推送。
- 容器创建。
- 多节点或远程 Docker Host。
- Kubernetes。
- 长周期资源监控采集。
- 单独操作日志表。

## 4. OpenAPI 契约草案

OpenAPI tag 使用 `container`，展示说明为“容器管理”。不得使用 `docker` 作为业务 tag 或业务前缀。

路径：

```text
GET  /api/ops/containers
GET  /api/ops/containers/{id}
GET  /api/ops/containers/{id}/logs
POST /api/ops/containers/{id}/start
POST /api/ops/containers/{id}/stop
POST /api/ops/containers/{id}/restart
```

`id` 表示 container id 或 name。OpenAPI description 必须明确：

- 客户端调用前必须对 `id` 执行 `encodeURIComponent`。
- 后端读取 path 参数后必须执行 `PathUnescape`。
- unescape 后必须拒绝空值、斜杠和控制字符。
- 校验失败返回 `ops.container.error.invalidContainerRef`。

日志查询参数：

- `tail`：整数，默认 `200`，最大 `2000`。
- `since`：时间戳或持续时间语义由 OpenAPI source 明确，后端必须稳定解析并返回本地化错误。
- `timestamps`：是否包含日志时间戳。
- `stdout`：是否读取 stdout。
- `stderr`：是否读取 stderr。

响应模型至少包含：

- `ContainerSummary`
- `ContainerDetail`
- `ContainerPort`
- `ContainerMount`
- `ContainerNetwork`
- `ContainerLogResponse`
- `ContainerActionResponse`
- `ContainerRuntimeInfo`

契约约束：

- `ContainerDetail` 不暴露敏感环境变量。
- 日志响应不返回超出 `max_tail` 的内容。
- action 响应必须返回稳定结果、容器标识、操作名、runtime 和操作后的状态。
- OpenAPI source 修改后，generated Go / TypeScript 产物只是 derived artifact，必须由后续阶段同步生成。

## 5. 权限与错误 Key

只注册 MVP 实际使用权限：

```text
ops.container.view
ops.container.detail
ops.container.logs
ops.container.start
ops.container.stop
ops.container.restart
```

不要提前注册 `inspect`、`delete`、`prune`、`create` 等未落地权限。

错误 key 草案：

```text
ops.container.error.runtimeDisabled
ops.container.error.runtimeSocketMissing
ops.container.error.runtimePermissionDenied
ops.container.error.runtimeUnavailable
ops.container.error.containerNotFound
ops.container.error.invalidContainerRef
ops.container.error.invalidState
ops.container.error.logsTooLarge
ops.container.error.invalidLogQuery
ops.container.error.timeout
ops.container.error.dangerousActionsDisabled
```

错误响应必须保留 stable key 和 fallback message，禁止把 Docker SDK 原始错误直接泄露给前端。

## 6. 后端模块设计

推荐模块路径：

```text
server/modules/container
```

不要放入 `server/internal/**`，因为容器管理是业务模块，不是 core runtime。

建议结构：

```text
server/modules/container/
├─ descriptor.go
├─ module.go
├─ module_registration.go
├─ route_registration.go
├─ route_errors.go
├─ service.go
├─ runtime.go
├─ runtime_adapter.go
├─ mapper_http.go
└─ contract/
   ├─ permission.go
   ├─ route.go
   └─ message.go
```

模块接入：

- 通过 `NewModuleSpec()` 接入 `server/internal/moduleregistry`。
- 新增模块后运行 `cd server && go generate ./internal/moduleregistry`。
- `Register` 阶段注册路由、菜单、权限、i18n、系统配置定义。
- `Boot` 阶段只启动已声明 runtime 行为；MVP 不需要后台 goroutine。
- `Shutdown` 阶段关闭 Docker client 相关资源。

第一版不建数据库表：

- 容器状态来自本地容器运行时 API 实时读取。
- 操作记录复用 audit log。
- 不新增单独 action log 表。

## 7. Runtime 接口

后端 service 只依赖 `ContainerRuntime`，第一版实现 `DockerRuntime`。

```go
type ContainerRuntime interface {
    Info(ctx context.Context) (ContainerRuntimeInfo, error)
    List(ctx context.Context, query ContainerListQuery) ([]ContainerSummary, error)
    Detail(ctx context.Context, id ContainerRef) (ContainerDetail, error)
    Logs(ctx context.Context, id ContainerRef, query ContainerLogQuery) (ContainerLogs, error)
    Start(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
    Stop(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
    Restart(ctx context.Context, id ContainerRef) (ContainerActionResult, error)
}
```

第一版 `DockerRuntime`：

- 默认 endpoint 通过 `ops.container.docker.endpoint` 表达。
- 只支持 local Docker。
- 运行时 socket 不存在映射为 `ops.container.error.runtimeSocketMissing`。
- 当前进程无权限访问 socket 映射为 `ops.container.error.runtimePermissionDenied`。
- 运行时 endpoint、socket、权限或 daemon 连接失败映射为连接类错误 key，例如
  `ops.container.error.runtimeUnavailable`。
- 容器不存在映射为 `ops.container.error.containerNotFound`。
- 操作状态不允许映射为 `ops.container.error.invalidState`。
- 操作超时映射为 `ops.container.error.timeout`。

运行时 client 优先使用官方 SDK；引入依赖前必须记录许可证兼容性、socket 权限风险和本地部署前提。

## 8. 审计设计

`start` / `stop` / `restart` 必须在 service 层发布 audit event，成功和失败都记录。

审计字段：

- `ResourceType`: `container`
- `ResourceID`: container id
- `ResourceName`: container name
- `action`:
  - `ops.container.start`
  - `ops.container.stop`
  - `ops.container.restart`

metadata：

- `container_id`
- `container_name`
- `image`
- `action`
- `runtime`: 首个本地容器运行时适配器 key
- `result`
- `error`
- `status_before`
- `status_after`

审计记录不得包含敏感环境变量、secret、token、authorization header、Docker inspect 原始敏感字段。

## 9. 系统配置设计

第一版配置 key：

```text
ops.container.runtime.enabled = false
ops.container.runtime = first-adapter
ops.container.docker.endpoint = unix:///var/run/docker.sock
ops.container.logs.default_tail = 200
ops.container.logs.max_tail = 2000
ops.container.actions.dangerous_enabled = false
```

实施约束：

- 模块加载开关仍然只由启动期配置控制，例如 `GRAFT_MODULES_ENABLED` / `modules.enabled`。
- 系统配置不能被解释为 backend module enable/disable authority；模块运行时页只读展示当前进程内模块注册、启用、
  依赖、迁移、schema 和配置状态。
- `ops.container.runtime.enabled` 是容器运行时访问开关，不是 container 模块启用开关。
- 当 `ops.container.runtime.enabled=false` 时，container 模块仍可加载，但 API 不得访问 Docker/container runtime，
  并返回 `ops.container.error.runtimeDisabled`；前端文案应表达“容器运行时访问未启用”。
- 当 `ops.container.runtime.enabled=true` 时，仅表示允许访问容器运行时，仍可能因 endpoint、socket 或权限返回
  `ops.container.error.runtimeUnavailable` 等连接类错误。
- `ops.container.actions.dangerous_enabled` 是独立高危操作开关，只控制 `start` / `stop` / `restart` 以及未来
  `delete`、`exec`、批量删除等高危能力；即使 runtime enabled 为 true，该开关为 false 时仍只允许列表、详情、日志等
  只读能力。
- `runtime.enabled` 和 `actions.dangerous_enabled` 可以走现有 boolean resolver。
- `runtime`、`endpoint`、`default_tail`、`max_tail` 第一版优先用模块默认值或进程配置，不为 MVP 扩展全局
  `SystemConfigResolver`。
- 如果 `endpoint` 进入系统配置定义，必须标注 `Sensitive` 或 `RestartRequired`。
- 配置定义的 schema 是系统配置 UI 渲染 authority；前端不得按 key 写本地兼容映射。
- bootstrap 菜单由后端统一裁剪，依据至少包含模块是否加载、用户权限和功能开关；内部可在 menu item 上声明 feature
  gate，但不得在 OpenAPI/bootstrap menu wire shape 暴露该内部字段。
- 当 `ops.container.runtime.enabled=false` 时，MVP 先隐藏容器管理菜单；容器模块的系统配置项必须仍可在系统配置页搜索和编辑。


## 11. 安全边界与错误处理

安全边界：

- 默认不做服务器终端。
- 不做 exec。
- 不做文件编辑。
- 不做删除容器。
- 不做删除镜像。
- 不做 prune。
- 不做容器创建。
- 所有写操作必须同时通过权限、`ops.container.actions.dangerous_enabled` 和审计。

错误场景：

- 运行时 socket 不存在。
- 当前用户无权限访问运行时 socket。
- 运行时 daemon 未启动。
- 容器不存在。
- 容器状态不允许执行操作。
- 日志过大。
- 操作超时。
- 容器运行时访问未启用。
- 高危操作未启用。

API 必须返回可本地化错误 key 和 fallback message。

## 12. 分阶段实施计划

- Phase 0：设计和 active topic 持久化。
  - 新增本文档。
  - 新增 `ai-plan/public/container-management-mvp/**`。
  - 不改业务代码、不改 OpenAPI source。
- Phase 1：OpenAPI 与 contract source。
  - 修改 `openapi/**`，新增 paths/schemas/tag。
  - 生成或检查 backend/web OpenAPI derived artifacts。
  - 新增后端模块 contract 常量草案，不实现 runtime。
- Phase 2：后端模块骨架。
  - 新增 `server/modules/container` 模块骨架。
  - 注册菜单、权限、i18n、系统配置定义。
  - 接入 `moduleregistry`。
- Phase 3：后端 `DockerRuntime` adapter 与 API。
  - 实现 `ContainerRuntime` 和 `DockerRuntime`。
  - 实现 list/detail/logs/start/stop/restart。
  - 接入权限、错误映射和审计。
- Phase 4：前端页面。
  - 新增 `web/src/modules/container`。
  - 实现列表、详情 Drawer、日志 Drawer/Dialog、确认弹窗和权限按钮。
- Phase 5：验证、治理收尾和归档。
  - 完整 backend/web 验证。
  - Docker 集成测试默认跳过，只有显式环境变量启用。
  - 更新 tracking/trace，满足后归档。
