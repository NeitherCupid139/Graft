# 后端安全与信任边界治理规范

## 1. 目标

本规范定义 `Graft` 后端在认证、授权、审计、危险操作保护与信任边界上的最低治理要求。

目标：

- 防止把 `web`、外部调用方或批量任务输入误当成可信 authority。
- 将高风险操作的授权、审计与执行保护收敛到可复核的后端边界。
- 让 AI agent、人工开发、code review 与 CI 对同一套规则达成一致。

非目标：

- 不用前端提示、隐藏按钮或路由守卫替代后端授权。
- 不把一次性临时脚本、运维习惯或人工约定视为安全控制。

## 2. 信任边界

### 2.1 基本原则

- 身份、权限、租户、资源归属、可操作范围的 canonical authority 必须在 `server`。
- `web`、CLI 请求体、query、header、cookie、批量导入文件、回调参数默认都不可信。
- 所有高风险操作都必须在后端重新完成身份解析、权限判定、资源归属校验与业务前置条件校验。

### 2.2 明确禁止

以下字段不得直接信任前端传值作为最终 authority：

- `user_id`、`operator_id`、`tenant_id`
- `role`、`roles`
- `permission`、`permissions`
- `resource_owner_id`、`resource_scope`
- `is_admin`、`is_system`、`is_superuser`
- 任意表示“可越权范围”的布尔值、枚举值或 ID 列表

允许前端上传这些字段作为查询条件或候选输入，但后端必须：

- 丢弃并改用服务端上下文重建 authority，或
- 将其视为待校验输入，与服务端真实归属逐项比对

## 3. 认证与授权

### 3.1 认证最小要求

- 认证中间件必须在后端生成稳定的调用者上下文。
- 下游 handler / service 不得重新解析零散 header 作为身份真值。
- 匿名访问、系统任务、内部调用必须有显式调用者类型，禁止用空用户或默认管理员兜底。

### 3.2 授权最小要求

- 权限判定必须发生在后端，且靠近真实执行边界。
- 授权不仅校验“是否有该 permission code”，还必须校验资源级可见范围与操作前置条件。
- 对批量操作，必须逐资源或逐批次验证是否允许操作，禁止只校验入口 permission 一次后直接全量执行。
- 若一个操作会跨模块读取 authority，必须调用稳定公共服务或稳定契约，不得猜测下游模块内部规则。

### 3.3 禁止模式

- 仅因前端按钮可见就默认允许后端执行。
- 将资源 ID 可枚举性误当成授权。
- 在 service 内发现校验麻烦时，把控制退回前端。
- 用“管理员页面不会暴露给普通用户”作为授权替代。

## 4. 危险操作保护

以下操作默认视为危险操作：

- 删除、停用、封禁、踢下线、重置凭证、授权提升
- 导出敏感数据、查看密钥类字段、批量写入
- 调度立即执行、批量重试、批量回滚
- 跨租户、跨组织、跨资源所有者的批量变更

危险操作至少满足：

- 明确 permission code
- 明确资源归属校验
- 明确审计日志
- 明确失败语义
- 明确幂等或重复提交保护

对高风险或不可逆操作，至少再满足一项：

- 二次确认语义在后端可验证
- 幂等键或操作令牌
- 冷却时间 / 速率限制
- 双人复核或显式审批状态

前端确认弹窗不是安全证据；只有后端验证到的确认令牌、审批状态或幂等约束才算控制面。

## 5. 高风险批量操作

高风险批量操作必须额外治理：

- 定义最大批量上限；超过上限必须分页、异步化或走任务系统。
- 逐项记录成功、失败、跳过原因，禁止只返回“部分成功”。
- 不允许因为一条记录有权限就默认整批都有权限。
- 输入包含资源 ID 列表时，必须检查重复、越权、跨租户混入与不存在资源。
- 需要快照或版本条件时，必须在后端校验版本漂移，不能依赖前端列表页旧数据。

以下场景默认不得直接同步执行：

- 无上限的全量批处理
- 影响多个 authority owner 的跨模块批改
- 会触发外部副作用且无法回滚的批量操作

## 6. 审计日志要求

### 6.1 必记事件

以下事件必须写审计日志：

- 登录成功 / 失败、登出、会话踢下线
- 权限变更、角色绑定、授权提升、敏感配置修改
- 危险操作与高风险批量操作
- 被拒绝的越权尝试或信任边界违规输入

### 6.2 最低字段

审计日志最低应包含：

- 操作人身份
- 操作时间
- 操作类型
- 目标资源类型与标识
- 请求结果
- 失败原因或拒绝原因
- 请求来源上下文
- 关联请求 ID / trace ID

对批量操作还必须包含：

- 批量规模
- 成功数、失败数、跳过数
- 抽样或完整失败原因

### 6.3 禁止事项

- 只记录“成功”，不记录拒绝和失败。
- 把敏感明文凭证、token、密钥直接写入日志。
- 用普通业务日志替代结构化审计日志。

## 7. AI Guardrail

AI 生成或修改后端安全相关代码时，默认必须满足：

- 不新增“信任前端身份 / 权限 / 资源归属字段”的逻辑。
- 不在 `handler` 外绕过统一认证上下文。
- 不为了兼容旧接口而默默保留越权字段直通。
- 不把批量操作从逐项校验降级为整批一次校验。
- 不删除审计日志而只保留业务日志。

出现以下请求时，agent 默认拒绝直接落地，必须先升级为设计或人工决策：

- 跨模块授权重构
- 批量绕过资源级权限
- 用前端字段直传替代服务端 authority 重建
- 关闭危险操作审计

## 8. 证据要求

涉及本规范的变更，closeout 至少记录：

```text
security evidence:
- authn_context: reused | changed | not-applicable
- authz_boundary: handler | service | both
- frontend_trust_fields: none | rejected | validated
- audit_log: added | preserved | not-applicable
- dangerous_op_guard: permission / ownership / idempotency / rate-limit / approval
- batch_risk: none | bounded | async-required
- validation: <command or review evidence>
```

若本次改动未触发某项，也要明确写 `not-applicable`，不得留空。

## 9. CI 与 Review 适配

适合进入 lint / CI / 自动检查的规则：

- 禁止新增明显前端信任字段直通到 service / store
- 危险操作入口必须包含审计调用或审计封装
- 高风险批量入口必须声明上限或异步任务化路径

仍需保留在文档 / review 的规则：

- 资源归属模型是否正确
- 授权粒度是否足够细
- 审计字段是否满足业务排障需要
- 是否需要审批、冷却时间或双人复核

## 10. 评审清单

- 身份是否完全来自后端上下文，而非前端字段。
- 权限校验是否覆盖资源级归属，而不只是入口 permission。
- 危险操作是否有审计、幂等或重复提交保护。
- 批量操作是否有上限、逐项结果与越权混入检查。
- 是否误把前端按钮、页面路径或隐藏字段当成安全控制。
