# 后端测试与可维护性治理规范

## 1. 目标

本规范定义 `Graft` 后端改动的最小测试、回归、注释与可维护性要求。

目标：

- 让不同类型的改动都有最低可执行验证，不靠“看起来没问题”收尾。
- 把复杂度、注释、魔法值与兼容逻辑控制在可 review、可演进的范围内。
- 区分适合进入 lint / CI 的硬规则，与仍需文档 / review 判断的软规则。

## 2. 按改动类型的最小测试要求

### 2.1 纯文案、注释、无行为变化重构

- 至少运行受影响包可通过的最小编译或测试命令。
- 若没有直接测试入口，需提供“无行为变化”的差异说明与 reviewer 可复核证据。

### 2.2 普通业务逻辑改动

- 至少补一个直接覆盖新增或变更分支的单元测试。
- 若逻辑依赖 store / repository / service 协作，优先补 service 层测试而不是只测最外层 handler。

### 2.3 查询、过滤、排序、分页改动

- 至少覆盖正常路径与一个边界路径。
- 若改动容易引发返回集合漂移，必须补回归样本或表驱动测试。

### 2.4 权限、安全、审计、危险操作改动

- 至少覆盖允许与拒绝两条路径。
- 高风险批量操作至少覆盖越权输入、部分失败或超上限场景之一。

### 2.5 Bug 修复

- 默认先补能复现旧 bug 的回归测试，再修实现。
- 若确实无法稳定复现，closeout 必须说明为什么无法测试，以及用了什么替代证据。

### 2.6 兼容层、fallback、alias、临时桥接

- 必须覆盖新旧路径各一条。
- 必须覆盖清理条件或失效条件，避免兼容逻辑永久存活。

## 3. 回归策略

- 变更应优先补最靠近 authority 的回归测试，而不是只在下游表现层兜底。
- 修复由共享 contract、公共 service、query helper 引起的问题时，回归测试必须放在可保护未来调用方的位置。
- 对历史 bug 热点，新增测试要明确保护的失效模式，避免只复制当前实现细节。

以下场景适合表驱动回归测试：

- 权限矩阵
- 状态流转
- 过滤 / 排序组合
- 错误码与边界条件
- 兼容路径与清理路径并存

## 4. Query Count 回归适用性

仅当改动满足以下至少两项时，才适合引入 query-count 回归：

- 热路径列表接口或批量接口
- 已知存在 N+1 风险
- 之前出过性能回归
- 查询数量相对稳定，且不会因合法策略变化频繁波动

以下场景通常不适合把 query-count 作为硬断言：

- 初始化阶段或强依赖 fixture 的测试
- 合法查询计划经常随 preload / feature flag 变化
- 当前仓库还没有稳定的查询计数测试基础设施

若不适合硬断言，但 reviewer 仍担心查询膨胀，应在 review 或性能专项中记录，不要伪造脆弱测试。

## 5. 注释与文档要求

以下代码边界必须有中文说明，且与实现同步：

- public API
- public service
- 复杂函数
- 兼容层、fallback、alias、桥接逻辑

所有导出符号必须满足对应语言的导出文档基线；Go 导出符号文档仍以符号名开头。

注释重点解释：

- 职责
- 调用约束
- 失败语义
- 生命周期
- 为什么需要兼容或不能进一步简化

禁止机械注释与过期 TODO 泄漏到完成态代码。

## 6. 魔法值与复杂度

### 6.1 魔法值

以下值不得随意散落：

- 权限码
- 错误码
- 配置 key
- header 名称
- 会影响分支语义的状态枚举

局部样例值、测试数据、对可读性更友好的简单字面量可以保留，不追求零字面量。

### 6.2 函数复杂度

- 单个函数默认保持单一职责。
- 出现多层嵌套、跨阶段状态变更、权限与查询拼接混杂时，应优先拆分。
- 如果复杂度难以下降，必须用注释明确阶段边界与失败语义，而不是仅靠长函数硬撑。

以下模式应优先重构：

- 一个函数同时负责参数解析、授权、查询拼接、审计与响应转换
- 兼容分支长期与主路径交织
- 多个布尔参数控制完全不同流程

## 7. 哪些规则适合进 lint / CI

适合进入 lint / CI 的规则：

- 导出符号文档存在性与 Go doc comment 形状
- 基础复杂度阈值
- 明显未使用代码、死分支、重复定义
- TODO / FIXME 格式基线
- 受影响包最小测试命令执行

更适合保留在文档 / review 的规则：

- 测试是否真正覆盖 authority owner
- 回归样本是否足够代表历史 bug
- query-count 是否值得作为硬门禁
- 兼容层是否应该存在
- 函数是否还可以继续拆分

不要把高语义判断全部硬塞进 lint；lint 负责可执行基线，文档与 review 负责设计质量。

## 8. AI Guardrail

AI agent 在后端改动中必须遵守：

- 不因“补测试麻烦”而跳过最小直接覆盖。
- 不用更外层、更脆弱的测试替代 authority 边界回归。
- 不把 TODO、占位断言、空测试作为完成态。
- 不顺手扩散无关重构、命名整理或兼容清理。

若当前切片无法补足理想测试，至少要满足：

- 有最小直接验证
- closeout 写明缺口
- 风险落点清晰

## 9. 证据要求

涉及本规范的变更，closeout 至少记录：

```text
maintainability evidence:
- change_type: refactor | feature | bugfix | security | compatibility
- minimal_test: <command or not-run reason>
- regression_added: yes | no | not-applicable
- query_count_rule: hard-check | review-only | not-applicable
- public_docs_updated: yes | no | not-applicable
- compatibility_comment: yes | no | not-applicable
- complexity_risk: low | medium | high
```

## 10. 评审清单

- 改动类型对应的最小测试是否满足。
- bug 修复是否有回归保护，或已明确说明为何无法补。
- 复杂函数、公共服务和兼容逻辑是否有足够注释。
- 是否把本该进入 contract 的高风险字面量散落到实现里。
- 是否把应留在 review 的高语义判断错误硬编码成脆弱 lint。
