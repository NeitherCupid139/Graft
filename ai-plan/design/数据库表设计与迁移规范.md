# 数据库表设计与迁移规范

## 1. 目标

这份文档定义 `Graft` 后端新增或修改数据库表、Ent schema、Atlas migration、索引和数据库注释时的治理规则。

目标：

- 让每张 live 表都有明确 module owner、审计字段、软删除语义、索引计划和可回放 migration。
- 统一新表的软删除模型，避免 nullable timestamp、`deleted_at IS NULL`、`deleted_at = 0` 混用。
- 保持 Ent schema 注释、migration SQL 注释和数据库真实结构一致。
- 降低多模块并行开发时 migration 版本号和唯一索引冲突。

非目标：

- 不重写历史 mixed Atlas migration。
- 不维护旧 Ent/manual replay 迁移链；当前 live owner-aligned migrations 是唯一迁移权威。
- 不把数据库兼容层变成永久第二真相。

## 2. Authority 与 Owner

新增表前必须先确认：

- 表 owner 是 `server/modules/<name>` 下的业务 module，还是少数 core-owned 基础设施。
- 表是否修改其它 module 的 schema、migration 或 Ent 生成产物。
- 表是否需要跨模块 capability / stable DTO，而不是直接暴露 Ent entity、repository 或 storeent。

规则：

- 业务表默认属于 `server/modules/<name>/**`。
- 模块拥有自己的 `ent/schema/**`、`ent/**` 生成产物、`migrations/**`、store / storeent 和最小测试。
- 一个 migration 只能修改当前 owner 拥有的表或明确 core-owned 表。
- 禁止一个 module 的 migration 修改其它 module 的表。
- 跨模块关系优先使用稳定标识符、稳定外键或 application-level contract；不要通过跨模块 Ent edge 暴露内部实现。

## 3. 基础字段

新表默认包含这些字段，除非设计文档明确说明不适用：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | 按模块现有主键策略 | 是 | 按现有策略 | 主键 |
| `created_at` | timestamp / timestamptz | 是 | 按现有策略 | 创建时间 |
| `updated_at` | timestamp / timestamptz | 是 | 按现有策略 | 更新时间 |
| `created_by` | bigint / 对齐用户标识类型 | 按业务需要 | 无 | 创建人 |
| `updated_by` | bigint / 对齐用户标识类型 | 按业务需要 | 无 | 更新人 |
| `deleted_at` | bigint | 是 | `0` | 软删除时间戳；`0` 表示未删除 |
| `deleted_by` | bigint / 对齐用户标识类型 | 按业务需要 | 无 | 删除人 |

审计字段规则：

- 新 live 表必须有主键。
- 需要审计时间的业务表应包含 `created_at`、`updated_at`。
- 需要软删除的业务表应包含 `deleted_at`；是否必须软删除由业务域决定，但一旦有 `deleted_at`，必须配套中文注释。
- 时间字段类型、默认值、索引策略应遵循现有项目风格，不允许每个模块随意变化。
- `created_at`、`updated_at` 必须表达真实写入语义，不得伪造业务事件时间。
- 涉及管理后台操作、配置、策略、任务、权限、资源或其它可追责业务表时，优先补齐 `created_by`、`updated_by`、`deleted_by`。
- `deleted_by` 与 `deleted_at` 必须成对设计查询语义；软删除操作能识别操作者时必须写入删除人。
- 业务事件时间另建专用字段，例如 `occurred_at`、`started_at`、`ended_at`，不要复用审计时间。

## 4. 软删除模型

新表统一使用：

```sql
deleted_at BIGINT NOT NULL DEFAULT 0
```

语义：

- `deleted_at = 0` 表示 live row。
- `deleted_at > 0` 表示已软删除，值为删除时间戳。
- live query、store query、唯一索引和外部可见列表默认只读取 `deleted_at = 0`。
- 删除操作应同时写入 `deleted_at` 和可获得时的 `deleted_by`。

禁止在新表中使用：

- nullable timestamp soft delete。
- `deleted_at IS NULL` 作为 live row 判断。
- `deleted_at` 为 timestamp 且通过 null / non-null 表示删除状态。

兼容规则：

- 历史表若已经使用 nullable timestamp soft delete，可作为 existing-table compatibility 暂时保留。
- 保留兼容时必须记录 canonical owner、为什么本切片不直接迁移、受影响 consumer、清理触发条件和验证范围。
- 新增查询或索引不得把历史兼容语义扩散到新表。

## 5. 注释治理

当前默认 migration 链中的每张 live 表都必须有中文表注释，每个 live 列都必须有中文列注释。

Ent schema 要求：

- 新增表必须在 schema 中声明 `schema.Comment(...)`。
- 新增字段必须声明 `field.Comment(...)`。
- 表级 SQL comment 输出能力必须开启，例如结合 `entsql.WithComments(true)`。
- 注释必须表达业务语义，不得只把字段名直译成中文。

Migration SQL 要求：

- migration 必须显式包含 `COMMENT ON TABLE`。
- migration 必须显式包含每个新增或改名列的 `COMMENT ON COLUMN`。
- Ent schema 注释和 migration SQL 注释必须一致或能解释差异。
- 旧 `server/internal/ent/migrate/migrations/**` replay 链已经删除，不再作为注释治理或迁移回放来源。

注释写法：

- 使用中文。
- 避免中英混写、空泛词、实现细节或与枚举语义不一致的描述。
- 枚举、状态、来源、类型字段必须说明业务含义或值域来源。

### 5.1 Migration SQL 注释治理

所有 live migration SQL 都必须把数据库注释落到同一个 migration 文件中，不能只依赖设计文档、Ent schema
或代码注释说明字段含义。当前可执行校验入口是：

```bash
python3 scripts/validate_sql_migrations.py
```

该脚本扫描当前默认 live migration 链，并阻断缺失注释、
非中文注释、占位注释、直接复述标识符以及 live migration 版本号重复。

#### 表注释规则

所有 live migration 中的 `CREATE TABLE` 必须在同文件内提供中文表注释：

```sql
COMMENT ON TABLE notification_events IS '通知事件表';
```

要求：

- 必须是中文。
- 不允许英文说明。
- 不允许 `TODO`、`TBD`、`placeholder`、`待补充`、`临时说明`。
- 不允许仅复述表名。
- 表注释应说明业务含义，而不是技术结构。

正确示例：

```sql
COMMENT ON TABLE notification_events IS '通知事件表';
```

错误示例：

```sql
COMMENT ON TABLE notification_events IS 'Notification events';
COMMENT ON TABLE notification_events IS 'notification_events';
COMMENT ON TABLE notification_events IS 'TODO';
```

#### 列注释规则

所有 live migration 中 `CREATE TABLE` 创建的每个列都必须在同文件内提供中文列注释：

```sql
COMMENT ON COLUMN notification_events.id IS '通知事件主键';
```

`ALTER TABLE ... ADD COLUMN` 新增的每个列也必须在同文件内提供 `COMMENT ON COLUMN`。

要求：

- `id`、`created_at`、`updated_at`、`deleted_at` 等通用字段也必须注释。
- `JSON` / `JSONB` 字段必须说明内容结构、业务用途或消费边界。
- 枚举、状态字段必须说明可用值或引用的枚举 / typed contract 定义。
- 外键字段必须说明关联对象。
- 布尔字段必须说明 `true` / `false` 语义。
- 不允许英文、拼音、`TODO`、`TBD`、`placeholder`、`待补充`、`临时说明`。
- 不允许仅复述字段名。

正确示例：

```sql
COMMENT ON COLUMN notification_events.id IS '通知事件主键';
COMMENT ON COLUMN notification_events.context_json IS '通知上下文数据，用于前端本地化插值和详情展示';
COMMENT ON COLUMN scheduled_tasks.enabled IS '是否启用定时任务，true 表示参与调度，false 表示暂停调度';
```

错误示例：

```sql
COMMENT ON COLUMN notification_events.id IS 'id';
COMMENT ON COLUMN notification_events.context_json IS 'context json';
COMMENT ON COLUMN scheduled_tasks.enabled IS 'TODO';
```

#### JSON 字段规则

`JSON` / `JSONB` 字段不能只写“扩展字段”。

要求：

- 必须说明该 JSON 的边界：配置、上下文、结果、快照、外部 payload 或兼容字段。
- 如果 JSON 字段会被前端或其它模块消费，应在列注释或相邻 SQL 普通注释中说明主要 key 的语义。
- 不允许把结构化核心字段偷懒塞进 JSON，除非设计文档明确允许。

#### 索引与约束说明

PostgreSQL 不强制要求给索引写 `COMMENT`，但复杂索引、唯一约束、部分索引、表达式索引应在 SQL
邻近位置用普通 SQL 注释说明设计意图。

要求：

- 外键、唯一约束命名应稳定、可读，避免依赖自动生成的不可读名称。
- 如果新表使用部分索引，例如 `deleted_at = 0`，必须说明这是为了软删除唯一性或查询性能；`deleted_at IS NULL`
  只允许出现在历史 nullable timestamp 软删除表的兼容说明中。

#### live migration 边界

- live migration 是当前应用启动、部署、模块迁移会执行的迁移链，必须遵守当前规范。
- 旧 Ent/manual replay 迁移链已经删除，不作为 fallback、兼容回放或第二真相。
- 项目早期阶段、且无需兼容已部署数据库时，允许按 owner-aligned 目录压缩迁移链；压缩后必须从空库验证默认
  `graft migrate up` 链路。
- 如果新增 live migration 目录，必须同步更新 SQL migration 校验脚本的扫描范围，或确保它能从默认
  module registry / core migration dir 发现该目录。

#### 最终回执要求

涉及 migration SQL 的任务最终回执必须包含：

- 修改的 migration 文件列表。
- 新增 / 修改的表。
- 新增 / 修改的列。
- 是否补齐 `COMMENT ON TABLE` / `COMMENT ON COLUMN`。
- 运行的校验命令。
- 校验结果。
- 是否压缩 live migration 链；若压缩，必须说明空库回放验证结果。

## 6. 索引与唯一约束

索引设计必须先从 store query 语义出发：

- 列出主要查询路径、过滤条件、排序字段和分页方式。
- live row 查询必须包含 `deleted_at = 0`。
- 高频列表查询优先设计组合索引，字段顺序按等值过滤、范围过滤、排序字段排列。
- 外键或标识符过滤字段按实际查询频率建立索引，不机械给所有字段建索引。

唯一约束规则：

- 对软删除表，业务唯一性优先使用 PostgreSQL partial unique index：

  ```sql
  CREATE UNIQUE INDEX <index_name>
    ON <table_name> (<business_columns>)
    WHERE deleted_at = 0;
  ```

- 不要用包含 `deleted_at` 的普通唯一索引替代 live row 唯一性，除非能证明恢复、重复删除和查询语义都正确。
- 唯一索引名称必须表达表、业务列和 live row 语义。
- 跨租户或跨组织唯一性必须显式包含租户 / 组织 owner 字段。

索引注释与验证：

- 重要索引应在设计说明或 migration review 中说明服务的 query。
- 修改索引时必须说明受影响 store query 和兼容风险。

## 7. Migration 版本与文件

默认 migration 链会把所有 live module-owned migration 目录聚合成一个 Atlas 目录。

规则：

- live migration 文件版本号必须在全仓库 live migration 范围内唯一。
- `server/modules/<name>/migrations/*.sql` 的数字版本前缀必须跨所有 live module migrations 全局唯一。
- 不得只保证单个 module 目录内唯一。
- 版本号冲突时，后创建的 migration 必须改用新的全局唯一前缀。
- 推荐格式保持当前项目既有风格，例如 `202606110001_xxx.sql`。
- 新增 migration 前必须搜索现有版本号，避免与任一 live migration 目录冲突。
- migration 文件必须可审计、可回放、可按版本追踪。
- 禁止把 schema 变更藏进 `graft serve`、runtime 自动同步或启动逻辑。

Migration 内容应包含：

- owner module 能识别的表名和约束名。
- 新表的 audit / soft-delete 字段。
- 中文 `COMMENT ON TABLE` 和 `COMMENT ON COLUMN`。
- 必要索引和 partial unique index。
- 回滚或兼容说明按当前 Atlas 迁移格式执行，不另创流程。

## 8. Ent 与生成流程

当任务修改以下内容时，必须按 `server/AGENTS.md` 的后端验证链执行对应生成和测试：

- 任一 live Ent schema 路径，例如 `server/modules/<name>/ent/schema/**`。
- 任一 live Ent 生成入口，例如 `server/modules/<name>/ent/generate.go`。
- Atlas / Ent migration 生成配置。
- 任何影响 schema 语义的手写代码。

完成态不得缺少：

- 受影响 live Ent 包的 `go generate`。
- 显式 migration 文件生成或更新。
- 受影响 Ent 包的最小直接 `go test`。
- 后端验证链中要求的直接覆盖验证。
- live 表 / 列中文注释检查。

如果当前切片只是治理文档或设计，不改真实 schema / migration，应在 closeout 中说明未运行 Ent generate 和后端 runtime 验证的原因。

## 9. 设计输出模板

设计或评审数据库表时，至少输出：

```text
Table design summary:
- owner_module:
- table_name:
- fields:
- audit_fields:
- soft_delete_model:
- store_query_semantics:
- index_plan:
- comments_plan:
- migration_version:
- validation:
- compatibility_notes:
```

`soft_delete_model` 对新表必须写明 `deleted_at BIGINT NOT NULL DEFAULT 0` 和 live query `deleted_at = 0`。
