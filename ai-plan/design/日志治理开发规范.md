# 日志治理开发规范

## 1. 目标与适用范围

本规范定义 `Graft` 在 MVP 阶段的后端功能开发日志治理规则，统一约束以下能力边界：

- `App Log`
- `Access Log`
- `Error Log`
- `Audit Event`
- `Security Event`
- `Metric Candidate / Metric Placeholder`

本规范适用于：

- `server/internal/**`
- `server/modules/**`
- `server/cmd/**`
- 任何会影响 `server -> web` 审计、访问控制、审计控制台消费语义的共享字段定义

本规范不引入：

- OpenTelemetry
- Prometheus / exporter
- 完整 metrics 后端
- 远端日志 SaaS
- 把 audit log 当普通 app log 的统一抽象

Authority summary：

- `server/internal/logger/**` 是 `App Log / Error Log` 的后端主基线
- `server/internal/httpx/**` 是 HTTP request correlation、`Access Log` 与 HTTP `Security Event` bridge authority
- `server/internal/audit/**` + `server/modules/audit/**` 是 `Audit Event` / `Security Event` 持久化 authority
- `web` 只消费审计/安全治理产物，不拥有这些后端字段语义的最终 authority

Observability authority overlay：

- `server/modules/audit/**` 拥有 audit facts、incident read model、audit analytics 与 audit evidence consumer 语义
- `server/modules/monitor/**` 拥有 monitor facts、server status、anomaly、trend 与 monitor-owned evidence 语义
- `openapi/**` 拥有共享 HTTP contract authority；`server/internal/contract/openapi/**`、`openapi/dist/**` 与 `web` generated schema 都是 derived artifacts
- `EvidenceLink` 是 backend-owned canonical drilldown contract；`web` 只消费 `target_kind`、`link_state`、`audit_context`、`incident_seed`、`time_window`
- `audit overview -> audit logs` 业务钻取必须由 backend-owned `scope + business_category` contract 定义；`web` 不得再通过普通 query 参数拼接业务语义筛选
- `web` 可以拥有返回导航、页面 origin、筛选器状态等 UI consumer context，但这些值不得提升为 audit/monitor/evidence authority
- `traceId` 在当前 MVP 仍等价于 `requestId` 的相关性字段，不表示独立 tracing authority
- `Metric Candidate / Metric Placeholder` 仍是治理占位；除 monitor 当前 short-retention trend read model 外，不存在通用 `MetricsEmitter` authority

## 2. 日志 / 事件分类

### 2.1 App Log

用于记录应用运行过程中的结构化业务外信息，帮助排查状态迁移、启动/关闭、后台任务执行、外部依赖调用结果、显式分支决策。

当前 canonical API：

- 默认使用 `server/internal/logger.AppLogger`
- `AppLogger` 负责：
  - 统一 severity surface：`Debug / Info / Warn / Error`
  - 统一字段清洗与敏感值脱敏
  - 自动拼装来自请求上下文的 correlation 字段
  - `Named("module.component")` 形式的 component 标识
- raw `*zap.Logger` 是底层 backend，不再是普通应用运行日志的新默认调用面

应写 `App Log` 的场景：

- module `Register / Boot / Shutdown` 的关键生命周期节点
- 后台任务开始、结束、跳过、重试
- 外部集成适配器的请求发送、响应摘要、降级路径
- 明确需要保留的业务关键流程摘要，但该事件本身不构成审计证据

不应写 `App Log` 的场景：

- 每个普通 CRUD 成功都重复记一条“操作成功”
- HTTP 请求入口/出口访问明细
- 已由 `Audit Event` 或 `Security Event` 覆盖的同语义证据
- 高频 repository 正常读写

`AppLogger` 采用规则：

- 运行时普通应用日志默认写 `AppLogger`
- 当日志需要请求关联字段时，必须优先传递现有 `ctx` 给 `AppLogger`
- 不得为了迁移便利在 handler / service 内继续手工补 `requestId`、`traceId`、`route`、`method`
- 若一段日志明确属于 access / audit / security authority，则不得强行迁移到 `AppLogger`

### 2.2 Access Log

用于记录 HTTP 请求访问事实，属于请求级观测，不承载业务审计结论。

当前 runtime 事实：

- canonical owner：`server/internal/httpx/**`
- durable storage：PostgreSQL `access_logs`，由 `server/internal/httpx/**` 拥有 schema、repository、explorer read handlers 与 live migrations
- retention lifecycle：`server/internal/httpx/**` 拥有 access-log cleanup/retention runtime
- boundary：`Access Log` 仍只记录 request fact，不替代 `Audit Event`、`Security Event` 或 `App Log`

应写 `Access Log` 的场景：

- 所有 HTTP 请求由统一 middleware 记录

禁止：

- 在 handler / service 中手工补第二条 access log
- 用 access log 替代审计

### 2.3 Error Log

用于记录需要排查的失败、异常、降级、重试耗尽、依赖错误和不可忽略的系统状态偏差。

应写 `Error Log` 的场景：

- 返回上层前需要保留排障信息的错误
- 异步审计/安全事件写入失败
- 外部依赖超时、连接失败、协议错误
- 后台任务失败、部分失败、最终失败
- shutdown/flush 失败

禁止：

- 同一错误沿调用链每层都打一遍
- 已有统一 error handling 时再局部重复打印
- 记录敏感明文

raw `*zap.Logger` 允许例外：

- `server/internal/logger/**`
  - logger 构造、全局替换、关闭、底层 zap field 拼装
- `server/internal/httpx/**`
  - `Access Log`
  - access-retention runtime
  - security-event publish bridge
- `server/internal/audit/**`、`server/modules/audit/**`
  - audit-owned 记录、策略跳过与读模型错误
- `server/internal/eventbus/**`、`server/internal/scheduler/**`
  - 基础设施运行时内部失败
- Ent debug、CLI bootstrap fallback、测试代码、第三方/生成代码

除这些边界外，新增普通应用错误日志默认应落到 `AppLogger`。

### 2.4 Audit Event

用于记录可被产品和合规面消费的审计证据，强调“谁在什么上下文对什么资源做了什么，结果如何”。

必须写 `Audit Event` 的场景：

- 后台用户主动发起的资源变更
- 角色/权限/访问控制配置变更
- 安全相关人工操作，如登录、登出、撤销会话、重置凭据、启停高风险开关
- 明确需要进入审计控制台的数据

禁止：

- 把系统调试日志、普通请求噪音、无资源语义的技术明细塞进审计表
- 绕过统一 audit recorder 直接写 repository

### 2.5 Security Event

用于记录安全语义事件，是 `App Log` 与 `Audit Event` 之间的桥接层，强调认证、鉴权、拒绝、异常令牌、可疑输入、策略阻断等安全事实。

必须写 `Security Event` 的场景：

- token 缺失 / 过期 / 非法
- 权限拒绝
- 高风险安全策略命中
- 需要后续进入审计策略判断的安全相关拒绝/失败

规则：

- `Security Event` 可以同时产生 `App Log` 与 `Audit Event` 效果，但 authority 仍在统一安全事件入口和统一 audit path
- 不得在业务代码里平行维护第二套安全事件模型

### 2.6 Metric Candidate / Metric Placeholder

用于标记“未来可能需要指标化”的关键路径，不是实际指标实现。

适用场景：

- 请求耗时分布点
- 审计写入失败计数点
- 安全拒绝事件计数点
- 后台任务成功/失败计数点
- 外部集成调用耗时与失败点

规则：

- 只记录为注释、文档、命名常量、结构化字段约定或 inventory 条目
- 不输出 Prometheus endpoint
- 不引入伪 `MetricsEmitter` 后端

### 2.6.1 Logging Readiness Gate

进入未来 Phase D logging / log explorer 主题前，至少要满足以下治理前提：

- `AppLogger`
  - authority 已明确且统一收口在 `server/internal/logger/**` 与前端 shell-owned logger baseline
- `AccessLogger`
  - authority 已明确且统一收口在 `server/internal/httpx/**`
- `AuditRecorder`
  - authority 已明确且通过统一 audit path 持久化，不允许旁路 repository
- `SecurityEvent`
  - authority 已明确且通过统一安全事件入口桥接到 audit path
- `MetricsEmitter`
  - 若仍不存在 canonical runtime surface，则 readiness 只能判定为 `Partially Ready` 或 `Not Ready`

只要上述任一条仍停留在 archive evidence 拼接、临时 metadata 推断、或 future placeholder 阶段，就不得把系统判定为 Phase D fully ready。

### 2.7 禁止写日志的场景

- 已有上游统一入口记录，当前层只是在透传
- 高频成功路径没有诊断价值
- 包含密码、token、secret、cookie、完整 Authorization、邮箱等敏感字段
- 只为了“看起来做了观测”而重复记录

## 3. 后端功能开发切口矩阵

| Layer | App Log | Error Log | Audit Event | Security Event | Metric Candidate | Notes |
| --- | --- | --- | --- | --- | --- | --- |
| HTTP middleware | 仅平台级 middleware 生命周期或关键分支可写 | 记录 middleware 自身失败或恢复事件 | 不直接写普通审计；只允许统一入口构造请求审计候选 | HTTP 安全拒绝、token 异常必须在统一入口发布 | 可标记请求耗时、拒绝计数候选 | `Access Log` 只在 middleware 统一写 |
| HTTP handler | 默认不写成功日志 | 可在返回前记录无法上抛或需要补充上下文的错误 | 对资源变更应调用下游统一审计入口，不直接持久化 | 不单独拼安全事件，优先调用统一安全事件/鉴权入口 | 可标记 handler 级慢操作候选 | handler 只编排输入输出，不应堆积日志逻辑 |
| application service | 可写关键业务步骤摘要 | 记录关键失败、重试耗尽、业务异常 | 负责触发业务审计事件候选 | 仅当服务拥有明确安全语义且没有上游统一入口时触发 | 可标记关键业务路径候选 | 必须继承 `ctx`，不要新造 correlation |
| domain service | 默认极少写 | 仅记录领域计算异常且上层无法补足语义时写 | 通常不直接写；由应用服务决定是否审计 | 通常不直接写 | 可标记未来热点候选 | 保持纯语义，避免基础设施噪音 |
| repository | 禁止成功日志 | 仅记录存储层自身错误且会丢失关键信息时写 | 禁止直接写审计仓储旁路 | 禁止 | 可标记慢查询/失败候选 | repository 不拥有审计/安全语义 |
| module bootstrap/register/boot | 允许记录注册、启动、关闭摘要 | 记录注册冲突、依赖缺失、启动失败 | 禁止 | 禁止 | 可标记启动耗时候选 | 属于平台/模块生命周期日志 |
| CLI command | 允许记录显式执行阶段摘要 | 记录参数错误、执行失败、退出前 fatal | 一般不写，除非 CLI 真正触发需留痕的管理动作 | 涉及安全治理 CLI 时可触发 | 可标记命令耗时候选 | CLI 输出与日志分离；面向用户的 stdout/stderr 不等于 app log |
| scheduled/background job | 允许记录作业开始、结束、跳过、批次摘要 | 必须记录失败、重试、最终失败 | 作业触发的资源变更若需留痕必须写 | 安全巡检或策略作业命中时写 | 可标记 job latency / failure 候选 | 非 HTTP 场景必须用 job ctx 继承 correlation |
| external integration adapter | 允许记录调用摘要、下游状态 | 必须记录调用失败、超时、协议异常 | 默认不写，除非外部操作本身是受审计管理动作 | 外部认证/授权异常可映射为 security candidate | 可标记 dependency latency/failure 候选 | 严禁输出敏感请求体、token、secret |

补充约束：

- 上表中的普通 `App Log / Error Log` 默认通过 `AppLogger` 写出
- 只有表中已经标记为 access / audit / security / infrastructure owner 的条目，才允许把 raw zap 当一线 API

## 4. Correlation 规范

### 4.1 Canonical fields 来源

- `requestId`
  - HTTP 场景来自 `server/internal/httpx` 统一 request-id middleware
- `traceId`
  - MVP 阶段与 `requestId` 同值
  - 当前语义是“统一请求关联标识”，不是分布式 tracing span/trace
- `actorId`
  - 来自认证后的 request auth context、显式 job actor、或 CLI/system actor
- `route`
  - HTTP 场景来自 Gin 路由模板或统一 request audit snapshot
- `method`
  - HTTP 场景来自请求 method
- `clientIp`
  - HTTP 场景来自统一 request audit snapshot
- `userAgent`
  - HTTP 场景来自统一 request audit snapshot

### 4.2 HTTP 场景

- 必须通过 `httpx` 把 correlation 注入 `context.Context`
- handler、service、repository、audit recorder、security-event publisher 全部继承同一个 `ctx`
- 不允许在 handler/service 中手工拼第二套 request correlation struct

### 4.3 非 HTTP 场景

非 HTTP 场景必须显式提供一个受控 `ctx`：

- scheduled job
  - 使用 job lifecycle ctx，必要时补 `actorType=system`
- CLI
  - 使用命令执行 ctx，必要时补 `actorType=operator` 或 `system`
- event-driven / background worker
  - 使用调度/消费入口 ctx

降级规则：

- 没有 HTTP request 时，`requestId` 可为空或使用明确生成的 job/command correlation id
- `traceId` 在无 request 时可与该 job/command correlation id 相同
- 不得为了“字段齐全”而伪造来自 HTTP 的 route/clientIp/userAgent

### 4.4 禁止事项

- 禁止手工拼装第二套 correlation context
- 禁止在 module 内部长期维持私有 request-id 传播模型作为 canonical path
- 兼容 helper 可暂存，但不得成为新开发默认入口
- 禁止把前端 query、页面私有 route 拼接、或 metadata fallback 提升为 correlation / evidence authority

### 4.5 EvidenceLink Governance

- `EvidenceLink` 的 canonical owner 在 backend authority：
  - audit-owned read model DTO
  - OpenAPI source schema
  - monitor capability 暴露的 bounded evidence contract
- 前端只能消费以下稳定语义：
  - `target_kind`
  - `link_state`
  - `audit_context`
  - `incident_seed`
  - `time_window`
- 前端允许拥有：
  - return-to-monitor 一类 UI origin context
  - 页面级可用 link 选择策略
  - 仅影响展示的本地提示文案
- 前端不得拥有：
  - 新的 evidence target kind
  - 新的 incident seed 语义
  - 基于 metadata 的长期路由 authority
  - 与 OpenAPI / backend contract 并行的第二套 EvidenceLink shape
- `metadata` 只能作为过渡回退，不得继续充当前端路由、evidence target、或 audit/monitor authority 的长期真相

## 4.6 Observability Boundary Matrix

| Surface | Canonical owner | Can expose to others | Must not do |
| --- | --- | --- | --- |
| Audit | `server/modules/audit/**` | audit logs, incidents, analytics, audit evidence consumer read model | 推断 monitor anomaly truth、把 access log 直接当 audit truth |
| Monitor | `server/modules/monitor/**` | server status, anomalies, bounded evidence links, short-retention trends | 拥有 audit incident truth、推断 audit policy/result |
| Logging | `server/internal/logger/**` + `server/internal/httpx/**` | app/error/access/security bridge fields | 取代 audit/monitor 的业务 authority |
| Metrics | current monitor read model only | bounded trend payloads | 与 audit analytics 混用、伪造通用 metrics backend |
| Tracing | none in MVP | requestId alias semantics only | 把 `traceId=requestId` 解释为已落地 tracing platform |

## 4.7 Log Explorer Authority Model

Phase D `Log Explorer` 必须建立在独立的 logging authority 之上，而不是建立在 `audit` 表、monitor trend payload、或前端 metadata 推断之上。

### 4.7.1 Domain boundary

- `Audit Domain`
  - canonical owner: `server/modules/audit/**`
  - owns:
    - `Audit Log`
    - `Audit Event`
    - audit incident read model
    - audit analytics
    - audit evidence consumer semantics
- `Log Explorer Domain`
  - current access-log owner: `server/internal/httpx/**`
  - future app-log owner: `server/internal/logger/**` only after durable storage/readiness approval
  - currently exposes:
    - `Access Log Explorer`
  - may eventually expose only after approval:
    - `Application Log Explorer`
    - bounded correlation search over canonical logging fields
- `Security Event`
  - current authority is the unified `server/internal/httpx/** -> server/internal/audit/** -> server/modules/audit/**` path
  - 在 MVP 当前阶段它属于 `Audit Domain` 的可持久化安全证据，不属于独立 `Log Explorer` authority
- `MetricsEmitter`
  - 当前不存在 canonical runtime authority
  - 不属于 `Log Explorer` 设计输入 authority；只能作为 readiness gap 单独记录

### 4.7.2 Audit vs Log Explorer

- `Audit Log`
  - 属于 `Audit Domain`
  - 不属于 `Log Explorer` 底座
- `Access Log`
  - 属于 current `Log Explorer Domain` 的 request-fact surface
  - 不属于 `Audit Domain`，除非已有 audit policy 明确把某些请求候选升级为审计记录
- `Application Log`
  - 属于 future `Log Explorer Domain`
  - 不属于 `Audit Domain`
- `Security Event`
  - 当前属于 `Audit Domain`
  - 允许未来在 `Log Explorer` 中以关联证据方式跳转查看对应 request/app log 上下文
  - 禁止把 security-event persistence authority 从 audit path 平移为 log-explorer-owned truth

允许关联：

- `Audit Incident -> related Access Log`，仅通过 canonical correlation fields，例如 `requestId`、`traceId`、`actorId`、`resource`、bounded time window
- `Access Log -> related Audit Log`，仅当 audit authority 已存在匹配记录
- `Application Log -> related Access Log`，仅通过 canonical logging correlation fields

禁止混用：

- 直接把 `audit_logs` 表作为 `Access Log Explorer` 或 `App Log Explorer` 的底层存储
- 把 access log 数量、app log 明细或 monitor trend payload 解释为 audit truth
- 通过前端 query、metadata fallback、或 page-local heuristics 发明第二套 log correlation authority

### 4.7.3 Retention authority

当前仓库中以下 retention authority 是显式已定义的：

- `server/modules/monitor/**`
  - short-retention trend authority
  - current evidence:
    - `maxTrendRetentionWindow = 1h`
    - storage TTL = `2h`
- `server/internal/httpx/**`
  - access-log durable lifecycle authority
  - current evidence:
    - PostgreSQL `access_logs`
    - access-log explorer read handlers
    - access-log retention cleanup runtime
- `server/modules/audit/**`
  - audit-log durable evidence lifecycle authority
  - current evidence:
    - PostgreSQL `audit_logs`
    - audit-owned retention config
    - audit-owned retention cleanup runtime registered through scheduler mechanics

当前仓库中以下 retention 已在 Phase D App Log Durable Explorer Batch 1 形成正式 authority：

- `Application Log` retention
  - PostgreSQL `app_logs`
  - logger-owned repository sink
  - logger-owned retention cleanup runtime

仍未形成正式 authority：

- `Application Log` archive/export

因此：

- `App Log` purge 只能通过 logger-owned retention cleanup over `app_logs`
- 不得在 runtime 中发明默认 archive/export 行为并反向要求文档追认

### 4.7.3.1 Runtime logging matrix

| Surface | Canonical owner | Current write path | Current storage authority | Lifecycle owner | Current retention authority |
| --- | --- | --- | --- | --- | --- |
| `AppLogger` / `App Log` | `server/internal/logger/**` | `server/internal/app/**`、`server/cmd/graft/**` 与 modules 共享 `AppLogger`/`*zap.Logger`；Zap output 始终保留，repository sink 可选启用 | PostgreSQL `app_logs` owned by `server/internal/logger/**` when `GRAFT_LOG_APP_LOG_PERSIST=true` | core runtime owns logger init/close；logger owns repository sink and cleanup lifecycle | implemented in `server/internal/logger/**` |
| `AccessLogger` / `Access Log` | `server/internal/httpx/**` | unified access-log middleware persists request facts through httpx-owned repository while retaining request correlation semantics | PostgreSQL `access_logs` owned by `server/internal/httpx/**` | HTTP runtime middleware lifecycle plus httpx-owned explorer/retention wiring | implemented in `server/internal/httpx/**` |
| `AuditRecorder` / `Audit Log` | `server/internal/audit/**` + `server/modules/audit/**` | `requestAuditMiddleware` 或 `eventbus -> moduleapi.AuditEvent -> recorder.RecordCandidate -> audit repository` | PostgreSQL `audit_logs` table owned by `server/modules/audit/**` | audit module register / boot / shutdown; repository lifecycle follows shared DB runtime | implemented in `server/modules/audit/**` |
| `SecurityEvent` | publish authority in `server/internal/httpx/**`; persistence authority in `server/internal/audit/**` + `server/modules/audit/**` | `RequirePermission` failure path -> `SecurityAuditPublisher` -> `eventbus` -> audit recorder | no standalone storage; persisted as audit records in PostgreSQL when policy includes them | HTTP auth/authz guard publishes; audit module persists | inherits current audit-log retention governance state |

Matrix constraints:

- `AccessLogger` 已经拥有 repository-owned durable storage、explorer read surface 与 retention cleanup；该 authority 只覆盖 request-fact truth。
- `AppLogger` 当前拥有 logger-owned durable storage foundation；它仍不替代 access / audit / security authority。
- `SecurityEvent` 当前不是独立日志产品；其最终持久化 truth 仍落在 audit path，而不是 log-explorer-owned storage。
- `AuditRecorder` 已经明确落到 durable store，audit-log retention 也已由 `server/modules/audit/**` 拥有并通过 scheduler 机制执行。
- `AppLogger` 的 adoption rule 是“语义与存储同归 logger authority”：普通应用运行日志进入 `AppLogger`，durable sink 只持久化 canonical App Log troubleshooting fields。

### 4.7.3.2 Retention policy authority

本仓库在 MVP 阶段对 retention 的治理结论必须以“当前 authority 已存在什么”为前提，而不是反推理想平台形态。

`Audit Log`:

- canonical owner: `server/modules/audit/**`
- retention authority: audit module owns retention policy definition because only audit module currently owns durable audit evidence truth
- runtime responsibility: audit-owned cleanup job under scheduler lifecycle
- MVP governance recommendation:
  - keep the explicit retention window in audit-owned config
  - archive is `not-ready`; current repo has no second durable audit archive authority
  - purge may only target `audit_logs` through the audit-owned retention cleanup path

`Access Log`:

- canonical owner: `server/internal/httpx/**` for request semantics, storage, explorer read handlers, and retention cleanup
- current storage authority: PostgreSQL `access_logs` owned by `server/internal/httpx/**`
- retention verdict: implemented for repository-owned access-log durable lifecycle
- MVP governance recommendation:
  - keep access-log data as request-fact truth only
  - do not use access-log storage as audit/security/app-log storage
  - future production retention tuning, partitioning, or explorer expansion must stay in new bounded topics under `server/internal/httpx/**`

`Application Log`:

- canonical owner: `server/internal/logger/**` for semantics
- current storage authority: PostgreSQL `app_logs` owned by `server/internal/logger/**`
- retention verdict: implemented for repository-owned app-log durable lifecycle
- MVP governance recommendation:
  - keep app-log data as runtime troubleshooting truth only
  - do not use app-log storage as access/security/audit storage
  - archive/export remains `not-ready`; current repo has no second durable App Log archive authority

Authority rule:

- only persisted runtime datasets may own repository cleanup jobs
- therefore repository-owned cleanup authority can currently be assigned to `access_logs`、`audit_logs` and `app_logs`
- `application log` cleanup must target logger-owned `app_logs`, not stdout/stderr output

### 4.7.3.3 Storage authority

Allowed current storage authorities:

| Surface | Allowed now | Reason |
| --- | --- | --- |
| Audit log | PostgreSQL `audit_logs` owned by `server/modules/audit/**` | already implemented durable authority with repository-owned schema and repository |
| Security event persistence | PostgreSQL `audit_logs` through audit path | security events currently become audit evidence, not a second storage line |
| App log | PostgreSQL `app_logs` owned by `server/internal/logger/**` | implemented runtime troubleshooting storage, repository read model foundation, and retention cleanup authority |
| Access log | PostgreSQL `access_logs` owned by `server/internal/httpx/**` | implemented request-fact storage, explorer read surface, and retention cleanup authority |

Disallowed current storage authorities:

| Candidate | Verdict | Reason |
| --- | --- | --- |
| PostgreSQL for app logs outside `server/internal/logger/**` | forbidden | app-log durable storage authority already belongs to `server/internal/logger/**` |
| PostgreSQL for access logs outside `server/internal/httpx/**` | forbidden | access-log durable storage authority already belongs to `server/internal/httpx/**` |
| Redis for access/app logs | forbidden-now | Redis is only proven for bounded monitor short-retention data, not logging truth |
| Reusing `audit_logs` as access/app log store | forbidden | violates `Audit Domain` vs `Log Explorer Domain` boundary |
| OpenAPI log query contract before storage authority | forbidden | shared contract cannot outrun canonical storage authority |

Future observability only:

| Candidate | Verdict | Reason |
| --- | --- | --- |
| External log platform / SaaS / OTLP sink | future observability | outside MVP; no canonical contract or lifecycle owner yet |
| Metrics/tracing backend derived from logs | future observability | explicitly out of scope and forbidden for this topic |
| Unified searchable log platform across app/access/audit | future observability | requires new storage, retention, and contract authority topic |

### 4.7.3.4 Cleanup ownership

| Operation | Surface | Recommended authority owner | Current runtime status |
| --- | --- | --- | --- |
| retention cleanup | audit logs | `server/modules/audit/**` job using scheduler-owned runtime lifecycle | implemented |
| archive | audit logs | none in MVP | not-ready |
| purge | audit logs | audit-owned retention cleanup over `audit_logs` | implemented |
| retention cleanup | access logs | `server/internal/httpx/**` owns request-fact cleanup over `access_logs` | implemented |
| archive | access logs | none | not-ready |
| purge | access logs | `server/internal/httpx/**` cleanup runtime over `access_logs` | implemented |
| retention cleanup | app logs | `server/internal/logger/**` cleanup runtime over `app_logs` | implemented |
| archive | app logs | none | not-ready |
| purge | app logs | logger-owned retention cleanup over `app_logs` | implemented |

Ownership rules:

- cleanup owner must be the same authority that owns durable persisted truth
- scheduler provides runtime execution mechanics, not policy ownership
- core runtime may host shared scheduler/logger/database primitives, but must not silently seize audit retention policy ownership from the audit module
- deployment environment may rotate stdout/file sinks outside repo authority, but that is not repository runtime cleanup authority

### 4.7.3.5 Remaining governance gaps

- `App Log` OpenAPI and web Explorer are downstream from the logger-owned read model and remain Batch 2 scope.
- `App Log` archive/export remains `not-ready` until a second durable archive authority is approved.
- `App Log` must not expose access/audit/security-owned query fields through API or UI.

Required-before-next-runtime-slice:

1. keep access-log governance wording aligned with the implemented `server/internal/httpx/**` durable storage, explorer, and retention facts
2. keep audit-log retention authority and cleanup ownership aligned with `server/modules/audit/**`
3. keep app-log storage, repository, and retention cleanup aligned with `server/internal/logger/**`
4. add OpenAPI and web Explorer only as downstream consumers of logger-owned canonical fields and permission semantics
5. keep App Log archive/export as `not-ready` until a second durable store authority is approved

### 4.7.3.6 App Log storage foundation

`App Log` 的最小 storage authority foundation 固定为：

- canonical owner
  - `server/internal/logger/**`
- runtime entry
  - `AppLogger`
- current storage mode
  - `repository_durable_store`
- current retention owner
  - `server_internal_logger`
- current default retention policy
  - `GRAFT_LOG_APP_LOG_RETENTION`，默认 local/test/dev 为 3 天、staging 为 7 天、production 为 14 天

canonical persisted fields：

- `occurred_at`
- `severity`
- `component`
- `message`
- `operation`
- `request_id`
- `trace_id`
- `route`
- `method`
- `error`
- `fields`

forbidden persisted fields：

- access-log owned
  - `path`
  - `status_code`
  - `request_size`
  - `response_size`
  - `client_ip`
  - `user_agent`
- audit / security owned
  - `actor_id`
  - `actor_type`
  - `resource_type`
  - `resource_id`
  - `action`
  - `decision`
  - `policy`
  - `permission`
  - `audit_id`
  - `security_event_id`
  - `session_id`
- identity / credential-like
  - `user_id`
  - `username`
  - `authorization`
  - `cookie`

boundary rules：

- `AppLogger` 负责 runtime 写出语义，并可通过 logger-owned repository sink 写入 durable storage
- `AppLogRepository` 只能归属 `server/internal/logger/**`
- `app_logs` schema、repository implementation、operator workflow、cleanup lifecycle 已在 Phase D App Log Durable Explorer Batch 1 批准并落地
- query API 与 explorer UI 仍为 Batch 2 downstream scope，不得绕过 OpenAPI / permission authority
- 不得把 access / audit / security 字段通过 `fields` map 绕道塞回 App Log persisted truth

minimum operator workflow before approving durable storage / explorer：

- developer
  - 负责开发期与本地/测试环境下的 runtime 分支、启动、任务失败排查
  - 默认依赖 `process_output_only` 与部署外部采集，不构成仓库内 durable-store 的直接批准理由
- operator / admin
  - 负责生产或准生产环境中的 runtime 故障调查
  - 只允许围绕 bounded troubleshooting workflow 提出 future query/explorer 需求
- support / debugging role
  - 负责按 `request_id` / `trace_id` 协助定位一次请求或一次后台执行链路
  - 只能消费 logging correlation，不拥有 access / audit / security authority

future `App Log` operator workflow 必须限制为以下最小 use cases：

1. runtime error investigation
   - 从错误告警、error message 或 component seed 开始
   - 在 bounded time window 内查看同 component / same operation 的 `error`、`warn`、必要时 `info`
   - 目标是确认 runtime 故障点、降级路径、重试结果
2. background job failure investigation
   - 从 job name / component / failure time 开始
   - 查看作业开始、跳过、重试、最终失败摘要
   - 目标是判断是否需要重跑、回滚、或进入更高层 incident 流程
3. module startup/config failure investigation
   - 从 module/component + startup window 开始
   - 查看 `Register / Boot / Shutdown` 关键失败摘要与配置校验失败
   - 目标是判断模块依赖缺失、配置错误、外部依赖不可达等启动阻断原因
4. request-correlated troubleshooting
   - 从 `request_id` 或 `trace_id` 开始
   - 只查看 bounded app-log context，并可在 future explorer 中跳转到 related access/audit evidence
   - `App Log` 自身不补充 access facts、actor facts、audit facts

required app-log query dimensions：

- `occurred_at`
  - required as time-window boundary
- `severity`
  - required for error-first / warn-first narrowing
- `component`
  - required for module/runtime ownership定位
- `operation`
  - required for job、startup step、integration action narrowing
- `request_id`
  - required for request-correlated troubleshooting
- `trace_id`
  - required as the MVP alias of `request_id`
- `message` keyword search
  - allowed only as bounded keyword/message search, not as arbitrary DSL

forbidden app-log query dimensions / surfaces：

- access-traffic analytics
  - `path`、`status_code`、`request_size`、`response_size`、`client_ip`、`user_agent`
- audit / compliance review
  - `actor_id`、`actor_type`、`resource_type`、`resource_id`、`action`、`decision`、`policy`、`permission`
- security incident timeline
  - standalone `security_event_id`、session timeline、credential / token timeline
- metrics dashboard
  - aggregation-first count、latency percentile、trend board、SLO / SLA semantics
- tracing span view
  - span tree、parent/child span、service map、distributed trace waterfall
- generic free-form search platform
  - arbitrary JSON filter、regex DSL、field-path query、cross-domain mixed query

relationship constraints：

- `App Log Explorer`
  - future runtime-fact troubleshooting surface only
- `Access Log Explorer`
  - owns request traffic facts, transport attributes, and access timeline query
- `Audit` / `Security Event`
  - own compliance, actor, permission decision, and security evidence truth
- `Monitor`
  - owns anomaly, trend, and monitor evidence semantics
- operator jump rules
  - allow `App Log -> Access Log -> Audit/Security` through canonical correlation only
  - forbid using `App Log` as a substitute root for audit review, access analytics, or monitor dashboards

workflow decision after Batch 1 approval：

- current App Log storage mode
  - `repository_durable_store` when `GRAFT_LOG_APP_LOG_PERSIST=true`; zap output remains the always-on process sink
- deployment-owned external collector
  - still allowed as deployment concern, but no longer the only repository-supported durable path
- internal durable store
  - approved for canonical troubleshooting fields only
- internal explorer
  - not approved in Batch 1; explorer remains Batch 2 downstream from OpenAPI and permission contract
- durable-store status after Batch 1
  - `approved-implemented`
  - retention owner: `server_internal_logger`
  - cleanup lifecycle: `logger.app-log-retention-cleanup`
  - default retention window: local/test/dev 3 days, staging 7 days, production 14 days

retention/authz/storage readiness verdict：

- runtime readiness
  - `Ready for Batch 2 downstream contract and explorer`
- durable storage approval
  - `Approved`
- schema approval
  - `Approved`
- API/UI approval
  - `Not Approved in Batch 1`
- current repository retention owner
  - `server_internal_logger`
- current repository retention window
  - `GRAFT_LOG_APP_LOG_RETENTION`
- required next approval gate
  - Batch 2 must define read permission, OpenAPI paths, and web Explorer consumer using only logger-owned canonical fields
- forbidden shortcuts
  - do not reuse `audit_logs`, `access_logs`, Redis TTL storage, `access_log.read`, `audit.read`, frontend metadata, or menu visibility as App Log durable-storage or authorization authority

### 4.7.4 Contract ownership

未来 `Log Explorer` 的 authority chain 必须固定为：

1. logging semantics owner
   - `server/internal/logger/**` for `AppLogger` / `Error Log`
   - `server/internal/httpx/**` for `Access Log` and request correlation
2. shared wire contract owner
   - `openapi/**` once log explorer APIs are explicitly approved in a later runtime topic
3. generated artifact consumers
   - `server/internal/contract/openapi/**`
   - `web/src/contracts/openapi/generated/**`
4. frontend module consumer
   - future `web/src/modules/<log-explorer>/**`

`server/modules/audit/**` 可以作为 downstream consumer 提供 `Audit -> Log` 关联入口，但不是 `Log Explorer` 的 canonical contract owner。

### 4.7.5 Canonical investigation workflow

未来运维调查应遵循以下 canonical workflow：

1. start from one canonical seed:
   - `requestId`
   - `traceId`
   - `audit event id`
   - `user`
   - `ip`
2. if seed is an audit incident or security event:
   - `audit` owns the incident truth
   - jump from audit evidence to bounded related access/application logs only by canonical correlation fields
3. if seed is an access log:
   - logging authority owns request fact truth
   - operator may jump to matching audit records only when audit authority confirms them
4. if seed is an application log:
   - logging authority owns runtime/app fact truth
   - operator may jump to bounded request context and then to audit/security evidence if present
5. if seed is monitor evidence:
   - monitor remains a separate authority
   - monitor may link into audit incidents through canonical `EvidenceLink`
   - monitor does not become log-explorer authority

### 4.7.6 Runtime Phase D readiness

`Phase D Log Explorer` readiness judgment rules:

- `Ready`
  - only when access/app log authority, retention authority, shared contract owner, and operator workflow are all explicit
- `Partially Ready`
  - when logging boundaries and workflow are explicit, but retention or runtime storage contract is still undefined
- `Not Ready`
  - when the system still expects audit tables, metadata fallback, or frontend heuristics to substitute for logging authority

基于当前仓库事实：

- judgment: `Partially Ready`
- evidence:
  - `AppLogger` authority exists
  - `AccessLogger` authority, durable `access_logs` storage, explorer read surface, and retention cleanup exist under `server/internal/httpx/**`
  - `Security Event -> Audit` authority exists
  - `Audit vs Log Explorer` boundary can now be stated explicitly
  - `Audit Log` durable storage and retention cleanup exist under `server/modules/audit/**`
  - `Application Log` durable storage and retention authority exist under `server/internal/logger/**`
  - app-log runtime read contract, including OpenAPI/authz/query/explorer approval, remains `deferred-not-approved`

未来 topic 必须遵守：

- `Audit` 不得侵入 `Monitor` 拥有的 anomaly / status truth
- `Monitor` 不得推断 `Audit` 拥有的 incident / policy / audit fact truth
- `Log Explorer` 不得直接把 audit 表当成日志底座
- `Metrics` 不得通过解析 audit / app log 伪装落地

## 5. 异步写入规范

### 5.1 App Log

- 默认同步写入当前 logger backend
- 原因：需要在错误路径和 shutdown 前可立即观测

### 5.2 Error Log

- 默认同步写入
- 原因：失败路径不能通过异步队列丢失关键排障证据

### 5.3 Audit Event

- 允许异步，但必须经统一 audit recorder / audit module path
- 对用户资源变更、权限变更、访问控制变更等关键管理动作：
  - 至少保证“已尝试写入审计”
  - 审计异步失败时必须产生可检索的 `Error Log`

### 5.4 Security Event

- 允许通过异步事件总线进入审计持久化
- 但安全拒绝、认证失败等关键事件若持久化失败，必须留下失败回退日志
- 不能因为异步化而静默丢弃

### 5.5 日志异步 / 审计异步 / 安全事件异步的区分

- `App Log` / `Error Log`
  - 当前 MVP 默认同步
- `Audit Event`
  - 可以异步入库，但必须有失败回退日志
- `Security Event`
  - 可以异步桥接到审计，但桥接失败必须有同步错误记录

### 5.6 flush / shutdown

- runtime shutdown 必须给异步审计/安全事件留下有限 flush 窗口
- shutdown 失败必须打 `Error Log`
- 不得用 `context.Background()` 绕开 shutdown 语义

### 5.7 测试要求

- 测试环境必须提供可断言路径
- 对异步审计/安全事件：
  - 要么用同步 fake / stub bus
  - 要么提供显式 drain / wait 机制
- 测试不能依赖不确定 sleep

## 7. Metric Placeholder 规范

- 复用这里的 candidate inventory

## 8. AI 开发检查清单

后续 AI 或开发者在新增后端功能时，必须至少检查：

- 是否真的需要 `App Log`，还是已有统一入口足够
