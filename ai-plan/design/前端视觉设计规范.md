# Graft 前端视觉设计规范

## 1. 目标

这份规范把 Graft 的前端视觉基线固定下来，作为 `web` 的设计真值补充。

目标不是“做得更花”，而是让所有后台页面都满足：

- 结构统一
- 状态清晰
- 主题可切换
- 组件可复用
- AI 可稳定生成

## 2. 视觉定位

Graft 是组合式后台平台控制台，因此视觉策略应偏向：

- 控制台感
- 信息密度高但不拥挤
- 品牌感来自 token、节奏和层次，不来自大面积装饰
- 允许少量科技感，但不能偏营销页

保留项：

- 当前主题工作台的悬浮按钮栏和 token 编辑感可以作为正向样本

禁止项：

- 过度渐变
- 复杂背景纹理
- 通用 SaaS 营销风大横幅
- 与 TDesign 背离的第二套视觉系统

## 3. 页面类型注册表

### 3.1 基础母版

- `shell`
  - 应用壳层、导航、顶部栏、内容容器
- `auth`
  - 登录、会话失效、认证相关页面
- `overview-dashboard`
  - 概览、监控、状态页、统计面板
- `list-form-detail`
  - 用户、角色、权限、配置、资源管理等 CRUD 页面

### 3.2 扩展类型

- `settings`
- `workflow`
- `editor`
- `query-builder-list-detail`
- `log-audit`
- `error-result`
- `docs-help`

扩展页型补充约束：

- `query-builder-list-detail`
  - 用于带主搜索、动态筛选构建器、快捷筛选、表格结果面和详情抽屉的增强列表页
  - 快捷筛选必须收口为当前筛选器 authority 已支持的字段组合，不得引入 UI 无法表达的隐藏条件
  - 快捷筛选应用后，所有条件必须可回显、可修改、可移除、可手动重构
- `log-audit`
  - 属于日志工作台变体，默认复用 `query-builder-list-detail` 的查询 authority 规则
  - 只允许在后端已支持数组查询的字段上提供多值快捷筛选；单值契约不得由前端擅自升级为多值语义

### 3.3 注册规则

- 每个前端需求必须先声明页面类型
- 首阶段内置 4 类基础页面母版，用于覆盖当前 Graft 高频后台页面；它们不是页面类型全集
- 若需求无法自然归入这 4 类，必须先登记为扩展页面类型，并补充：
  - 信息层级
  - 组件组合
  - 状态集合
  - 主题响应要求
  - i18n 要求
  - 验收规则

## 4. 页面骨架

### 4.1 通用后台骨架

- 页面应优先思考这四个表面：
  - `page header`
  - `primary action area`
  - `main content surface`
  - `feedback surface`
- 不同页面类型可按母版裁剪
- 不强制所有页面都出现 `table` / `card` / `detail`

### 4.2 壳层页

统一使用：

- 顶部导航
- 侧边导航
- 面包屑
- 内容区
- 底部 Footer
- 统一安全留白，避免底部悬浮工具条遮挡内容
- 用户操作区

### 4.3 标准业务页

默认顺序：

1. 页面标题与摘要
2. 主操作区或筛选区
3. 列表 / 卡片 / 图表主内容
4. 详情抽屉、弹窗或辅助面板
5. 空态、异常态与提示区

补充规则：

- 后台管理页应优先复用共享页面容器与宽度变量策略，保持统一居中与稳定边距
- 当表格页出现空态时，应优先使用 TDesign 原生空态能力，并保持表头、表格主体和分页结构连续

### 4.4 仪表页

推荐结构：

- page header
- 顶部概览卡
- 趋势或指标图卡
- 依赖状态卡
- 运行时明细卡

### 4.5 登录页

推荐结构：

- 左侧品牌与说明
- 右侧表单
- 底部版权与辅助入口

## 5. 颜色与状态

- 页面色彩优先使用 `--td-*` token。
- 状态语义必须映射到 TDesign 语义色。
- 图表颜色必须随 mode、brand theme 和容器变化同步。
- 只有 token 缺失时，才允许十六进制兜底。

## 6. 组件组合

优先组合：

- `Card` + `Table` + `Form`
- `Drawer` + `Form`
- `Dialog` + `Form`
- `Tag` + `Alert`
- `Tabs` + `Card`
- `Result` + 空态插画

原则：

- 列表页优先用表格
- 编辑交互优先用抽屉或弹窗
- 详情页优先用卡片分区
- 批量操作必须显式可见
- 后台大屏容器优先通过共享容器和变量覆盖扩展，不通过局部偏移 hack 追求视觉居中

## 7. 模板落点

可复用参考模板放在：

- `ai-plan/design/graft-design-system/shell.md`
- `ai-plan/design/graft-design-system/auth.md`
- `ai-plan/design/graft-design-system/overview-dashboard.md`
- `ai-plan/design/graft-design-system/list-form-detail.md`

这些模板只作为参考资产，不是运行时依赖，也不是页面白名单。

## 8. 生成约束

AI 生成前端页面时，应先锁定：

- 页面类型
- 主信息层级
- 组件组合
- 状态表达
- 主题 token 依赖

再开始写代码。

任务大小分流规则：

- 新增页面、重构页面、复杂布局页面：
  - 先输出结构方案并等待确认
- 简单文案、样式、小交互修复：
  - 可直接实现
  - 但仍必须通过页面类型、i18n、主题和可见文案自检

## 9. 验收标准

一页是否符合本规范，至少看这几件事：

- 组件结构是否清晰
- 状态是否可读
- 颜色是否 token 驱动
- 是否能在浅色/深色模式下成立
- 是否仍然像 Graft，而不是 demo 模板
- 用户可见文案是否避免实现期和调试期术语
